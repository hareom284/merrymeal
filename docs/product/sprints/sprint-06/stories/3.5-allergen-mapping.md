## User story

> **As a** dietitian
> **I want** ingredients to declare which allergens they contain
> **so that** the planner and member dashboard can flag dangerous meals
> automatically, without me hand-checking every recipe before each
> publish.

## Tech / library notes

- M2M `through` table name: **explicitly set** via
  `through="kitchens.IngredientAllergy"` *only* if we need extra columns;
  otherwise let Django manage it but set `db_table="ingredient_allergy"`
  via `class Meta: db_table = ...` on an explicit through model. The
  schema-corrected SQL doesn't define this join (the table is new in
  this story), so the project gets to pick. We pick `ingredient_allergy`
  (singular both sides) for consistency with `allergy_user` and
  `diet_preference_user`.
- The service is pure (no DB writes). It does one `meal.ingredients.all()`
  with `prefetch_related("contains_allergens")` and intersects in Python.
- Seed mappings table — *minimum set* for v1; the dietitian can extend
  via admin:
  - `peanut` → `Peanut`
  - `peanut oil` → `Peanut`
  - `prawn` / `shrimp` / `crab` / `lobster` → `Shellfish`
  - `cow milk` / `butter` / `cheese` → `Milk`
  - `egg` → `Egg`
  - `wheat flour` / `bread` / `pasta` → `Wheat`
  - `soy sauce` / `tofu` / `edamame` → `Soy`
  - `salmon` / `tuna` / `cod` → `Fish`
  - `almond` / `cashew` / `walnut` → `Tree nut`
- The seed creates any missing `Allergy` rows by name first, then
  matches ingredients by case-insensitive substring (`name__icontains`).
  Substring matching is best-effort; the `--audit` flag exists precisely
  because we expect gaps.

### Task 3: The service (GREEN — part 2)

**Files:** `apps/planning/services/allergen.py`

```python
# apps/planning/services/allergen.py
from __future__ import annotations


def meal_allergens_for_member(meal, member) -> list:
    """Return the de-duplicated intersection of the meal's ingredient
    allergens and the member's declared allergies, as a list[Allergy].

    Pure — no writes. One query for ingredients (with prefetch), one
    for the member's allergies.
    """
    member_allergy_ids = set(member.allergies.values_list("id", flat=True))
    if not member_allergy_ids:
        return []

    seen: set[int] = set()
    matched: list = []
    qs = meal.ingredients.prefetch_related("contains_allergens")
    for ing in qs:
        for a in ing.contains_allergens.all():
            if a.id in member_allergy_ids and a.id not in seen:
                seen.add(a.id)
                matched.append(a)
    return matched
```

SEED_MAP = {
    "Peanut":    ["peanut", "peanut oil"],
    "Shellfish": ["prawn", "shrimp", "crab", "lobster"],
    "Milk":      ["cow milk", "butter", "cheese"],
    "Egg":       ["egg"],
    "Wheat":     ["wheat flour", "bread", "pasta"],
    "Soy":       ["soy sauce", "tofu", "edamame"],
    "Fish":      ["salmon", "tuna", "cod"],
    "Tree nut":  ["almond", "cashew", "walnut"],
}

added = present = 0
for allergy_name, ingredient_terms in SEED_MAP.items():
    allergy, _ = Allergy.objects.get_or_create(name=allergy_name)
    for term in ingredient_terms:
        for ing in Ingredient.objects.filter(name__icontains=term):
            if ing.contains_allergens.filter(pk=allergy.pk).exists():
                present += 1
            else:
                ing.contains_allergens.add(allergy)
                added += 1

self.stdout.write(f"seed_dietary: linked {added} new pairs, {present} already present")

## Pitfalls reviewer will check

- **Table name `kitchens_ingredient_contains_allergens`.** Django's
  auto-naming creates that monstrosity. Set `db_table="ingredient_allergy"`
  via an explicit through model.
- **Forgetting `blank=True`.** Without it, the admin form forces a
  selection — most ingredients won't have a mapping.
- **Substring match too greedy.** "peanut" matches "peanut" and
  "peanut oil" — fine. "milk" would match "milk chocolate" though,
  which is also a milk allergen, so OK. **But** "egg" matches
  "eggplant", which is not. Either accept that the dietitian audits
  via admin, or use `__iexact` and curate a longer list. v1 takes the
  former path; note this in the migration comment.
- **N+1 in service.** The `prefetch_related("contains_allergens")` is
  essential — without it, one query per ingredient.
- **De-duplication.** A meal that uses both `butter` and `cheese` must
  show "Milk" exactly once. Tests cover this; reviewer will check.
- **Member with zero allergies.** The early-return short-circuits the
  prefetch — leave it; it saves work on the common case.

## User story

> **As an** admin / dietitian
> **I want** to schedule one meal per (kitchen, service_date)
> **so that** delivery and inventory have a single source of truth for what is being cooked when and where.

## Acceptance criteria

- [ ] New Django app `apps/planning/` registered in `INSTALLED_APPS`.
- [ ] `apps/planning/models/meal_plans.py::MealPlan` matches the schema in
      `merrymeal_schema_corrected.sql`:
  - `meal` — FK to `kitchens.Meal`.
  - `kitchen` — FK to `kitchens.Kitchen`.
  - `service_date` — `DateField` (local date, Australia/Melbourne).
  - `day_of_week` — choices enum `mon..sun`.
  - `meal_type` — choices enum `fresh|frozen`, default `fresh`.
  - `planned_quantity` — `PositiveIntegerField`, default `0`.
  - `published_by` — FK to `accounts.User`.
  - `created_at` — `auto_now_add`.
- [ ] DB table name is `meal_plans` (matches the corrected SQL schema).
- [ ] Unique constraint on (`kitchen_id`, `service_date`) — exactly one
      plan per kitchen per day.
- [ ] Django admin CRUD with filters by `kitchen` and week.
- [ ] **Model is schema-only.** No `save()` override, no business rules.
      Auto-deriving `day_of_week` from `service_date` is **out of scope
      for this story** — that lives in Story 3.3's form/service.

## Tech / library notes

- `service_date` is a **local date** (`DateField`), not a `DateTimeField`.
  The frozen-weekend rule keys off the local weekday in Melbourne; using
  UTC would shift Saturday in the wrong direction.
- `day_of_week` is denormalised: it duplicates information derivable from
  `service_date`. We keep it because (a) it matches the schema and (b) the
  planner UI sorts on it cheaply. Tests assert "if set, value matches
  weekday" — they do **not** auto-fix it.
- Unique constraint goes in `Meta.constraints` (not `unique_together`).
- Admin uses `list_filter = ("kitchen", "service_date")` plus a custom
  `get_search_fields` covering meal name and kitchen name.

```python
from django.conf import settings
from django.db import models


class MealPlan(models.Model):
    """One planned meal per (kitchen, service_date).

    Schema-only. The fresh/frozen automation lives in
    `apps.planning.services.assignment` (Story 3.2); the planner UI sets
    `meal_type` at write time (Story 3.3).
    """

    DAY_CHOICES = [
        ("mon", "Mon"), ("tue", "Tue"), ("wed", "Wed"),
        ("thu", "Thu"), ("fri", "Fri"),
        ("sat", "Sat"), ("sun", "Sun"),
    ]

    TYPE_CHOICES = [
        ("fresh", "Fresh"),
        ("frozen", "Frozen"),
    ]

    meal = models.ForeignKey(
        "kitchens.Meal", on_delete=models.PROTECT, related_name="plans"
    )
    kitchen = models.ForeignKey(
        "kitchens.Kitchen", on_delete=models.PROTECT, related_name="plans"
    )
    service_date = models.DateField()
    day_of_week = models.CharField(max_length=3, choices=DAY_CHOICES)
    meal_type = models.CharField(
        max_length=6, choices=TYPE_CHOICES, default="fresh"
    )
    planned_quantity = models.PositiveIntegerField(default=0)
    published_by = models.ForeignKey(
        settings.AUTH_USER_MODEL,
        on_delete=models.PROTECT,
        related_name="published_meal_plans",
    )
    created_at = models.DateTimeField(auto_now_add=True)

    class Meta:
        app_label = "planning"
        db_table = "meal_plans"
        constraints = [
            models.UniqueConstraint(
                fields=["kitchen", "service_date"],
                name="uq_meal_plan_kitchen_date",
            ),
        ]
        indexes = [
            models.Index(fields=["service_date"], name="idx_plan_date"),
        ]

    def __str__(self) -> str:
        return f"{self.kitchen_id}/{self.service_date} → meal {self.meal_id}"
```

## Pitfalls reviewer will check

- **`db_table` mismatch.** Must be `meal_plans` (plural, snake_case). The
  SQL schema uses that exact name; Story 3.7 reads it via raw queries.
- **`unique_together` instead of `UniqueConstraint`.** Django still
  supports the former but the project uses `Meta.constraints` everywhere
  else — stay consistent.
- **`service_date` typed as `DateTimeField`.** Must be `DateField`. A
  Melbourne Saturday at 23:30 UTC is still Sunday locally; we never want
  to chase that bug.
- **Business logic on the model.** No `save()` override, no
  `assign_meal_type` call inside the model — those live in Story 3.2's
  service. A reviewer who sees them rejects the PR.
- **Missing `related_name`.** `meal.plans.all()` and
  `kitchen.plans.all()` are used in later stories' queries.
- **`on_delete=CASCADE` on `published_by`.** Use `PROTECT`: an admin user
  may leave the team; their published plans must not disappear.

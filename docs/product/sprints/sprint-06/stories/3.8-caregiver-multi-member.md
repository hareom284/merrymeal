## User story

> **As a** caregiver looking after two or more members
> **I want** one screen listing every member I look after with each one's
> "today" card summary
> **so that** I do not have to log in as each member to know what they
> are getting and whether any allergens are flagged.

## Acceptance criteria

- [ ] Route `/dashboard/` when `request.user.role == 'caregiver'` renders
      the caregiver list view. (Story 3.4 already added a router skeleton;
      this story registers the named URL it redirects to.)
- [ ] List one row per linked `member` via the `member_caregivers` link
      table. Each row shows:
  - Member name (and photo placeholder).
  - Today's meal name (or "No meal today").
  - Allergen flag if any (red dot + "⚠ Contains <allergens, …>", truncated to one line).
  - Delivery-status placeholder (matches Story 3.4's `delivery_status_label`).
- [ ] Tapping a row navigates to a member-detail page that reuses the
      member's today card — same template fragment as Story 3.4.
- [ ] Mobile-first **375 × 667 px**. List rows have ≥ 44 px touch height.
- [ ] If the caregiver has zero linked members, show a friendly empty
      state: "You are not linked to any members yet. Ask your admin."
- [ ] Query budget: a caregiver with N linked members should trigger
      no more than `2 + 2N` queries (factory limit; the
      `django_assert_max_num_queries` test pins this).

## Tech / library notes

- The `member_caregivers` join table is in Epic 01. The caregiver→members
  reverse relation should already be exposed as
  `caregiver.cared_members` (or whatever Story 1.x named it). Read the
  related-name before writing the query.
- Reuse Story 3.4's `get_today_card(member)` verbatim. Wrap it in
  `get_caregiver_summary(caregiver)` so the view stays thin.
- Extract Story 3.4's card markup into a partial (`_partials/today_card.html`)
  in *this* story. The member view and the caregiver detail page both
  `{% include %}` it. Coordinate with whoever finishes 3.4 — if 3.4
  hasn't merged yet, do the extraction here and submit a small PR back.
- For the 44 px touch target: use Tailwind `min-h-[44px] flex
  items-center` on the row anchor element.
- The row must be a real link (`<a href=…>`), not a `div` with a click
  handler, so it's keyboard-accessible.

```python
# apps/dashboards/services/caregiver_today.py
from __future__ import annotations

from apps.dashboards.services.member_today import get_today_card


def get_caregiver_summary(caregiver) -> list[dict]:
    """Return one entry per linked member with a compact today-card."""
    # Read the M2M name from Epic 01's link table.
    linked = caregiver.cared_members.all().select_related().order_by("full_name")
    summaries = []
    for member in linked:
        card = get_today_card(member)
        summaries.append({
            "member": member,
            "card": card,
            "allergen_label": (
                "⚠ Contains " + ", ".join(a.name for a in card["allergens"])
                if card["allergens"] else ""
            ),
        })
    return summaries
```

**Files:** `apps/dashboards/views/caregiver.py`

```python
from django.contrib.auth.decorators import login_required
from django.http import HttpResponseForbidden
from django.shortcuts import get_object_or_404, render

from apps.accounts.models import User
from apps.dashboards.services.caregiver_today import get_caregiver_summary
from apps.dashboards.services.member_today import get_today_card


def _require_role(user, role):
    return user.is_authenticated and user.role == role


@login_required
def caregiver_list_view(request):
    if not _require_role(request.user, "caregiver"):
        return HttpResponseForbidden("caregivers only")
    summaries = get_caregiver_summary(request.user)
    return render(request, "dashboards/caregiver/list.html", {
        "summaries": summaries,
        "greeting_name": request.user.full_name.split()[0],
    })


@login_required
def caregiver_member_detail_view(request, pk: int):
    if not _require_role(request.user, "caregiver"):
        return HttpResponseForbidden("caregivers only")
    member = get_object_or_404(User, pk=pk, role="member")
    if not request.user.cared_members.filter(pk=pk).exists():
        return HttpResponseForbidden("not linked to this member")
    card = get_today_card(member)
    return render(request, "dashboards/caregiver/member_detail.html", {
        "member": member,
        "card": card,
    })
```

## Pitfalls reviewer will check

- **`cared_members` named relation.** If Epic 01 named it differently
  (`members`, `linked_members`), update the service. Don't query
  `MemberCaregiver` directly — that's the through table and using it
  bypasses the related manager.
- **Linked-member check forgotten.** Without the
  `cared_members.filter(pk=pk).exists()` guard, any caregiver could read
  any member's detail page — a real PII leak. Reviewer will write a
  fuzz-style test in their head before approving.
- **Re-using `member.html` template.** Don't `{% include
  "dashboards/member.html" %}` from the detail page — that template
  `{% extends "base.html" %}` and including extending templates
  silently breaks. Extract the shared partial (Task 2) and include
  *that*.
- **Touch target < 44 px.** The Tailwind `min-h-[44px]` on the anchor
  matters; `card` alone may not produce a 44 px row when content is
  one short line. Playwright assertion catches it.
- **N+1 on allergens.** `get_today_card` does its own prefetch — make
  sure that wins; do not re-query allergens per row from the
  template.
- **Caregiver with 30 members.** v1 doesn't paginate. Acceptable; note
  in retro if a caregiver ever has > 10 linked members.

## User story

> **As a** backend developer
> **I want** a single pure function that returns `'fresh'` or `'frozen'`
> for a `(member, kitchen, service_date)` tuple
> **so that** every place in the codebase that asks
> *"what kind of meal does Margaret get on Wednesday?"* gets the same answer
> and the 10 km / weekend rule lives in exactly one file.

## Acceptance criteria

- [ ] `apps/planning/services/assignment.py::assign_meal_type(member, kitchen, service_date) -> Literal['fresh', 'frozen']`.
- [ ] Logic — applied **in this order**:
  - If `service_date.weekday() in (5, 6)` (Saturday or Sunday) → `'frozen'`.
  - Else compute `haversine_km(...)` between
    `member.primary_address.latitude/longitude` and
    `kitchen.latitude/longitude`:
    - distance `<=` `kitchen.service_radius_km` → `'fresh'`.
    - distance `>`  `kitchen.service_radius_km` → `'frozen'`.
  - Edge case: distance **exactly equal** to the radius → `'fresh'`
    (members on the boundary stay in the fresh group).
- [ ] If the member has no primary address (or it has no lat/lng), raise
      `AddressMissingError` from
      `apps.planning.services.exceptions`.
- [ ] **Pure function.** No DB writes, no logging side effects, no
      `timezone.now()` calls — the caller always passes `service_date`.
- [ ] **5 unit tests minimum**:
  1. Weekday + inside radius → `'fresh'`.
  2. Weekday + outside radius → `'frozen'`.
  3. Weekend regardless of radius → `'frozen'`.
  4. Boundary (distance == radius) → `'fresh'`.
  5. No primary address → `AddressMissingError`.

## Tech / library notes

- Use `apps.core.geo.haversine_km(lat1, lon1, lat2, lon2) -> float`
  (verified signature — returns kilometres as float).
- "Primary address" convention: the first `UserAddress` ordered by `id`,
  or whatever Epic 01 already defines as `member.primary_address`. If
  that property does not yet exist, fall back to
  `member.user_addresses.order_by("id").first()` — wrap that lookup in a
  small helper inside `assignment.py`.
- `kitchen.service_radius_km` is a `Decimal` field; cast to `float` for
  the comparison or use `Decimal(str(distance_km))` to compare in the
  same domain. The float-cast is simpler and fine at this precision.
- Tests use **`freezegun`** *only* if the function reads the clock
  internally — it does not, so freezegun is NOT required here. Tests
  pass an explicit `service_date`. (Story 3.7 *does* need freezegun.)
- The "no address" path raises a custom exception (not `ValueError`) so
  callers can handle it specifically; the admin planner UI should surface
  it as a friendly warning.

```python
# apps/planning/services/assignment.py
from __future__ import annotations

import datetime as dt
from typing import Literal

from apps.core.geo import haversine_km
from apps.planning.services.exceptions import AddressMissingError

MealType = Literal["fresh", "frozen"]


def _primary_address(member):
    """Return the member's primary address (with lat/lng) or None.

    Falls back to "first by id" if a richer `primary_address` property
    is not defined on the user model yet.
    """
    addr = getattr(member, "primary_address", None)
    if addr is None:
        # Reverse-relation may be `user_addresses` (preferred) or `addresses`.
        rel = getattr(member, "user_addresses", None) or getattr(member, "addresses", None)
        if rel is None:
            return None
        addr = rel.order_by("id").first()
    if addr is None:
        return None
    if addr.latitude is None or addr.longitude is None:
        return None
    return addr


def assign_meal_type(member, kitchen, service_date: dt.date) -> MealType:
    """Return 'fresh' or 'frozen' for a (member, kitchen, service_date).

    Rules:
      1. Saturday/Sunday → 'frozen'.
      2. Weekday + member within `kitchen.service_radius_km` → 'fresh'.
      3. Weekday + member outside → 'frozen'.
      4. Boundary (distance == radius) → 'fresh'.

    Raises:
      AddressMissingError — if the member has no usable address.
    """
    # Weekend short-circuit. weekday() is 0=Mon..6=Sun.
    if service_date.weekday() in (5, 6):
        return "frozen"

    addr = _primary_address(member)
    if addr is None:
        raise AddressMissingError(
            f"member id={getattr(member, 'pk', None)} has no usable address"
        )

    distance_km = haversine_km(
        float(addr.latitude), float(addr.longitude),
        float(kitchen.latitude), float(kitchen.longitude),
    )
    radius_km = float(kitchen.service_radius_km)

    return "fresh" if distance_km <= radius_km else "frozen"
```

## Pitfalls reviewer will check

- **Rule order.** Weekend check must come *before* the distance check.
  Otherwise a missing address raises on a Saturday — wrong; the member
  still gets a frozen meal that day.
- **Strict-greater vs greater-or-equal.** Boundary == radius → fresh, so
  the comparison is `distance_km <= radius_km`, not `<`.
- **Decimal/float mix.** `kitchen.service_radius_km` is a `Decimal`;
  comparing `float <= Decimal` works but is subtle. Cast explicitly.
- **`weekday()` vs `isoweekday()`.** Python's `date.weekday()` returns
  0–6 with Monday=0; `isoweekday()` is 1–7. The story uses `weekday() in
  (5, 6)`. Mixing them silently makes "Saturday" mean "Sunday".
- **Silent fallback to a default address.** If the member has multiple
  addresses, this story picks the first by id; later epics may add a
  `is_primary` flag. Don't pick "any address with lat/lng" — be
  deterministic.
- **DB writes sneaking in.** If you find yourself caching a computed
  meal_type on `MealPlan` here, stop — that's Story 3.3's responsibility.

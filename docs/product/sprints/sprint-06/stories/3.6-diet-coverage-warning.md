## User story

> **As an** admin / dietitian
> **I want** the planner to warn me, per cell, when the planned meal is
> incompatible with one or more declared diets of members served by that
> kitchen on that day
> **so that** I do not publish a Wednesday menu that leaves three halal
> members without a suitable meal.

## Acceptance criteria

- [ ] New M2M `Meal.diets` → `dietary.DietPreference` (tagging which
      diets the meal is compatible with).
- [ ] New column `MealPlan.warnings_acknowledged_by` (FK to `accounts.User`, nullable).
- [ ] `apps/planning/services/coverage.py::diet_warnings(meal_plan) -> dict[DietPreference, int]`:
  - For each `DietPreference` declared by any member who is geographically
    served by `meal_plan.kitchen` (within `service_radius_km`) on
    `meal_plan.service_date`, count how many of those members hold that
    diet **and** the planned meal is *not* tagged with that diet.
  - Returns `{}` for the empty case.
- [ ] On the admin planner grid (extends Story 3.3), each cell with a
      non-empty warning dict shows a yellow badge with the count
      (`Σ values`). Hover/tap reveals an Alpine.js popover listing
      `"<n> members are <diet name> — this meal is not <diet name>"`.
- [ ] An "Override (acknowledged)" link in the popover calls
      `apps/planning/services/coverage.py::acknowledge(meal_plan, user)`
      which sets `warnings_acknowledged_by = user`. The badge then
      renders in muted grey instead of yellow.
- [ ] Compatibility logic for v1 = pure tag match. No nutritional
      analysis, no ingredient introspection. A meal is "compatible"
      with diet X iff `meal.diets.filter(pk=X.pk).exists()`.

## Tech / library notes

- "Members served by kitchen K on date D" for v1 = members whose primary
  address sits within `K.service_radius_km` **and** for whom
  `assign_meal_type(member, K, D)` would return `"fresh"`. Weekend
  members are excluded — they get a frozen meal regardless of diet
  coverage, and frozen meals are pre-batched, so the warning would be
  noise. Doc this explicitly inside `coverage.py`.
- The candidate-members query can be expensive on a 10k-member dataset.
  v1 doesn't optimise: it loads members for the kitchen, runs
  `assign_meal_type` per member in Python. The risk note in the sprint
  plan covers this; profile on a 1000-member fixture in week 2 before
  deciding to optimise.
- The badge UI uses Alpine.js (`x-data="{ open: false }"`) — no new
  JavaScript dependencies.
- Acknowledgement is per-`MealPlan` row. If the admin edits the cell
  later, the acknowledgement is **cleared** automatically by re-using
  `upsert_cell` (Story 3.3) — make sure `upsert_cell` resets the field
  to `None` on update. Update the test in Story 3.3's suite.

### Task 3: Service implementation (GREEN — part 2)

**Files:** `apps/planning/services/coverage.py`

```python
# apps/planning/services/coverage.py
from __future__ import annotations

from collections import Counter

from apps.accounts.models import User
from apps.planning.services.assignment import assign_meal_type
from apps.planning.services.exceptions import AddressMissingError


def _candidate_members(meal_plan):
    """Members served *fresh* by this plan's kitchen on the plan's date."""
    candidates = User.objects.filter(
        role="member", is_active=True,
    ).prefetch_related("diet_preferences", "user_addresses")
    for m in candidates:
        try:
            outcome = assign_meal_type(m, meal_plan.kitchen, meal_plan.service_date)
        except AddressMissingError:
            continue
        if outcome == "fresh":
            yield m


def diet_warnings(meal_plan) -> dict:
    """Return {DietPreference: count} for diets the meal does not cover."""
    covered_ids = set(meal_plan.meal.diets.values_list("id", flat=True))
    counts: Counter = Counter()
    for member in _candidate_members(meal_plan):
        for diet in member.diet_preferences.all():
            if diet.id not in covered_ids:
                counts[diet] += 1
    return dict(counts)


def acknowledge(meal_plan, user) -> None:
    meal_plan.warnings_acknowledged_by = user
    meal_plan.save(update_fields=["warnings_acknowledged_by"])
```

## Pitfalls reviewer will check

- **Acknowledgement persists across meal swap.** If the admin swaps the
  meal in the cell, the new meal's compatibility profile is different —
  the old ack is no longer meaningful. `upsert_cell` must clear it.
- **N+1 explosion.** `_candidate_members` walks every member; the
  `prefetch_related` is essential. Reviewer will load 1000 members and
  watch the planner load time stay under 1 second per kitchen.
- **`acknowledge` requires admin.** The endpoint is decorated, but the
  service function itself does not check role — that's deliberate (it's
  a service, not a view), but the URL must be gated.
- **Yellow contrast.** `bg-yellow-200 text-yellow-900` passes WCAG AA at
  normal-text sizes. Do not use `bg-yellow-400 text-white`.
- **"Diet not declared by any local member."** If no candidate member
  holds the diet, the warning never fires — correct.
- **Stretch flag confusion.** If this story slips, mark the
  `MealPlan.warnings_acknowledged_by` migration as also slipped — do
  *not* land a half-applied schema. Either both migrations ship in 06
  or both move to 07.

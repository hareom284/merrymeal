## User story

> **As a** prospective member
> **I want** to declare my diet preferences and allergies up-front
> **so that** the kitchen never serves me food I cannot eat.

## Acceptance criteria

- [ ] Route `GET /apply/dietary/` renders step 3 of the wizard.
- [ ] Step 3 requires `request.session["application_id"]` and `status='draft'` (same guard as step 2). Otherwise redirect to `/apply/`.
- [ ] Multi-select chips for **diet preferences** (one chip per row in `DietPreference`). Each chip has `min-height: 44px` and `min-width: 44px`.
- [ ] Multi-select chips for **allergies** (one chip per row in `Allergy`).
- [ ] An "I don't have any" toggle for each section. When toggled on, the chip group is disabled and the corresponding list submits empty.
- [ ] On valid POST: `Application.dietary_ids` and `Application.allergy_ids` are written (JSON lists of ids); status transitions to `submitted`; `request.session["application_id"]` is cleared; a confirmation screen renders.
- [ ] A confirmation email is sent to the applicant's email address: subject `"We've got your MerryMeal application"`, body says "We'll be in touch in 3–5 business days."
- [ ] If `applying_for_other` is true (Story 1.10), the email also goes to the **caregiver's** email (cc).
- [ ] The service function `submit_application(application_id=...)` is the only code path that flips the status to `submitted` and dispatches the email.

## Tech / library notes

- Diet + allergy rows come from `apps.dietary.models` (Story 1.5). Import paths: `from apps.dietary.models import DietPreference, Allergy`.
- Email rendering uses `render_to_string("accounts/emails/application_received.html", ctx)` + `.txt` variant; send via `django.core.mail.EmailMultiAlternatives`. The unit test asserts `mail.outbox` length and the `to=[...]` list.
- Chips are plain `<button type="button">` toggling a hidden `<input type="checkbox">` via Alpine.js — keeps the form submission a normal form-POST. Avoid HTMX here; the wizard is small enough not to need it.
- The "I don't have any" toggle is an Alpine boolean. When true, the chip group has `aria-disabled="true"` and the form payload sends an empty list (Alpine clears the array before submit via `@change`).
- JSON payload: form `cleaned_data["dietary_ids"]` is a `list[int]`. The service casts each to `int` defensively and validates each id exists in the corresponding table. Unknown ids raise `ValueError`.

```python
from django.core.mail import EmailMultiAlternatives
from django.template.loader import render_to_string
from django.utils import timezone

# ... existing imports / functions above ...


def submit_application(
    *,
    application_id: int,
    dietary_ids: list[int],
    allergy_ids: list[int],
) -> Application:
    """Step 3 → transitions a draft to submitted, writes the selected
    dietary/allergy ids, and dispatches the confirmation email.
    """
    from apps.dietary.models import Allergy, DietPreference

    try:
        application = Application.objects.get(id=application_id)
    except Application.DoesNotExist as exc:
        raise ValueError(f"Application {application_id} not found") from exc

    if application.status != Application.STATUS_DRAFT:
        raise ValueError(
            f"Application {application.id} is not in draft state "
            f"(status={application.status})"
        )

    diet_ids = [int(x) for x in (dietary_ids or [])]
    allergy_id_list = [int(x) for x in (allergy_ids or [])]

    known_diet = set(
        DietPreference.objects.filter(id__in=diet_ids).values_list("id", flat=True)
    )
    unknown_diet = [x for x in diet_ids if x not in known_diet]
    if unknown_diet:
        raise ValueError(f"Unknown diet preference ids: {unknown_diet}")

    known_allergy = set(
        Allergy.objects.filter(id__in=allergy_id_list).values_list("id", flat=True)
    )
    unknown_allergy = [x for x in allergy_id_list if x not in known_allergy]
    if unknown_allergy:
        raise ValueError(f"Unknown allergy ids: {unknown_allergy}")

    application.dietary_ids = diet_ids
    application.allergy_ids = allergy_id_list
    application.status = Application.STATUS_SUBMITTED
    application.save(update_fields=[
        "dietary_ids", "allergy_ids", "status", "updated_at",
    ])

    _send_confirmation_email(application)
    return application


def _send_confirmation_email(application: Application) -> None:
    ctx = {
        "application": application,
        "applicant_first_name": application.full_name.split()[0],
        "sent_at": timezone.now(),
    }
    subject = "We've got your MerryMeal application"
    text_body = render_to_string("accounts/emails/application_received.txt", ctx)
    html_body = render_to_string("accounts/emails/application_received.html", ctx)

    cc = []
    if application.applying_for_other and application.caregiver_email:
        cc.append(application.caregiver_email)

    msg = EmailMultiAlternatives(
        subject=subject,
        body=text_body,
        to=[application.email],
        cc=cc or None,
    )
    msg.attach_alternative(html_body, "text/html")
    msg.send()
```

```python
class ApplicationDietaryForm(forms.Form):
    """Step 3 — diet preferences + allergies, both optional multi-selects.

    The widget side (chip toggles) lives in the template via Alpine; this
    form only validates that the submitted ids are integers.
    """

    dietary_ids = forms.TypedMultipleChoiceField(
        required=False,
        coerce=int,
        choices=[],  # populated at __init__ from DB
        widget=forms.MultipleHiddenInput(),
    )
    allergy_ids = forms.TypedMultipleChoiceField(
        required=False,
        coerce=int,
        choices=[],
        widget=forms.MultipleHiddenInput(),
    )

    def __init__(self, *args, **kwargs):
        super().__init__(*args, **kwargs)
        from apps.dietary.models import Allergy, DietPreference
        self.fields["dietary_ids"].choices = [
            (p.id, p.name) for p in DietPreference.objects.all().order_by("name")
        ]
        self.fields["allergy_ids"].choices = [
            (a.id, a.name) for a in Allergy.objects.all().order_by("name")
        ]
```

```python
from django.shortcuts import redirect, render
from django.views.decorators.http import require_http_methods

from apps.accounts.forms import (
    ApplicationAddressForm,
    ApplicationContactForm,
    ApplicationDietaryForm,
)
from apps.accounts.models import Application
from apps.accounts.services import (
    create_draft_application,
    submit_application,
    update_application_address,
)
# _load_draft_from_session defined in Story 1.8


@require_http_methods(["GET", "POST"])
def application_step_3(request):
    application = _load_draft_from_session(request)
    if application is None:
        return redirect("/apply/")

    from apps.dietary.models import Allergy, DietPreference

    if request.method == "POST":
        form = ApplicationDietaryForm(request.POST)
        if form.is_valid():
            submit_application(
                application_id=application.id,
                dietary_ids=form.cleaned_data["dietary_ids"],
                allergy_ids=form.cleaned_data["allergy_ids"],
            )
            request.session.pop("application_id", None)
            return redirect("/apply/done/")
    else:
        form = ApplicationDietaryForm(
            initial={
                "dietary_ids": application.dietary_ids or [],
                "allergy_ids": application.allergy_ids or [],
            }
        )

    return render(
        request,
        "accounts/application/step_3.html",
        {
            "form": form,
            "step": 3,
            "diet_choices": list(DietPreference.objects.order_by("name")),
            "allergy_choices": list(Allergy.objects.order_by("name")),
        },
    )


@require_http_methods(["GET"])
def application_done(request):
    return render(request, "accounts/application/done.html")
```

```html
function dietaryForm({ initialDiet, initialAllergy }) {
  return {
    diet: initialDiet || [],
    allergy: initialAllergy || [],
    noDiet: false,
    noAllergy: false,
    toggleDiet(id) {
      this.diet = this.diet.includes(id)
        ? this.diet.filter(x => x !== id)
        : [...this.diet, id];
    },
    toggleAllergy(id) {
      this.allergy = this.allergy.includes(id)
        ? this.allergy.filter(x => x !== id)
        : [...this.allergy, id];
    },
    init() {
      this.$watch('noDiet', v => { if (v) this.diet = []; });
      this.$watch('noAllergy', v => { if (v) this.allergy = []; });
    }
  };
}
```

## Pitfalls reviewer will check

- **Through-tables written too early.** This story must NOT write `diet_preference_user` / `allergy_user`. Those need a `users.id` to FK from. Grep for `UserDietPreference.objects.create` and confirm absent in this PR.
- **Unknown ids accepted.** A maliciously crafted POST `dietary_ids=[99999]` must be rejected by the service.
- **Email not sent on validation failure.** If the form is invalid, the service is never called and `mail.outbox` must be empty.
- **Status flipped to `submitted` even when the email send raises.** Not a hard requirement, but the reviewer will ask: do we want the row to be in `submitted` if SMTP errored? Current implementation: yes (commit before send). Acceptable for v1.
- **JSON list ordering.** Use `[d.id]` (a real list), not a `QuerySet` or `set` — Django JSONField will serialise either but the order in the DB should be stable for golden-master tests.
- **Alpine.js loaded.** `base.html` already includes Alpine. If you also load it inside `step_3.html`, you get double-initialisation warnings in console. Don't.
- **Empty-state.** If `DietPreference` / `Allergy` tables are empty (e.g. `seed_dietary` not run), the chip groups render empty. The form still validates and submits — but the reviewer will ask why no chips show. Document the seed dependency.
- **CSRF on the form.** `{% csrf_token %}` is present in the template. Without it, POST 403s in production.

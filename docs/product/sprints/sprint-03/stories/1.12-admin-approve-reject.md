## User story

> **As an** admin
> **I want** to approve or reject an application with one click
> **so that** I can clear the queue in under five minutes per application.

## Acceptance criteria

- [ ] `GET /admin/applications/<id>/` renders the application detail page with every field the applicant submitted (contact, address, dietary chips, allergy chips, caregiver block if applicable).
- [ ] Page shows a primary **Approve** button and a secondary **Reject** button (rejection requires a free-text reason).
- [ ] **Approve** action (`POST /admin/applications/<id>/approve/`):
  - Calls `approve_application(application, admin_user)` service.
  - The service performs ALL writes inside a single `transaction.atomic()`:
    - Member `User` row via `apps.accounts.services.users.create_user` (role=`member`, `is_active=False`, password set to an unusable value via `set_unusable_password()` so login works only via the welcome link).
    - Caregiver `User` row if `applying_for_other=True` AND no existing `users.email` matches `caregiver_email`. If one exists, **reuse** that user (no duplicate; no password reset).
    - `Address` row linked to the new member, populated from `Application.address_label`/`street`/`postal_code`/`city_id`.
    - `UserDietPreference` rows from `Application.dietary_ids`.
    - `UserAllergy` rows from `Application.allergy_ids`.
    - `CaregiverLink` row if `applying_for_other=True`.
    - Update the `Application`: `status='approved'`, `approved_by=admin_user.id`, `approved_at=timezone.now()`.
    - Issue a single-use password-setup token (Django `signing` payload `{"uid": <member.id>}`, max-age 7 days; persisted in a new `password_setup_tokens` table for single-use enforcement).
    - Send the "welcome — set your password" email to the member (cc the caregiver if applicable) with the link `/accounts/set-password/?token=<token>`.
  - If any one write fails, **no rows commit** (atomicity).
  - Audit log entry written via `django-auditlog`.
  - Redirect to `/admin/applications/` with a success flash.
- [ ] **Reject** action (`POST /admin/applications/<id>/reject/`):
  - Requires non-empty `reason` form field.
  - Updates `Application`: `status='rejected'`, `rejected_reason=<reason>`, `approved_by=admin_user.id`, `approved_at=timezone.now()`.
  - Sends a polite rejection email to the applicant referencing the reason.
  - Audit log entry written.
  - Redirect to `/admin/applications/` with a success flash.
- [ ] Service-level pytest covers:
  - Approve member-only path.
  - Approve caregiver+member path (new caregiver user).
  - Approve caregiver+member path with **existing** caregiver email (no duplicate user created).
  - Reject path.
  - Transaction atomicity: simulate one downstream write failing and assert nothing committed.
- [ ] Password-setup token:
  - Generated with `django.core.signing.dumps({"uid": ...})`.
  - Stored in `password_setup_tokens` table: columns `id`, `user_id`, `token_hash` (SHA-256 of the signed token), `expires_at`, `used_at`, `created_at`. `used_at IS NULL` ⇒ valid.
  - The `/accounts/set-password/` view (out of scope here — landed in earlier sprint or stubbed) consumes the token by marking `used_at=now()`. Setting the same password a second time with the same token returns 410 Gone.
  - For Story 1.12 we only have to **create** the token and include it in the welcome email. The "set password" view is a follow-up but the table + service must exist for the unit test to assert the row.
- [ ] Playwright happy path: admin logs in → reviews application → clicks Approve → confirms member receives the welcome email (asserted via `mail.outbox`-equivalent in the dev console backend) → opens the link → can set a password → can log in.

## Tech / library notes

- **`django-auditlog`** is in `requirements.txt` (added in Sprint 02 for user CRUD auditing). `Application` and `User` are already registered. The service uses `auditlog.context.set_actor(admin_user)` so the entry attributes the change correctly.
- **Token format.** `django.core.signing.dumps({"uid": user.id})` produces a URL-safe signed string. We **also** persist a SHA-256 hash of the token in `password_setup_tokens.token_hash` so that:
  - The signed string alone cannot be reused — single-use is enforced by the DB row `used_at`.
  - A leaked DB doesn't leak the raw token (we store the hash).
- **Atomicity.** The whole approval is wrapped in `transaction.atomic()`. The email send happens **after** the `commit` — by `transaction.on_commit(lambda: send_welcome_email(...))`. Rationale: if the SMTP raises, the DB rows still stick (the admin doesn't want to re-approve); and if the DB rolls back, the email is never queued.
- **Audit log.** Use `from auditlog.context import set_actor`. Without that wrap, `django-auditlog` records the change but with `actor=None`.
- **Welcome email** subject: `"Welcome to MerryMeal — set your password"`. Body links to `{settings.SITE_URL}/accounts/set-password/?token=<token>`. `SITE_URL` is set in `config/settings/base.py` (Sprint 01).
- **Rejection email** subject: `"Update on your MerryMeal application"`. Body includes `Application.rejected_reason` rendered as `{{ reason|linebreaksbr }}`.
- **`password_setup_tokens` table.** New table for this story — see Tech notes above for columns. Index on `(user_id, used_at)` is enough; we never look up by `token_hash` alone in this story (the consume view does, but that's a follow-up story).
- **Existing caregiver dedup.** If `caregiver_email` lower-cased matches an existing `users.email`, reuse that user — do NOT issue a token for the caregiver and do NOT change their password. The welcome email cc still goes to them, but with copy that says "log in to confirm the link" (the welcome HTML template branches on this).
- **Diet / Allergy through-tables.** Schema uses composite PKs (`PRIMARY KEY (user_id, diet_preference_id)`). Story 1.5 set those up. The service uses `UserDietPreference.objects.bulk_create([...])` with `ignore_conflicts=True` so a hypothetical retry is safe.

```python
from django.conf import settings
from django.db import models


class PasswordSetupToken(models.Model):
    """Single-use, time-bounded token for the post-approval password-set flow.

    The signed token string itself is mailed to the user. We store only a
    SHA-256 hash here so that a database leak does not leak live tokens.
    """

    user = models.ForeignKey(
        settings.AUTH_USER_MODEL,
        on_delete=models.CASCADE,
        related_name="password_setup_tokens",
        db_column="user_id",
    )
    token_hash = models.CharField(max_length=64)
    expires_at = models.DateTimeField()
    used_at = models.DateTimeField(null=True, blank=True)
    created_at = models.DateTimeField(auto_now_add=True)

    class Meta:
        app_label = "accounts"
        db_table = "password_setup_tokens"
        indexes = [
            models.Index(fields=["user", "used_at"]),
            models.Index(fields=["token_hash"]),
        ]

    def __str__(self) -> str:
        return f"PasswordSetupToken<user={self.user_id} used={self.used_at is not None}>"
```

```python
"""Single-use, time-bounded password-setup tokens.

Token life-cycle:
    issue → mailed to user → user clicks link → consume (atomic, single-use)

The signed token string is what travels by email. We never persist it
verbatim. The DB row stores `sha256(token)` so that:
- a DB dump does not contain live credentials;
- consuming the token is a single UPDATE that flips `used_at`, enforcing
  single use via a `SELECT … FOR UPDATE` inside `transaction.atomic`.
"""
import hashlib
from datetime import timedelta

from django.core.signing import BadSignature, SignatureExpired, dumps, loads
from django.db import transaction
from django.utils import timezone

from apps.accounts.models import PasswordSetupToken, User


TOKEN_TTL = timedelta(days=7)
TOKEN_SALT = "merrymeal.password_setup_token.v1"


class InvalidTokenError(Exception):
    """Signature failed or payload malformed."""


class ExpiredTokenError(Exception):
    """Token age > TTL or DB row past `expires_at`."""


class ConsumedTokenError(Exception):
    """Token was already used. Reuse is forbidden."""


def _hash(token: str) -> str:
    return hashlib.sha256(token.encode("utf-8")).hexdigest()


def issue_password_setup_token(user: User) -> str:
    """Returns the signed token string. Persists the hash and expiry."""
    token = dumps({"uid": user.id}, salt=TOKEN_SALT)
    PasswordSetupToken.objects.create(
        user=user,
        token_hash=_hash(token),
        expires_at=timezone.now() + TOKEN_TTL,
    )
    return token


def consume_password_setup_token(token: str) -> User:
    """Returns the User the token belongs to. Marks it used. Single-use.

    Raises InvalidTokenError / ExpiredTokenError / ConsumedTokenError.
    """
    try:
        payload = loads(token, salt=TOKEN_SALT, max_age=TOKEN_TTL.total_seconds())
    except SignatureExpired as exc:
        raise ExpiredTokenError(str(exc)) from exc
    except BadSignature as exc:
        raise InvalidTokenError(str(exc)) from exc

    uid = payload.get("uid")
    if not isinstance(uid, int):
        raise InvalidTokenError("payload missing uid")

    token_hash = _hash(token)
    with transaction.atomic():
        row = (
            PasswordSetupToken.objects
            .select_for_update()
            .filter(user_id=uid, token_hash=token_hash)
            .first()
        )
        if row is None:
            raise InvalidTokenError("token not found")
        if row.used_at is not None:
            raise ConsumedTokenError("token already used")
        if row.expires_at <= timezone.now():
            raise ExpiredTokenError("token expired (db row)")

        row.used_at = timezone.now()
        row.save(update_fields=["used_at"])

    return User.objects.get(pk=uid)
```

```python
import logging
import re

from auditlog.context import set_actor
from django.conf import settings
from django.core.mail import EmailMultiAlternatives
from django.db import transaction
from django.template.loader import render_to_string
from django.utils import timezone

# Existing imports above are unchanged.
from apps.accounts.models import Application, Address, CaregiverLink, User
from apps.accounts.services.tokens import issue_password_setup_token
from apps.accounts.services.users import create_user
from apps.dietary.models import Allergy, DietPreference, UserAllergy, UserDietPreference


logger = logging.getLogger(__name__)


# ---------- approve ----------

def approve_application(application: Application, admin_user: User) -> User:
    """Approve a submitted application and explode it into real rows.

    All writes are atomic. The welcome email is dispatched on commit so a
    rollback (any exception in the block) silently drops the message.
    """
    if application.status != Application.STATUS_SUBMITTED:
        raise ValueError(
            f"Application {application.id} is not submitted (status={application.status})"
        )

    with set_actor(admin_user), transaction.atomic():
        member = _create_member_user(application)
        caregiver, caregiver_was_existing = _create_or_reuse_caregiver(application)
        _create_member_address(member, application)
        _attach_diet_and_allergies(member, application)
        if caregiver is not None:
            _link_caregiver(member, caregiver, application.relationship)

        application.status = Application.STATUS_APPROVED
        application.approved_by = admin_user.id
        application.approved_at = timezone.now()
        application.save(update_fields=["status", "approved_by", "approved_at", "updated_at"])

        token = issue_password_setup_token(member)

        transaction.on_commit(
            lambda: _send_welcome_email(
                member=member,
                application=application,
                token=token,
                caregiver=caregiver,
                caregiver_was_existing=caregiver_was_existing,
            )
        )

    return member


def _create_member_user(application: Application) -> User:
    user = create_user(
        email=application.email,
        password=User.objects.make_random_password(length=32),  # never used
        full_name=application.full_name,
        role="member",
        is_active=False,
        dob=application.dob,
    )
    user.set_unusable_password()
    user.save(update_fields=["password"])
    return user


def _create_or_reuse_caregiver(application: Application):
    if not application.applying_for_other:
        return None, False

    email_norm = (application.caregiver_email or "").strip().lower()
    if not email_norm:
        return None, False

    existing = User.objects.filter(email__iexact=email_norm).first()
    if existing is not None:
        return existing, True

    caregiver = create_user(
        email=email_norm,
        password=User.objects.make_random_password(length=32),
        full_name=application.caregiver_full_name or email_norm,
        role="caregiver",
        is_active=False,
    )
    caregiver.set_unusable_password()
    caregiver.save(update_fields=["password"])
    return caregiver, False


def _create_member_address(member: User, application: Application) -> Address:
    if not application.city_id:
        raise ValueError("Application has no city_id; address cannot be created")

    return Address.objects.create(
        user=member,
        label=application.address_label or "Home",
        postal_code=application.postal_code or "",
        city_id=application.city_id,
    )


def _attach_diet_and_allergies(member: User, application: Application) -> None:
    diet_ids = application.dietary_ids or []
    if diet_ids:
        UserDietPreference.objects.bulk_create(
            [
                UserDietPreference(user=member, diet_preference_id=pid)
                for pid in diet_ids
            ],
            ignore_conflicts=True,
        )

    allergy_ids = application.allergy_ids or []
    if allergy_ids:
        UserAllergy.objects.bulk_create(
            [UserAllergy(user=member, allergy_id=aid) for aid in allergy_ids],
            ignore_conflicts=True,
        )


def _link_caregiver(member: User, caregiver: User, relationship: str | None) -> None:
    CaregiverLink.objects.get_or_create(

        caregiver=caregiver,

def reject_application(application: Application, admin_user: User, *, reason: str) -> Application:
    reason_clean = (reason or "").strip()
    if not reason_clean:
        raise ValueError("Rejection reason is required")

    with set_actor(admin_user), transaction.atomic():
        application.status = Application.STATUS_REJECTED
        application.rejected_reason = reason_clean
        application.approved_by = admin_user.id
        application.approved_at = timezone.now()
        application.save(
            update_fields=[
                "status",
                "rejected_reason",
                "approved_by",
                "approved_at",
                "updated_at",
            ]
        )

        transaction.on_commit(
            lambda: _send_rejection_email(application=application, reason=reason_clean)
        )

    return application
```

```python
@role_required("admin")
def admin_application_detail(request, pk: int):
    application = get_object_or_404(Application, pk=pk)
    suburb = (
        City.objects.filter(pk=application.city_id).values_list("name", flat=True).first()
        if application.city_id else None
    )
    diet_chips = list(
        DietPreference.objects
        .filter(id__in=application.dietary_ids or [])
        .values_list("name", flat=True)
    )
    allergy_chips = list(
        Allergy.objects
        .filter(id__in=application.allergy_ids or [])
        .values_list("name", flat=True)
    )
    return render(
        request,
        "dashboards/admin/application_detail.html",
        {
            "application": application,
            "suburb": suburb,
            "diet_chips": diet_chips,
            "allergy_chips": allergy_chips,
            "reject_error": request.GET.get("reject_error", ""),
        },
    )


@role_required("admin")
@require_POST
def admin_application_approve(request, pk: int):
    application = get_object_or_404(Application, pk=pk)
    try:
        approve_application(application, request.user)
    except ValueError as exc:
        messages.error(request, str(exc))
        return redirect(
            reverse("dashboards:admin_application_detail", args=[application.id])
        )
    messages.success(request, f"Approved {application.full_name}.")
    return redirect(reverse("dashboards:admin_applications"))


@role_required("admin")
@require_POST
def admin_application_reject(request, pk: int):
    application = get_object_or_404(Application, pk=pk)
    reason = (request.POST.get("reason") or "").strip()
    if not reason:
        # Re-render the detail page with an inline error.
        suburb = (
            City.objects.filter(pk=application.city_id).values_list("name", flat=True).first()
            if application.city_id else None
        )
        return render(
            request,
            "dashboards/admin/application_detail.html",
            {
                "application": application,
                "suburb": suburb,
                "diet_chips": [],
                "allergy_chips": [],
                "reject_error": "Please provide a reason.",
            },
            status=200,
        )

    try:
        reject_application(application, request.user, reason=reason)
    except ValueError as exc:
        messages.error(request, str(exc))
        return redirect(
            reverse("dashboards:admin_application_detail", args=[application.id])
        )
    messages.success(request, f"Rejected {application.full_name}.")
    return redirect(reverse("dashboards:admin_applications"))
```

- **Email send before commit.** If the welcome email is dispatched inline (not via `transaction.on_commit`), an SMTP error in a long transaction causes a partial state: the rows aren't committed but the SMTP server has already enqueued the message. Use `on_commit` and verify the `test_approve_rolls_back_when_address_write_fails` test sees `mail.outbox == []`.
- **`set_unusable_password()` called after `create_user`.** `create_user` hashes whatever we passed in. We then immediately mark the password unusable so the only way in is the welcome link. If you skip this step, the random 32-char password we passed becomes valid and the user could log in without ever seeing the token.
- **Audit log actor.** Without `set_actor(admin_user)`, `django-auditlog` records the change with `actor=None` and a future reviewer can't tell who approved. Wrap the whole atomic block.
- **`Application.approved_by` is a `BigIntegerField`, not a real FK** (Story 1.7 design). Assigning `application.approved_by = admin_user` would store the `User.__repr__` — wrong. Use `admin_user.id`.
- **Existing-caregiver match is case-insensitive.** `User.objects.filter(email__iexact=...)`. If you use `=`, a caregiver who originally registered as `Jane@Example.com` is missed and a duplicate row is created.
- **Token hash storage.** The raw signed token never lives in the DB. If a reviewer greps for the token in the `password_setup_tokens` table and finds it verbatim, fix the issue (`token_hash = sha256(token)`).
- **`CaregiverLink` unique constraint.** Story 1.3 imposed `unique_together(member_id, caregiver_id)`. The `_link_caregiver` helper uses `get_or_create` so re-approving an application doesn't crash on the second run.
- **Welcome cc to caregiver only when applicable.** If `caregiver_email == member_email` (data-entry error), the cc would duplicate the to-address. Guard against that.
- **`through-table` writes for diet/allergy require the user row to exist first.** Story 1.5 left the FK direction as `user → diet_preference`. Make sure `_create_member_user` runs before `_attach_diet_and_allergies` (it does in the order above; don't reorder for "cleanliness").
- **`PasswordSetupToken.user` FK on_delete.** `CASCADE` is correct — deleting the user (soft-delete is the actual operation, but hard-delete in tests) should remove their pending tokens.
- **Reject reason whitespace-only.** `"   "` must be treated as empty. The service strips and re-checks; the view does the same. If only one of them does, the inconsistency leaks.
- **`messages.success` requires the `django.contrib.messages` middleware.** Sprint 01 already enabled it; if a future revert breaks this, the redirects still work but the flash won't render. Reviewer will look at the redirected page and confirm the success banner shows.

## User story

> **As an** admin
> **I want** to see a list of submitted applications, newest first
> **so that** I know what is waiting for me and can clear the queue.

## Acceptance criteria

- [ ] Route `GET /admin/applications/` (NOT inside the Django admin; a dedicated tailored screen under the `dashboards` app).
- [ ] Restricted by `@role_required('admin')` from `apps.core.decorators`. Anonymous users redirect to login; non-admin authenticated users get HTTP 403.
- [ ] Lists `Application` rows where `status='submitted'`, ordered `-created_at`.
- [ ] Each row renders: `full_name`, `email`, suburb (from `city.name`, looked up via `Application.city_id`), dietary chips (resolved from `Application.dietary_ids` JSON), `created_at` ("submitted-at"), an "Applying for other" badge when `applying_for_other=True`.
- [ ] Filter `?city=<id>` narrows the list to applications with `city_id=<id>`.
- [ ] Filter `?has_allergies=1` narrows the list to applications whose `allergy_ids` JSON list is non-empty.
- [ ] Both filters compose (`?city=2&has_allergies=1`) and are reflected back into the form widgets.
- [ ] The city dropdown is populated from all `City` rows ordered by name.
- [ ] Each row is wrapped in `<a href="...">` pointing at the detail URL (Story 1.12 implements the detail view — this story stubs the URL `/admin/applications/<id>/` and returns a placeholder `HttpResponse("detail TODO")`).
- [ ] Desktop-primary layout: at viewports ≥ 1024 px the rows render as a table; at < 1024 px they collapse to stacked cards (still readable, no horizontal scroll).
- [ ] Empty state: when no rows match, render "No applications waiting." (don't show an empty table).
- [ ] Playwright at 1024 px asserts the rendered list contains a known application and that clicking a row navigates to the detail URL.

## Tech / library notes

- The `admin/applications/` URL collides namespace-wise with Django admin (`/admin/`). Avoid the collision by mounting the dashboards admin routes at `/admin/applications/` *after* `path("admin/", admin.site.urls)` in `config/urls.py` is moved to a longer prefix — or, simpler, mount this story's routes at `/admin/applications/` via a sibling include that Django resolves first. See Task 4.
- `Application.dietary_ids` is a JSON list of `DietPreference.id`. Resolve them in the view (single `DietPreference.objects.filter(id__in=...)` query per page), then attach `.dietary_chips: list[str]` to each application as a non-DB attribute. Same for the `has_allergies` filter.
- The "has allergies" filter uses MySQL `JSON_LENGTH(allergy_ids) > 0`. Django 5 exposes this via `Func`. The simplest path is `.exclude(allergy_ids=[])` which works because the JSONField default is `list` — but a Python-side fallback (`if app.allergy_ids:`) is acceptable for v1 if the SQL form fights you.
- Use `select_related` is not possible (`city_id` is a plain `BigIntegerField`, not a real FK in the `Application` model — Story 1.7 design). Instead, pre-fetch the city names with one extra query: `cities = dict(City.objects.values_list("id", "name"))`.
- City lookup table: `apps.accounts.models.City` (Story 1.1).

- [ ] **Step 2.3 — Mount the routes in `config/urls.py`.**

The Django admin lives at `/admin/`. We want **dashboard** admin routes at `/admin/applications/...` *without* clashing. The cleanest approach: register them explicitly via the `dashboards` namespace under their own prefix.

Open `config/urls.py` and add (above or below the existing patterns — order doesn't matter because Django admin matches `/admin/` exact + child URLConfs only on `/admin/login/`, `/admin/<app>/...`):

```python
path(
    "admin/",
    include(("apps.dashboards.urls.admin", "dashboards"), namespace="dashboards"),
),
```

> Django resolves the longest match first within a single `include`. `/admin/applications/` will hit our list view; `/admin/login/` still resolves to Django admin because that pattern is registered separately.
>
> If your repo namespaces `dashboards` elsewhere, drop the explicit `namespace=` arg above and ensure `app_name = "dashboards"` lives in `apps/dashboards/urls/__init__.py` (Step 2.2 does this).

```python
"""Admin approval queue — list + detail stub.

The list view (`admin_applications_list`) renders the queue of submitted
applications with two filters. The detail view (`admin_application_detail`)
is a stub here; Story 1.12 replaces it with the real approve/reject page.

Read-only: this story does not mutate any rows.
"""
from django.http import HttpResponse
from django.shortcuts import render

from apps.accounts.models import Application, City
from apps.core.decorators import role_required
from apps.dietary.models import Allergy, DietPreference


def _resolve_diet_chips(applications):
    """Attach `.dietary_chips` (list[str]) to each application in-place,
    using a single DietPreference query."""
    ids = {pid for app in applications for pid in (app.dietary_ids or [])}
    names = dict(
        DietPreference.objects.filter(id__in=ids).values_list("id", "name")
    )
    for app in applications:
        app.dietary_chips = [
            names[pid] for pid in (app.dietary_ids or []) if pid in names
        ]


def _resolve_allergy_chips(applications):
    """Attach `.allergy_chips` (list[str]) for the row tooltip."""
    ids = {aid for app in applications for aid in (app.allergy_ids or [])}
    names = dict(
        Allergy.objects.filter(id__in=ids).values_list("id", "name")
    )
    for app in applications:
        app.allergy_chips = [
            names[aid] for aid in (app.allergy_ids or []) if aid in names
        ]


def _resolve_suburbs(applications):
    """Attach `.suburb` (str | None) — the City name for `city_id`."""
    ids = {app.city_id for app in applications if app.city_id}
    names = dict(City.objects.filter(id__in=ids).values_list("id", "name"))
    for app in applications:
        app.suburb = names.get(app.city_id) if app.city_id else None


@role_required("admin")
def admin_applications_list(request):
    qs = (
        Application.objects
        .filter(status=Application.STATUS_SUBMITTED)
        .order_by("-created_at", "-id")
    )

    selected_city = request.GET.get("city")
    has_allergies = request.GET.get("has_allergies") == "1"

    if selected_city:
        try:
            qs = qs.filter(city_id=int(selected_city))
        except (TypeError, ValueError):
            selected_city = ""

    if has_allergies:
        # Python-side fallback for non-empty JSON arrays. Cheap because the
        # queue is small (admins clear it daily).
        applications = [a for a in qs if a.allergy_ids]
    else:
        applications = list(qs)

    _resolve_suburbs(applications)
    _resolve_diet_chips(applications)
    _resolve_allergy_chips(applications)

    context = {
        "applications": applications,
        "cities": list(City.objects.order_by("name")),
        "selected_city": int(selected_city) if selected_city else None,
        "has_allergies": has_allergies,
    }
    return render(
        request, "dashboards/admin/applications_list.html", context
    )


@role_required("admin")
def admin_application_detail(request, pk: int):
    """Stub. Story 1.12 implements the real approve/reject page."""
    # Return 501 to make it obvious in dev that the page isn't built yet.
    # Tests only assert the URL resolves.
    return HttpResponse(
        f"<h1>Application {pk}</h1><p>Detail page lands in Story 1.12.</p>",
        status=501,
        content_type="text/html; charset=utf-8",
    )
```

```typescript
import { expect, test } from '@playwright/test';

test.describe('admin applications queue', () => {
  test.use({ viewport: { width: 1024, height: 800 } });

  test('admin sees submitted applications in newest-first order', async ({ page }) => {
    // Pre-condition: an admin user `ops@merrymeal.test` (password `pw`)
    // exists and there is at least one submitted Application in the dev DB.
    // Seed via:
    //   docker compose exec web python manage.py shell -c "
    //     from apps.accounts.services.users import create_user
    //     from apps.accounts.models import Application
    //     create_user(email='ops@merrymeal.test', password='pw',
    //                 full_name='Ops Admin', role='admin')
    //     Application.objects.create(
    //       full_name='Margaret Whitlock', email='margaret@example.com',
    //       dob='1940-01-01', status='submitted')
    //   "
    await page.goto('/accounts/login/');
    await page.getByLabel('Email').fill('ops@merrymeal.test');
    await page.getByLabel('Password').fill('pw');
    await page.getByRole('button', { name: /sign in|log in/i }).click();

    await page.goto('/admin/applications/');

    await expect(page.getByRole('heading', { name: 'Applications' })).toBeVisible();
    await expect(page.getByText('Margaret Whitlock')).toBeVisible();

    // Click the row → URL navigates to the detail page (a stub for now).
    await page.getByText('Margaret Whitlock').click();
    await expect(page).toHaveURL(/\/admin\/applications\/\d+\/$/);
  });

  test('city filter narrows the list', async ({ page }) => {
    await page.goto('/accounts/login/');
    await page.getByLabel('Email').fill('ops@merrymeal.test');
    await page.getByLabel('Password').fill('pw');
    await page.getByRole('button', { name: /sign in|log in/i }).click();

    await page.goto('/admin/applications/?city=999999');  // unlikely id

  });
});
```

## Pitfalls reviewer will check

- **URL clash with Django admin.** Both this story and `django.contrib.admin` register routes under `/admin/`. Confirm `/admin/applications/` hits the new view AND `/admin/` (Django admin login) still works. Re-run `python3 manage.py shell -c "from django.urls import resolve; print(resolve('/admin/login/').func)"` — it must point at `django.contrib.admin.sites.login`, not our view.
- **`Application.city_id` is a `BigIntegerField`, not a real FK.** Story 1.7 documented this. Do not try `app.city.name` — it doesn't exist. Use the suburb-resolution helper instead.
- **JSON `allergy_ids` filter at the ORM layer.** `.exclude(allergy_ids=[])` works on Django 5 + MySQL 8 in *most* configurations but has historically been flaky on JSONField semantics for the empty-list case. The Python-side filter in `admin_applications_list` is intentional — don't "optimise" it back to ORM-only without a test that proves the corner cases.
- **N+1 city lookups.** If you render `{{ app.suburb }}` and that triggers a per-row `City.objects.get(...)`, the page goes from one query to N. Use the bulk dict (`_resolve_suburbs`) — the test for 100 applications + 5 cities should run in fewer than 10 SQL queries (`pytest --django-db-bench` or `assertNumQueries`).
- **Filter values reflecting back into widgets.** When `?city=2` is set, the `<option value="2">` for Melbourne must render `selected`. Same for the `has_allergies` checkbox. If a reviewer hits the URL with both set, then clicks Reset, the form must clear.
- **Empty state vs. "no rows match filter".** Both render the same "No applications waiting." message — that's intentional and simpler than two strings. If product disagrees, add a separate "No applications match these filters." string and key off `request.GET`.
- **`role_required('admin')` decorator** redirects anonymous users to `accounts:login`. If that namespace isn't yet declared (it is, since Story 0.5), the tests trip on `NoReverseMatch`. Don't shortcut — fix the namespace.
- **Pagination not in scope.** The Sprint-03 admin demo expects a handful of applications. If the queue grows past 50, add `?page=` in a follow-up story — don't gold-plate here.

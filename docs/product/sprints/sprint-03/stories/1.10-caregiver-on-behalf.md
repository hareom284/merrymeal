## User story

> **As a** caregiver
> **I want** to apply on behalf of someone I look after, declaring the relationship
> **so that** I can enrol my elderly relative who can't use a computer.

## Acceptance criteria

- [ ] Step 1 of the application wizard renders a checkbox "I'm applying for someone else".
- [ ] When the checkbox is ON, step 1 reveals four extra fields:
  - `caregiver_full_name` (required when ON)
  - `caregiver_email` (required when ON, validated as email)
  - `caregiver_phone` (optional)
  - `relationship` (required when ON; dropdown: `family` / `friend` / `nurse` / `social_worker` / `other`)
- [ ] The existing step-1 fields (`full_name`, `email`, `dob`, `phone`) continue to apply — they now represent the **member's** contact details.
- [ ] When the checkbox is OFF, the four caregiver fields are not required and stay hidden via Alpine.js.
- [ ] On valid POST with `applying_for_other=True`, `create_draft_application` persists `applying_for_other=True`, `caregiver_full_name`, `caregiver_email` (lower-cased), `caregiver_phone`, `relationship` on the `Application` row.
- [ ] On valid POST with `applying_for_other=False`, all four caregiver columns remain NULL.
- [ ] Validation: when `applying_for_other=True` and the **member's** `email` matches an existing `users.email`, the form rejects with "An account already exists for that email" (unchanged from Story 1.7).
- [ ] Validation: when `applying_for_other=True` and the **caregiver's** `caregiver_email` matches an existing `users.email`, the form does **NOT** reject — instead it sets a flag `existing_caregiver=True` on the cleaned data so that:
  - the row is still saved (status `draft`);
  - the confirmation banner (rendered on step 2/step 3 and the done page) prompts the existing caregiver to log in to confirm the link.
- [ ] Mobile-first 375 px, 44 px touch targets, base font 18 px on form rows.
- [ ] Playwright covers two happy paths:
  - applying for someone else with a **brand-new** caregiver email;
  - applying for someone else with an **existing** caregiver email (login-prompt banner visible).

## Tech / library notes

- The `applications` table already has `applying_for_other`, `caregiver_full_name`, `caregiver_email`, `caregiver_phone`, `relationship` columns — they were added in the Story 1.7 migration. **No new migration required for this story.**
- The "existing caregiver" detection happens at form-clean time but does **not** mutate any other table — the actual `CaregiverLink` row is created on approval in Story 1.12, possibly against the existing caregiver row.
- Persist `existing_caregiver` on the **session** (`request.session["existing_caregiver"] = True`) rather than on the `Application` row. It is a UI hint for the wizard banners — the source of truth at approval time is `User.objects.filter(email=...)`.
- Alpine.js: a single `x-data="{ forOther: false }"` boolean on the form. The caregiver fieldset uses `x-show="forOther"` and `x-cloak` so it doesn't flash on load.
- `RELATIONSHIP_CHOICES` are already defined on the `Application` model (Story 1.7). The form reuses `Application.RELATIONSHIP_CHOICES`.

```python
from django import forms

from apps.accounts.models import Application, User


class ApplicationContactForm(forms.Form):
    """Step 1 of the application wizard — applicant (and optionally
    caregiver) contact details.

    When ``applying_for_other`` is checked, the caregiver fields become
    required. The "member email" still has to be free of any existing
    ``users.email`` row, but the caregiver email is *allowed* to match an
    existing user — we use that as a signal that the link should attach to
    that caregiver instead of creating a new account.
    """

    full_name = forms.CharField(
        max_length=255,
        widget=forms.TextInput(
            attrs={
                "class": "input",
                "autocomplete": "name",
                "placeholder": "Margaret Whitlock",
            }
        ),
    )
    email = forms.EmailField(
        widget=forms.EmailInput(
            attrs={
                "class": "input",
                "autocomplete": "email",
                "inputmode": "email",
                "placeholder": "you@example.com",
            }
        )
    )
    dob = forms.DateField(
        widget=forms.DateInput(
            attrs={"class": "input", "type": "date", "autocomplete": "bday"}
        )
    )
    phone = forms.CharField(
        required=False,
        max_length=32,
        widget=forms.TextInput(
            attrs={
                "class": "input",
                "inputmode": "tel",
                "autocomplete": "tel",
                "placeholder": "0400 000 000",
            }
        ),
    )

    # ----- caregiver-on-behalf toggle (Story 1.10) -----
    applying_for_other = forms.BooleanField(required=False)

    caregiver_full_name = forms.CharField(
        required=False,
        max_length=255,
        widget=forms.TextInput(attrs={"class": "input", "autocomplete": "name"}),
    )
    caregiver_email = forms.EmailField(
        required=False,
        widget=forms.EmailInput(
            attrs={
                "class": "input",
                "autocomplete": "email",
                "inputmode": "email",
            }
        ),
    )
    caregiver_phone = forms.CharField(
        required=False,
        max_length=32,
        widget=forms.TextInput(
            attrs={"class": "input", "inputmode": "tel", "autocomplete": "tel"}
        ),
    )
    relationship = forms.ChoiceField(
        required=False,
        choices=[("", "Select…")] + Application.RELATIONSHIP_CHOICES,
        widget=forms.Select(attrs={"class": "input"}),
    )

    def clean_email(self) -> str:
        email = self.cleaned_data["email"].strip().lower()
        if User.objects.filter(email__iexact=email).exists():
            raise forms.ValidationError("An account already exists for that email")
        return email

    def clean_caregiver_email(self) -> str:
        raw = self.cleaned_data.get("caregiver_email") or ""
        return raw.strip().lower()

    def clean(self):
        cleaned = super().clean()
        for_other = bool(cleaned.get("applying_for_other"))
        cleaned["applying_for_other"] = for_other

        if for_other:
            required = {
                "caregiver_full_name": "Caregiver name is required",
                "caregiver_email": "Caregiver email is required",
                "relationship": "Please select your relationship",
            }
            for field, message in required.items():
                if not cleaned.get(field):
                    self.add_error(field, message)

            # The caregiver email may match an existing user — that's a feature.
            care_email = cleaned.get("caregiver_email")
            if care_email and User.objects.filter(email__iexact=care_email).exists():
                cleaned["existing_caregiver"] = True
            else:
                cleaned["existing_caregiver"] = False
        else:
            # Strip the caregiver fields so the service doesn't persist stray data.
            for field in (
                "caregiver_full_name",
                "caregiver_email",
                "caregiver_phone",
                "relationship",
            ):
                cleaned[field] = None
            cleaned["existing_caregiver"] = False

        return cleaned
```

@require_http_methods(["GET", "POST"])
def application_step_1(request):
    if request.method == "POST":
        form = ApplicationContactForm(request.POST)
        if form.is_valid():
            data = form.cleaned_data
            existing_caregiver = data.pop("existing_caregiver", False)
            app = create_draft_application(
                full_name=data["full_name"],
                email=data["email"],
                dob=data["dob"],
                phone=data.get("phone"),
                applying_for_other=data.get("applying_for_other", False),
                caregiver_full_name=data.get("caregiver_full_name"),
                caregiver_email=data.get("caregiver_email"),
                caregiver_phone=data.get("caregiver_phone"),
                relationship=data.get("relationship"),
            )
            request.session["application_id"] = app.id
            request.session["existing_caregiver"] = bool(existing_caregiver)
            return redirect("/apply/address/")
    else:
        form = ApplicationContactForm()

    return render(
        request,
        "accounts/application/step_1.html",
        {"form": form, "step": 1},
    )

## Pitfalls reviewer will check

- **Caregiver email lower-cased twice.** Both `clean_caregiver_email` and `create_draft_application` lower-case it. That's intentional — direct service callers (tests, fixtures) must get the same normalisation as form callers.
- **Member email vs caregiver email confusion.** The member's `email` collides → reject. The caregiver's `caregiver_email` collides → accept and flag. If those rules are reversed, the test `test_form_rejects_member_email_collision_even_when_for_other` will catch it; do not "fix" the test.
- **`existing_caregiver` leaking into the next application.** The session flag must be cleared by Story 1.9's `submit_application` view (it already does `request.session.pop("application_id", None)`; add the same pop for `existing_caregiver` if your local view doesn't already clear it). Not a blocking change for 1.10's PR but flag it in review.
- **`applying_for_other` posted as `"on"` vs `"true"` vs `"True"`.** Django's `BooleanField` handles `on` (the HTML default) only when `required=False`. Don't mark the toggle as required.
- **Alpine `x-cloak` missing.** Without it, the caregiver fieldset flashes on initial page load before Alpine boots — bad for low-vision users on slow networks.
- **No migration in this story.** The schema is already complete from Story 1.7. If a reviewer asks "where's the migration?" — point them at `apps/accounts/migrations/000X_create_applications.py`.
- **`RELATIONSHIP_CHOICES` duplicated.** The form must import the constant from `Application`, not redefine it. Drift between the two locations is a real bug source.
- **`existing_caregiver` is NOT persisted on the `Application` row.** It is a UI-only session flag. The source of truth at approval time is `User.objects.filter(email=app.caregiver_email)` — that runs in Story 1.12.

## User story

> **As a** prospective member
> **I want** to start an application by giving my contact details
> **so that** I can be reached without phoning the office.

## Acceptance criteria

- [ ] `applications` table exists via migration; columns match the schema laid out in **Tech / library notes** below.
- [ ] `Application` model is **schema only** (fields + `Meta` + `__str__` + choice constants — no business methods).
- [ ] Route `GET /apply/` renders step 1 of the application wizard.
- [ ] Form fields: `full_name` (required), `email` (required, validated as email), `dob` (required, date), `phone` (optional, max 32 chars).
- [ ] `email` must not collide with an existing `users.email` (validation error: "An account already exists for that email").
- [ ] Two drafts may share an email **only if** both have `status='draft'` for ≤24 h (the more recent one supersedes — see service implementation).
- [ ] `POST /apply/` with valid data creates an `Application` row in `status='draft'`, stores its id in the session (key `application_id`), and redirects to `/apply/address/` (step 2 — built in Story 1.8; for now the redirect target may 404 — that is acceptable).
- [ ] Inline validation errors render at 375 px without horizontal clipping; touch targets ≥ 44 px; base font 18 px.
- [ ] The progress indicator partial (`_progress.html`) renders "Step 1 of 3" with the first dot active.
- [ ] Service function `create_draft_application(*, full_name, email, dob, phone=None, ...)` is the **only** code path that writes the row. Views call it; tests call it directly.

## Tech / library notes

**The `Application` model is brand-new — not present in `merrymeal_schema_corrected.sql`.** It is the working draft of a person *before* an admin has approved them into the `users` table. On approval (Story 1.12) the row is read once and translated into `users` + `user_addresses` + `diet_preference_user` + `allergy_user` rows.

Define columns precisely as follows:

| Column | Type | Notes |
|---|---|---|
| `id` | BIGINT PK | auto |
| `status` | enum `draft`/`submitted`/`approved`/`rejected` | default `draft` |
| `full_name` | varchar(255) | applicant (or, in caregiver-on-behalf flow, the **member's** name) |
| `email` | varchar(255) | member's email; not globally unique because rejected drafts may share it. Service enforces "no live duplicate". |
| `dob` | date | required at step 1 |
| `phone` | varchar(32) | nullable |
| `applying_for_other` | bool | default false (Story 1.10 toggles this) |
| `caregiver_full_name` | varchar(255) nullable | Story 1.10 |
| `caregiver_email` | varchar(255) nullable | Story 1.10 |
| `caregiver_phone` | varchar(32) nullable | Story 1.10 |
| `relationship` | enum (`family`/`friend`/`nurse`/`social_worker`/`other`) nullable | Story 1.10 |
| `address_label` | varchar(120) nullable | Story 1.8 |
| `street` | varchar(255) nullable | Story 1.8 |
| `postal_code` | varchar(20) nullable | Story 1.8 |
| `city_id` | FK → `cities.id` nullable | Story 1.8 |
| `dietary_ids` | JSON (list of `diet_preferences.id`) | Story 1.9 — JSON because it's transient |
| `allergy_ids` | JSON (list of `allergies.id`) | Story 1.9 |
| `approved_by` | FK → `users.id` nullable | Story 1.12 |
| `approved_at` | datetime nullable | Story 1.12 |
| `rejected_reason` | text nullable | Story 1.12 |
| `created_at` | timestamp | auto |
| `updated_at` | timestamp | auto |

> **Why JSON for dietary/allergy ids?** Until approval there's no `users` row to FK from, so a through-table is impossible. JSON is fine for a transient working draft. On approval the service explodes the lists into real `diet_preference_user` / `allergy_user` rows.

The `_progress.html` partial accepts a single int context var `step` (1, 2, or 3).

Sessions: use `request.session["application_id"]`. The session cookie is HTTP-only by Django default — that is sufficient.

### Task 2: Define the `Application` model (GREEN — part 1)

- [ ] **Step 2.1 — Create `apps/accounts/models/applications.py`.**

```python
"""Application — the draft row created before an admin approves a member.

Convention: **Application = draft, User = approved.** An Application row is a
transient working copy. On approval (Story 1.12) it is read once and the data
is exploded into real rows: `users`, `user_addresses`, `diet_preference_user`,
`allergy_user`, and (if applying-for-other) `member_caregivers`.

This model is schema-only (fields, choices, Meta, __str__). Anything that
mutates state lives in apps.accounts.services.applications.
"""
from django.db import models


class Application(models.Model):
    STATUS_DRAFT = "draft"
    STATUS_SUBMITTED = "submitted"
    STATUS_APPROVED = "approved"
    STATUS_REJECTED = "rejected"
    STATUS_CHOICES = [
        (STATUS_DRAFT, "Draft"),
        (STATUS_SUBMITTED, "Submitted"),
        (STATUS_APPROVED, "Approved"),
        (STATUS_REJECTED, "Rejected"),
    ]

    RELATIONSHIP_CHOICES = [
        ("family", "Family"),
        ("friend", "Friend"),
        ("nurse", "Nurse"),
        ("social_worker", "Social worker"),
        ("other", "Other"),
    ]

    # ---- step 1: applicant contact ----
    full_name = models.CharField(max_length=255)
    email = models.EmailField(max_length=255)
    dob = models.DateField()
    phone = models.CharField(max_length=32, null=True, blank=True)

    # ---- step 1 (Story 1.10): caregiver-on-behalf toggle ----
    applying_for_other = models.BooleanField(default=False)
    caregiver_full_name = models.CharField(max_length=255, null=True, blank=True)
    caregiver_email = models.EmailField(max_length=255, null=True, blank=True)
    caregiver_phone = models.CharField(max_length=32, null=True, blank=True)
    relationship = models.CharField(
        max_length=20, choices=RELATIONSHIP_CHOICES, null=True, blank=True
    )

    # ---- step 2 (Story 1.8): address ----
    address_label = models.CharField(max_length=120, null=True, blank=True)
    street = models.CharField(max_length=255, null=True, blank=True)
    postal_code = models.CharField(max_length=20, null=True, blank=True)
    city_id = models.BigIntegerField(null=True, blank=True)

    # ---- step 3 (Story 1.9): dietary + allergies ----
    dietary_ids = models.JSONField(default=list, blank=True)
    allergy_ids = models.JSONField(default=list, blank=True)

    # ---- workflow ----
    status = models.CharField(
        max_length=20, choices=STATUS_CHOICES, default=STATUS_DRAFT
    )
    approved_by = models.BigIntegerField(null=True, blank=True)
    approved_at = models.DateTimeField(null=True, blank=True)
    rejected_reason = models.TextField(null=True, blank=True)

    created_at = models.DateTimeField(auto_now_add=True)
    updated_at = models.DateTimeField(auto_now=True)

    class Meta:
        app_label = "accounts"
        db_table = "applications"
        indexes = [
            models.Index(fields=["status", "-created_at"]),
            models.Index(fields=["email"]),
        ]

    def __str__(self) -> str:
        return f"Application<{self.id} {self.email} {self.status}>"
```

from datetime import date

from apps.accounts.models import Application, User


def create_draft_application(
    *,
    full_name: str,
    email: str,
    dob,
    phone: str | None = None,
    applying_for_other: bool = False,
    caregiver_full_name: str | None = None,
    caregiver_email: str | None = None,
    caregiver_phone: str | None = None,
    relationship: str | None = None,
) -> Application:
    """Create the step-1 draft row.

    Rules:
    - The member's email must not collide with any existing `users.email`
      (case-insensitive). A second pending draft with the same email is
      allowed: the latest draft is what step 2/3 continue to fill out.
    """
    email_norm = email.strip().lower()
    if User.objects.filter(email__iexact=email_norm).exists():
        raise ValueError(f"A user with email {email_norm} already exists")

    return Application.objects.create(
        full_name=full_name.strip(),
        email=email_norm,
        dob=dob if isinstance(dob, date) else dob,
        phone=(phone or "").strip() or None,
        applying_for_other=applying_for_other,
        caregiver_full_name=caregiver_full_name,
        caregiver_email=(caregiver_email or "").strip().lower() or None,
        caregiver_phone=caregiver_phone,
        relationship=relationship,
        status=Application.STATUS_DRAFT,
    )
```

from django import forms

from apps.accounts.models import User


class ApplicationContactForm(forms.Form):
    """Step 1 of the application wizard — applicant contact details.

    Format-level validation only. The service
    (`apps.accounts.services.create_draft_application`) re-checks the
    "email not already a user" rule so direct service callers stay safe.
    """

    full_name = forms.CharField(
        max_length=255,
        widget=forms.TextInput(
            attrs={
                "class": "input",
                "autocomplete": "name",
                "placeholder": "Margaret Whitlock",
            }
        ),
    )
    email = forms.EmailField(
        widget=forms.EmailInput(
            attrs={
                "class": "input",
                "autocomplete": "email",
                "inputmode": "email",
                "placeholder": "you@example.com",
            }
        )
    )
    dob = forms.DateField(
        widget=forms.DateInput(
            attrs={"class": "input", "type": "date", "autocomplete": "bday"}
        )
    )
    phone = forms.CharField(
        required=False,
        max_length=32,
        widget=forms.TextInput(
            attrs={
                "class": "input",
                "inputmode": "tel",
                "autocomplete": "tel",
                "placeholder": "0400 000 000",
            }
        ),
    )

    def clean_email(self) -> str:
        email = self.cleaned_data["email"].strip().lower()
        if User.objects.filter(email__iexact=email).exists():
            raise forms.ValidationError("An account already exists for that email")
        return email
```

from django.shortcuts import redirect, render
from django.views.decorators.http import require_http_methods

from apps.accounts.forms import ApplicationContactForm
from apps.accounts.services import create_draft_application


@require_http_methods(["GET", "POST"])
def application_step_1(request):
    if request.method == "POST":
        form = ApplicationContactForm(request.POST)
        if form.is_valid():
            app = create_draft_application(**form.cleaned_data)
            request.session["application_id"] = app.id
            return redirect("/apply/address/")
    else:
        form = ApplicationContactForm()

    return render(
        request,
        "accounts/application/step_1.html",
        {"form": form, "step": 1},
    )

import { expect, test } from '@playwright/test';

test('mobile applicant can complete step 1', async ({ page }) => {
  await page.goto('/apply/');

  // Progress indicator visible
  await expect(page.getByText('Step 1 of 3')).toBeVisible();

  // No horizontal scroll at 375 px
  const scrollWidth = await page.evaluate(() => document.documentElement.scrollWidth);
  expect(scrollWidth).toBeLessThanOrEqual(375);

  // Touch targets ≥ 44 px
  const submit = page.getByRole('button', { name: 'Continue' });
  const box = await submit.boundingBox();
  expect(box!.height).toBeGreaterThanOrEqual(44);

  // Happy path
  await page.getByLabel('Full name').fill('Margaret Whitlock');
  await page.getByLabel('Email').fill('margaret@example.com');
  await page.getByLabel('Date of birth').fill('1940-01-01');
  await page.getByLabel('Phone (optional)').fill('0400000000');

  await submit.click();
  await expect(page).toHaveURL(/\/apply\/address\/?$/);
});

## Pitfalls reviewer will check

- **Email normalisation.** Both form and service lowercase + strip — otherwise `Margaret@Example.com` slips past the duplicate-user check.
- **Business logic in the model.** If the `Application` class has any `save()` override, `submit()`, `approve()`, etc. — reject. Those belong in `services/applications.py` (Stories 1.9, 1.12).
- **Direct `Application.objects.create(...)` from the view.** The view must call the service. Tests reach into the service directly.
- **JSON default mutability.** `JSONField(default=list)` — the default must be the callable `list`, not the value `[]`, or every row shares the same list object.
- **Migration filename.** Auto-generated name OK; pin it in the commit so the staging DB applies the same name.
- **Session key collision.** `application_id` must not collide with any other session key in use. Grep `request.session[` across `apps/` before merging.
- **Namespace on `reverse("accounts:application_step_1")`.** If you mounted the urls outside the `accounts` namespace, the test breaks. Use `reverse()` once in a Django shell to confirm.
- **`/apply/address/` 404.** Acceptable until Story 1.8 lands — note it in the PR description so the reviewer doesn't think it's a regression.

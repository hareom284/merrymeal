## User story

> **As an** admin
> **I want** a single dashboard card per kitchen with three at-a-glance metrics
> **so that** I open one screen each morning and know whether anything needs intervention.

## Acceptance criteria

- [ ] Route `GET /dashboards/admin/kitchens/` renders one card per kitchen.
- [ ] Restricted by `@role_required('admin')` from `apps.core.decorators`. Anonymous → redirect to login; non-admin authenticated → 403.
- [ ] Each card shows three metrics:
  1. **Batches expiring within 3 days** — count, linked to `/admin/applications/...`'s sibling filtered list `/kitchen/stock/?kitchen=<id>&expiring=1` (a read-only view that already lives in Story 2.5/2.6; if not yet exposed, use the Django admin URL as the link target — see Tech notes).
  2. **Last 24 h food-safety pass-rate** — percentage with two decimal places (`{value}%`), linked to `/kitchen/safety/check/?kitchen=<id>&since=24h`.
  3. **Last food-safety failure** — `kitchen, check_type, checked_at` (or "No recent failures" if none in the last 7 days), linked to the failure's check detail (`/admin/food_safety/foodsafetycheck/<id>/change/` is acceptable until a dedicated detail screen exists).
- [ ] Traffic-light colours on each metric, based on per-card thresholds:
  - Expiring: green = 0; yellow = 1–`KITCHEN_EXPIRING_RED_THRESHOLD` (default 5); red = above the threshold.
  - Pass-rate: green = 100; yellow = `KITCHEN_PASSRATE_YELLOW_MIN` to 99.99 (default 90); red = below the yellow minimum. If no checks in the window → grey "no data".
  - Last failure: green = no failure in last 7 days; yellow = failure 1–7 days ago; red = failure in the last 24 h.
- [ ] Thresholds are read from `apps.dashboards.services.kitchen_summary` (constants) so they're easy to tune. Optionally overridable via `settings.KITCHEN_DASHBOARD_THRESHOLDS`.
- [ ] Mobile-readable; desktop-primary. At ≥ 1024 px, cards lay out in a CSS grid (`grid-cols-2 lg:grid-cols-3`); at 375 px they stack vertically.
- [ ] Pure read-only — this view does **not** mutate any data, write any audit log entries, or send any emails.
- [ ] Empty state (no kitchens): "No kitchens registered yet." (don't crash).
- [ ] Playwright at 1024 px asserts the page renders, the three metrics show, and clicking a metric navigates to its target URL.

## Tech / library notes

- The aggregation lives in a **service** module, not in the view. The view is thin: it fetches kitchens, calls `get_summary(k)` once per kitchen, and renders. This keeps the SQL surface easy to test in isolation and trivial to swap for a cached version later.
- For the expiring-batches count, reuse `apps.kitchens.services.expiry.find_expiring_batches(kitchen, within_days=3)` from Story 2.9. Calling `.count()` on the returned queryset is a single SQL query per kitchen.
- For pass-rate: count `FoodSafetyCheck` rows in the last 24 h, then count those with `result='pass'`, then divide. If denominator is 0 → return `None` and the template shows "no data".
- For last failure: `FoodSafetyCheck.objects.filter(kitchen=kitchen, result='fail').order_by('-checked_at').first()`. Index on `(kitchen_id, checked_at)` already exists per Story 2.7.
- Per-kitchen overall query count must stay ≤ 4 (expiring count, pass count, total count, last failure). For N kitchens the page issues ≤ 4N queries. That's fine for v1; if N > 50 we'll cache the page for 60 s in a follow-up.
- Thresholds (constants in the service file):
  ```python
  EXPIRING_RED_THRESHOLD = 5      # batches; > 5 = red
  PASSRATE_YELLOW_MIN = 90.0      # %; < 90 = red, 90-99.99 = yellow, 100 = green
  FAILURE_RED_HOURS = 24
  FAILURE_YELLOW_DAYS = 7
  ```
- Filter-list URLs: the kitchen-stock browser doesn't have a dedicated route yet — Story 2.5 left CRUD inside the Django admin. The card links to `/admin/kitchens/ingredientbatch/?kitchen__id__exact={id}&expiration_date__lte=...` (Django admin's built-in querystring filter). If a tailored screen lands later, swap the URL — the test just asserts `href` contains the kitchen id, not the exact path.

```python
"""Per-kitchen at-a-glance aggregation for the admin dashboard.

Pure read-only. No side effects. One function: ``get_summary(kitchen)``
returns a plain dict the template iterates over. Thresholds live as module
constants so a future settings override is a one-line change.
"""
from dataclasses import dataclass
from datetime import timedelta

from django.conf import settings
from django.utils import timezone

from apps.food_safety.models import FoodSafetyCheck
from apps.kitchens.models import Kitchen
from apps.kitchens.services.expiry import find_expiring_batches


# ---------- thresholds ----------

EXPIRING_WITHIN_DAYS = 3
EXPIRING_RED_THRESHOLD = 5          # > 5 expiring batches = red

PASSRATE_WINDOW_HOURS = 24
PASSRATE_YELLOW_MIN = 90.0          # 90.0 ≤ rate < 100 = yellow; < 90 = red

FAILURE_RED_HOURS = 24
FAILURE_YELLOW_DAYS = 7


def _override(name: str, default):
    """Allow per-environment tuning via settings.KITCHEN_DASHBOARD_THRESHOLDS."""
    overrides = getattr(settings, "KITCHEN_DASHBOARD_THRESHOLDS", {}) or {}
    return overrides.get(name, default)


# ---------- metric helpers ----------

def _expiring_status(count: int) -> str:
    if count == 0:
        return "green"
    if count <= _override("EXPIRING_RED_THRESHOLD", EXPIRING_RED_THRESHOLD):
        return "yellow"
    return "red"


def _pass_rate_status(rate: float | None) -> str:
    if rate is None:
        return "grey"
    if rate >= 100.0:
        return "green"
    if rate >= _override("PASSRATE_YELLOW_MIN", PASSRATE_YELLOW_MIN):
        return "yellow"
    return "red"


def _last_failure_status(failure: FoodSafetyCheck | None) -> str:
    if failure is None:
        return "green"
    now = timezone.now()
    if (now - failure.checked_at) <= timedelta(hours=_override("FAILURE_RED_HOURS", FAILURE_RED_HOURS)):
        return "red"
    if (now - failure.checked_at) <= timedelta(days=_override("FAILURE_YELLOW_DAYS", FAILURE_YELLOW_DAYS)):
        return "yellow"
    return "green"


# ---------- main ----------

@dataclass(frozen=True)
class _CardMetric:
    """Convenience container — the template treats it as a plain dict."""
    value: object
    status: str
    href: str
    label: str


def get_summary(kitchen: Kitchen) -> dict:
    """Aggregate the three glance metrics for one kitchen.

    Returns a dict with keys:
        kitchen, expiring_count, expiring_status, expiring_href,
        pass_rate, pass_rate_status, pass_rate_href,
        last_failure, last_failure_status, last_failure_href.
    """
    expiring_qs = find_expiring_batches(kitchen, within_days=EXPIRING_WITHIN_DAYS)
    expiring_count = expiring_qs.count()

    window_start = timezone.now() - timedelta(hours=PASSRATE_WINDOW_HOURS)
    checks = FoodSafetyCheck.objects.filter(
        kitchen=kitchen, checked_at__gte=window_start
    )
    total = checks.count()
    passed = checks.filter(result="pass").count()
    pass_rate = round((passed / total) * 100.0, 2) if total else None

    last_failure = (
        FoodSafetyCheck.objects
        .filter(kitchen=kitchen, result="fail")
        .order_by("-checked_at")
        .first()
    )

    # Link targets — see Tech notes for the rationale.
    expiring_href = (
        f"/admin/kitchens/ingredientbatch/"
        f"?kitchen__id__exact={kitchen.id}"
    )
    pass_rate_href = (
        f"/admin/food_safety/foodsafetycheck/"
        f"?kitchen__id__exact={kitchen.id}"
    )
    last_failure_href = (
        f"/admin/food_safety/foodsafetycheck/{last_failure.id}/change/"
        if last_failure else ""
    )

    return {
        "kitchen": kitchen,
        "expiring_count": expiring_count,
        "expiring_status": _expiring_status(expiring_count),
        "expiring_href": expiring_href,
        "pass_rate": pass_rate,
        "pass_rate_status": _pass_rate_status(pass_rate),
        "pass_rate_href": pass_rate_href,
        "last_failure": last_failure,
        "last_failure_status": _last_failure_status(last_failure),
        "last_failure_href": last_failure_href,
    }
```

## Pitfalls reviewer will check

- **Service does I/O outside aggregation.** If `get_summary` calls `send_mail`, writes to a log table, or hits an external HTTP API, reject. This view is a morning glance — it must be cheap and side-effect-free.
- **N+1 in the view.** If the test `test_query_count_stays_linear_in_n_kitchens` starts failing because someone added `{{ card.kitchen.partner.legal_name }}` to the template, fix the template (cache the value in the service) — don't loosen the test.
- **Pass-rate window vs. service window.** Story 2.9 talks about a 3-day window for expiry. This story uses a **24 h** window for pass-rate. The two windows are deliberately different — don't unify them "for symmetry".
- **`timezone.now()` vs `timezone.localdate()`.** Pass-rate is a wall-clock window (24 h ago), so `timezone.now() - timedelta(hours=24)`. Expiry is a calendar window (3 days from today), so `timezone.localdate() + timedelta(days=3)`. Don't mix.
- **Empty kitchen list.** If `Kitchen.objects.all()` returns nothing, the page must render the empty-state string, not crash on iterating `cards`. The view passes `has_kitchens` for this branch.
- **Traffic-light boundary off-by-one.** Pass-rate of exactly 90.0 is **yellow**, not red. Pass-rate of exactly 100.0 is **green**. The test `test_summary_pass_rate_yellow_band` pins this; do not "round down".
- **Last failure 7-day cutoff.** A failure exactly 7 days ago is yellow, not green. `>= timedelta(days=7)` → green; the `_last_failure_status` helper uses `<=` for the inclusive yellow band, so 7 days exact lands in yellow. Don't flip the comparison.
- **`FoodSafetyCheck` model not yet shipped on this branch.** Story 2.7 must be merged before 2.10. If the import fails, the dependency chain in the story header is wrong.
- **Linking to Django admin.** The `expiring_href` and `pass_rate_href` URLs point at Django admin URLs (`/admin/kitchens/ingredientbatch/?...`). If admin isn't enabled for kitchen_staff but is for admin users, these links work only for admins — which is fine because the whole view is admin-only.
- **Threshold overrides via settings.** `settings.KITCHEN_DASHBOARD_THRESHOLDS = {"EXPIRING_RED_THRESHOLD": 10}` should ripple through. The test suite doesn't cover this branch — flag it in review and add a follow-up if product wants to tune.
- **Card layout below 768 px.** Tailwind classes are `grid-cols-1 md:grid-cols-2 lg:grid-cols-3`. At 375 px the grid collapses to a single column. If a reviewer's screenshot at 375 px shows two cards side-by-side and clipped, the breakpoint is wrong — fix the `md:` to a higher number.

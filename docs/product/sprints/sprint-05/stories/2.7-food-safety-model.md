## User story

> **As a** kitchen staff member
> **I want** to record food-safety checks with the result and (where relevant) temperature
> **so that** audits have evidence and a failed check can block a meal plan.

## Tech / library notes

- **Schema-only models.** See README "Layering convention" — fields + Meta + `__str__`. Anything else is a code-review smell. Business rules (e.g. "temperature is required for `_temp` check types") belong in `apps.food_safety.forms.check.CheckForm.clean()` in Story 2.8.
- **`db_column` is required** on every FK. The SQL schema uses `kitchen_id` / `meal_plan_id` / `checked_by`; Django's default would be `kitchen_id` for `kitchen` FK (matches) but `checked_by_id` for `checked_by` FK (does **not** match). Always set `db_column` explicitly so the migration writes the right column name.
- **`meal_plan` forward reference.** `MealPlan` is not yet defined (Epic 03). Use the string app-label form `"meals.MealPlan"` so Django resolves the FK lazily. The column is nullable, so existing rows are valid even before the model lands.
- **Choice constants live on the model class** as `TextChoices` subclasses (`CheckType`, `Result`). External callers import them via `from apps.food_safety.models import FoodSafetyCheck` and read `FoodSafetyCheck.CheckType.STORAGE_TEMP`.
- **factory-boy** is already in `requirements.txt` (Sprint 01). New factories live next to the tests, not in a shared `factories/` package — one factory file per app keeps imports flat.
- **`pytest-django`** picks up tests from `apps/food_safety/tests/`. No extra config; the existing `pytest.ini` (`DJANGO_SETTINGS_MODULE=config.settings.test`) handles it.

```python
from django.conf import settings
from django.db import models


class FoodSafetyCheck(models.Model):
    """Audit row for one food-safety check on one kitchen at one moment.

    Schema-only model: fields + Meta + __str__. Business rules
    (e.g. temperature required for temp checks) live in forms/services.
    """

    class CheckType(models.TextChoices):
        STORAGE_TEMP = "storage_temp", "Storage temperature"
        COOKING_TEMP = "cooking_temp", "Cooking temperature"
        COLD_CHAIN = "cold_chain", "Cold chain"
        HYGIENE = "hygiene", "Hygiene"
        CLEANING = "cleaning", "Cleaning"
        PEST_CONTROL = "pest_control", "Pest control"

    class Result(models.TextChoices):
        PASS = "pass", "Pass"
        FAIL = "fail", "Fail"

    kitchen = models.ForeignKey(
        "kitchens.Kitchen",
        on_delete=models.PROTECT,
        db_column="kitchen_id",
        related_name="safety_checks",
    )
    meal_plan = models.ForeignKey(
        "meals.MealPlan",
        on_delete=models.PROTECT,
        db_column="meal_plan_id",
        related_name="safety_checks",
        null=True,
        blank=True,
    )
    check_type = models.CharField(max_length=20, choices=CheckType.choices)
    temperature_celsius = models.DecimalField(
        max_digits=5, decimal_places=2, null=True, blank=True
    )
    result = models.CharField(max_length=4, choices=Result.choices)
    checked_by = models.ForeignKey(
        settings.AUTH_USER_MODEL,
        on_delete=models.PROTECT,
        db_column="checked_by",
        related_name="safety_checks",
    )
    checked_at = models.DateTimeField()
    notes = models.TextField(null=True, blank=True)

    class Meta:
        db_table = "food_safety_checks"
        indexes = [
            models.Index(fields=["kitchen"], name="idx_fsc_kitchen"),
            models.Index(fields=["meal_plan"], name="idx_fsc_plan"),
        ]

    def __str__(self) -> str:
        return f"{self.check_type} @ {self.kitchen_id} = {self.result}"
```

## Pitfalls reviewer will check

- **No business logic in the model.** No `clean()`, no `save()` override, no helper methods. If you wrote `def is_failed(self)`, delete it — that lives in a service or template.
- **`db_column='checked_by'`** — without it the column becomes `checked_by_id` and the SQL schema breaks. Run `python3 manage.py sqlmigrate food_safety 0001` and grep for the column names.
- **`meal_plan` FK uses string `"meals.MealPlan"`** — `MealPlan` is not imported, so writing `meal_plan = models.ForeignKey(MealPlan, …)` crashes at import time.
- **`Meta.db_table = "food_safety_checks"`** (plural, with underscore). Django would otherwise emit `food_safety_foodsafetycheck`.
- **`PROTECT` not `CASCADE`** on every FK — losing a kitchen should not silently delete its safety audit trail. Compliance teams care about that.
- **Index names match the SQL schema.** `idx_fsc_kitchen`, `idx_fsc_plan`. Random Django-generated names like `food_safety_kitchen__1a2b3c` make the SQL diff noisy.
- **No `unique_together`.** Two checks of the same type on the same day are legal (morning + evening temperature). The form layer enforces "once per type per day" if needed (Story 2.8).

## User story

> **As an** admin
> **I want** an email every morning listing batches expiring within 3 days
> **so that** I can repurpose or discard them before they hit a member's meal.

## Acceptance criteria

- [ ] `apps/kitchens/services/expiry.py::find_expiring_batches(kitchen, within_days=3)` returns the `IngredientBatch` rows with `expiration_date <= today + within_days` AND `quantity > 0`, ordered by `(ingredient_id, expiration_date)`.
- [ ] `apps/kitchens/tasks/expiry_alerts.py::send_expiry_alerts()` is the Django-Q2 entry point. It:
  - iterates kitchens that have at least one expiring batch;
  - groups batches by ingredient;
  - sends **one email per kitchen** to that kitchen's admin contact (see notes);
  - sends **nothing** when the kitchen has zero matching batches;
  - is **idempotent within a calendar day** — re-running on the same day is a no-op email-wise.
- [ ] Idempotency is enforced by a new `ExpiryAlertLog` model:
  - Table `expiry_alert_logs`.
  - One row per `(kitchen_id, sent_date)` — `unique_together`.
  - Created inside the same DB transaction as the email send; if creation fails (race), the email is **not** sent.
- [ ] Django-Q2 `Schedule` row exists in a **data migration** so a fresh deploy registers the daily 06:00 Melbourne run automatically.
- [ ] Email templates `templates/kitchens/emails/expiring_batches.{html,txt}` render the kitchen name, the cutoff date, and one block per ingredient with batch rows (`lot_number`, `quantity`, `expiration_date`, days-until-expiry).
- [ ] `pytest` covers:
  - service returns the right batches;
  - task sends one email per kitchen;
  - task sends nothing when no batches match;
  - **task is idempotent**: calling it twice in the same day leaves `mail.outbox` length at 1.
  - the data migration creates a `Schedule` row with `cron='0 6 * * *'` and the correct task path.

## Tech / library notes

- **Django-Q2** is in `requirements.txt` (Sprint 01). Schedules are persisted in the `django_q_schedule` table; cluster process is the `worker` service in `compose.yaml` running `python manage.py qcluster`.
- **`Schedule` model lives in `django_q.models`.** Cron is a string like `'0 6 * * *'` evaluated in `settings.TIME_ZONE`. `settings.TIME_ZONE = 'Australia/Melbourne'` is set in Sprint 01; **verify** before relying on it (see Task 5 Step 5.1).
- **Email recipient.** v1 takes the first user with `role='admin'` (per Sprint plan risk row). The model has no per-kitchen contact yet. Fallback: if **no** admin exists, log a warning and skip the kitchen — never raise.
- **Idempotency design.** Two options:
  1. **`ExpiryAlertLog` table** (chosen). One row per kitchen per day. Survives cache flushes; auditable.
  2. Cache key. Rejected: rebooting Redis would re-send all emails the next morning.
- **The `ExpiryAlertLog` row is created BEFORE `send_mail` in a transaction.** Order matters: if the email send fails (SMTP outage), we want the log row to roll back so the next retry sends. If `IntegrityError` (someone else just created the row) fires, we silently skip — that is the "already sent today" branch.
- **Cutoff date is `timezone.localdate() + timedelta(days=within_days)`.** Local date in Melbourne. The query is `expiration_date <= cutoff AND quantity > 0`.
- **`quantity > 0`** — the field is `DECIMAL(10,2) NOT NULL` per the schema. A batch fully consumed (quantity=0) is irrelevant and would be noise.
- **Data migration uses `update_or_create`**, not `create`. Re-running `migrate` on a deploy must be safe. The lookup key is `func='apps.kitchens.tasks.expiry_alerts.send_expiry_alerts'`.
- **`pytz` vs `zoneinfo`** — Django 5 uses `zoneinfo` by default (`USE_DEPRECATED_PYTZ=False`). The `Schedule` row's cron is interpreted by Django-Q2 in `settings.TIME_ZONE`; no extra TZ arithmetic is needed here, but our tests **must** freeze time with the right zone (see `freezegun` examples).
- **`freezegun`** is in `requirements.txt`. Use `freeze_time("2026-06-02 06:00:00", tz_offset=10)` for Melbourne (AEST winter) — or `tz_offset=11` for AEDT summer. To stay tz-correct year-round, prefer `freeze_time(datetime(2026, 6, 2, 6, 0, tzinfo=ZoneInfo("Australia/Melbourne")))`.

```python
from datetime import timedelta

from django.db.models import QuerySet
from django.utils import timezone

from apps.kitchens.models import IngredientBatch


def find_expiring_batches(kitchen, *, within_days: int = 3) -> QuerySet[IngredientBatch]:
    """Batches in the given kitchen with quantity remaining whose expiry is
    today or within `within_days` days. Ordered by ingredient_id, expiration_date."""
    cutoff = timezone.localdate() + timedelta(days=within_days)
    return (
        IngredientBatch.objects
        .filter(
            kitchen=kitchen,
            expiration_date__lte=cutoff,
            quantity__gt=0,
        )
        .select_related("ingredient")
        .order_by("ingredient_id", "expiration_date")
    )
```

```python
from django.db import models


class ExpiryAlertLog(models.Model):
    """One row per (kitchen, sent_date). Used to make the daily expiry-alert
    task idempotent within a calendar day."""

    kitchen = models.ForeignKey(
        "kitchens.Kitchen",
        on_delete=models.CASCADE,
        db_column="kitchen_id",
        related_name="expiry_alert_logs",
    )
    sent_date = models.DateField()
    sent_at = models.DateTimeField(auto_now_add=True)

    class Meta:
        db_table = "expiry_alert_logs"
        constraints = [
            models.UniqueConstraint(
                fields=["kitchen", "sent_date"],
                name="uniq_alert_kitchen_day",
            ),
        ]

    def __str__(self) -> str:
        return f"alert log {self.kitchen_id} {self.sent_date}"
```

```python
import logging
from collections import defaultdict
from datetime import timedelta

from django.core.mail import EmailMultiAlternatives
from django.db import IntegrityError, transaction
from django.template.loader import render_to_string
from django.utils import timezone

from apps.accounts.models import User
from apps.kitchens.models import ExpiryAlertLog, Kitchen
from apps.kitchens.services.expiry import find_expiring_batches


logger = logging.getLogger(__name__)


def _recipient_email() -> str | None:
    """v1: first active user with role='admin'. Returns None if none exists."""
    admin = (
        User.objects
        .filter(role="admin", is_active=True)
        .order_by("id")
        .first()
    )
    return admin.email if admin else None


def _group_by_ingredient(batches):
    grouped: dict[int, list] = defaultdict(list)
    for b in batches:
        grouped[b.ingredient_id].append(b)
    # Preserve the order set by the service (ingredient_id ASC).
    return [(batches_for[0].ingredient, batches_for) for batches_for in grouped.values()]


def _send_for_kitchen(kitchen: Kitchen, *, recipient: str, today) -> bool:
    batches = list(find_expiring_batches(kitchen, within_days=3))
    if not batches:
        return False

    cutoff = today + timedelta(days=3)
    context = {
        "kitchen": kitchen,
        "today": today,
        "cutoff": cutoff,
        "ingredient_groups": _group_by_ingredient(batches),
    }

    try:
        with transaction.atomic():
            # Idempotency latch: this raises IntegrityError on second call same day.
            ExpiryAlertLog.objects.create(kitchen=kitchen, sent_date=today)

            subject = f"Expiring batches at {kitchen.name}"
            txt_body = render_to_string("kitchens/emails/expiring_batches.txt", context)
            html_body = render_to_string("kitchens/emails/expiring_batches.html", context)

            msg = EmailMultiAlternatives(subject, txt_body, to=[recipient])
            msg.attach_alternative(html_body, "text/html")
            msg.send(fail_silently=False)
    except IntegrityError:
        logger.info("expiry alert already sent today for kitchen=%s", kitchen.id)
        return False

    return True


def send_expiry_alerts():
    """Django-Q2 entry point. Runs daily at 06:00 Melbourne via the Schedule
    row created in migration `00XX_schedule_expiry_alerts`."""
    today = timezone.localdate()
    recipient = _recipient_email()

    for kitchen in Kitchen.objects.all().order_by("id"):
        if recipient is None:
            logger.warning("no admin recipient configured; skipping kitchen=%s", kitchen.id)
            continue
        _send_for_kitchen(kitchen, recipient=recipient, today=today)
```

```python
from django.db import migrations


TASK_FUNC = "apps.kitchens.tasks.expiry_alerts.send_expiry_alerts"


def create_schedule(apps, schema_editor):
    Schedule = apps.get_model("django_q", "Schedule")
    Schedule.objects.update_or_create(
        func=TASK_FUNC,
        defaults={
            "name": "expiry-alerts-daily-06",
            "schedule_type": "C",        # Schedule.CRON
            "cron": "0 6 * * *",         # 06:00 in settings.TIME_ZONE (Australia/Melbourne)
            "repeats": -1,
        },
    )


def remove_schedule(apps, schema_editor):
    Schedule = apps.get_model("django_q", "Schedule")
    Schedule.objects.filter(func=TASK_FUNC).delete()


class Migration(migrations.Migration):
    dependencies = [
        ("kitchens", "0007_expiry_alert_log"),
        ("django_q", "0001_initial"),
    ]
    operations = [
        migrations.RunPython(create_schedule, remove_schedule),
    ]
```

> **Why the literal `"C"`** — when you use `apps.get_model` in a migration you get a *historical* model that has no class-level constants like `Schedule.CRON`. The raw value is `"C"`.

## Pitfalls reviewer will check

- **`Schedule` row created by data migration, not by hand.** A fresh deploy must register the cron without operator action. If the reviewer runs `docker compose down -v && docker compose up && docker compose exec web python manage.py shell -c "from django_q.models import Schedule; print(Schedule.objects.count())"` and gets `0`, the migration is missing or wrong.
- **`ExpiryAlertLog` write is inside the transaction with the email send.** If the order is reversed (`send_mail` first, log second), an SMTP failure mid-batch causes the next run to skip the kitchen — silently losing the alert.
- **`update_or_create` not `create` in the migration.** Otherwise the second `migrate` on the same DB explodes with `IntegrityError` on the (yet-to-be-added) unique constraint on `func` if Django-Q2 ever adds one. Idempotent migrations are the rule.
- **`apps.get_model("django_q", "Schedule")` in the migration**, not `from django_q.models import Schedule`. The latter binds to the *current* model class and breaks historical migrations.
- **Cron string is `"0 6 * * *"` evaluated in `settings.TIME_ZONE`.** Confirm `TIME_ZONE = "Australia/Melbourne"` in `config/settings/base.py`. If it is `UTC`, change the cron to `"0 20 * * *"` (UTC = 06:00 AEST) — but the right fix is to set `TIME_ZONE` correctly and let Django-Q2 do the conversion.
- **`quantity > 0` filter.** Without it, fully-consumed batches show up as noise in tomorrow's email.
- **`timezone.localdate()` everywhere**, not `date.today()`. The CI runs in UTC and would otherwise filter for the wrong day at the 14:00 UTC boundary (= midnight Melbourne).
- **No `print()` debug statements.** Use `logger.info(...)`. The Docker `worker` service runs unattended; print output is invisible.
- **`fail_silently=False`** on `send_mail` — we want the transaction to roll back the `ExpiryAlertLog` row if SMTP fails, so the next morning's run retries.
- **Email body groups by ingredient.** Reviewer will read the test asserting `"LOT-A"` and `"LOT-B"` both appear under the same `Basmati rice` heading — this is the agreed UX with ops.

## Tech / library notes

- The `partners` table has no `deleted_at` in the schema — extend
  `TimeStampedModel`, not `SoftDeleteModel`.
- `type` is an ENUM in MySQL; Django side uses `CharField(max_length=20,
  choices=TYPE_CHOICES)`.
- The `User.partner_id` swap is the only "destructive" migration this sprint.
  The dev DB already has a superuser; their `partner_id` is `NULL`, so
  the FK addition is backwards-compatible. The migration plan:
  1. Add `apps.partners` to `INSTALLED_APPS`.
  2. `makemigrations partners` → creates the `partners` table.
  3. `makemigrations accounts` → alters `users.partner_id` from
     `BigIntegerField(null=True)` to `ForeignKey(..., null=True,
     db_column="partner_id")`. Django generates an `AlterField` op.
  4. Apply both. The `AlterField` adds the FK constraint; existing NULL
     values are valid.
- `on_delete=PROTECT` on `User.partner` — refusing to delete a partner that
  has users prevents accidental cascade of historical data.

```python
from django.db import models

from apps.core.models import TimeStampedModel


class Partner(TimeStampedModel):
    """Referring organisations: charities, restaurants, suppliers, corporates.

    Schema only — see merrymeal_schema_corrected.sql::partners.
    """

    TYPE_CHOICES = [
        ("charity", "Charity"),
        ("restaurant", "Restaurant"),
        ("supplier", "Supplier"),
        ("corporate", "Corporate"),
    ]

    legal_name = models.CharField(max_length=255)
    type = models.CharField(max_length=20, choices=TYPE_CHOICES)

    class Meta:
        app_label = "partners"
        db_table = "partners"
        ordering = ("legal_name",)

    def __str__(self) -> str:
        return self.legal_name
```

## Pitfalls reviewer will check

- **`INSTALLED_APPS` is `"apps.partners"`** (not `"partners"`). Without the
  `apps.` prefix the `apps.partners.models` import fails.
- **`db_column="partner_id"`** — without this, Django would generate
  `partner_id` automatically anyway, but being explicit insulates the schema
  from any future rename of the Python attribute.
- **`on_delete=PROTECT`** on the FK. CASCADE would let an admin delete a
  partner and silently nuke every linked user.
- **`related_name="users"`** — `partner.users.all()` is more readable than
  the Django default `partner.user_set.all()`.
- **Migration order matters.** Run `makemigrations partners` BEFORE
  `makemigrations accounts`; otherwise Django can't resolve the
  `"partners.Partner"` string.
- **Dev DB compatibility.** The migration alters an existing column. Verify
  on a populated DB that the superuser row survives (Step 6.5).
- **No `requests/` layer.** Per README convention, do not introduce one
  here — the admin is the only entry point for Partner CRUD in this sprint.

## Tech / library notes

- The schema has **no surrogate `id`** on `diet_preference_user` and
  `allergy_user` — the composite PK is `(user_id, diet_preference_id)` /
  `(user_id, allergy_id)`.
- Django supports composite PK via `class Meta: constraints =
  [models.UniqueConstraint(fields=[...], name='pk_...')]` but **doesn't
  expose composite PKs as the actual `pk`** until Django 5.2+. For 5.1
  (this project), the supported approach: declare the through model with
  `Meta.constraints` enforcing uniqueness on the pair, AND omit `id` by
  setting `Meta.managed = True` and supplying an explicit PK using a
  multi-field `UniqueConstraint`. We pragmatically still let Django add a
  `BigAutoField id` (Django requires a PK), but enforce uniqueness on the
  pair via `UniqueConstraint`. The extra `id` is harmless — call out in the
  PR description.
- Names on `DietPreference` / `Allergy` are stored case-insensitively for
  duplicate detection: seed everything lowercase, enforce uniqueness via
  `unique=True` on `name`.
- `seed_dietary` uses `get_or_create(name=...)` to be idempotent.
- Inlines on `UserAdmin` use the **through model** directly (`UserDietPreference`,
  `UserAllergy`), not a Django-style `ManyToManyField` on `User`. The User
  model stays untouched.

```python
from django.db import models


class DietPreference(models.Model):
    """Controlled list of diet preferences (vegan, halal, …).

    Schema only — see merrymeal_schema_corrected.sql::diet_preferences.
    """

    name = models.CharField(max_length=80, unique=True)

    class Meta:
        app_label = "dietary"
        db_table = "diet_preferences"
        ordering = ("name",)

    def __str__(self) -> str:
        return self.name


class UserDietPreference(models.Model):
    """Through-table: which users declared which preferences.

    DB schema uses a composite PK (user_id, diet_preference_id) with no
    surrogate `id`. Django 5.1 still requires a single-column PK, so we
    let Django add an auto `id` and enforce the pair-uniqueness via
    UniqueConstraint. Downstream code reads/writes via the FKs, never `id`.
    """

    user = models.ForeignKey(
        "accounts.User",
        on_delete=models.CASCADE,
        related_name="diet_preference_links",
        db_column="user_id",
    )
    diet_preference = models.ForeignKey(
        "dietary.DietPreference",
        on_delete=models.PROTECT,
        related_name="user_links",
        db_column="diet_preference_id",
    )

    class Meta:
        app_label = "dietary"
        db_table = "diet_preference_user"
        constraints = [
            models.UniqueConstraint(
                fields=["user", "diet_preference"],
                name="uq_user_diet_preference",
            ),
        ]

    def __str__(self) -> str:
        return f"User {self.user_id} → {self.diet_preference_id}"
```

```python
from django.db import models


class Allergy(models.Model):
    """Controlled list of allergies.

    Schema only — see merrymeal_schema_corrected.sql::allergies.
    """

    name = models.CharField(max_length=80, unique=True)

    class Meta:
        app_label = "dietary"
        db_table = "allergies"
        ordering = ("name",)

    def __str__(self) -> str:
        return self.name


class UserAllergy(models.Model):
    """Through-table: which users declared which allergies.

    See UserDietPreference for the composite-PK / surrogate-id note.
    """

    user = models.ForeignKey(
        "accounts.User",
        on_delete=models.CASCADE,
        related_name="allergy_links",
        db_column="user_id",
    )
    allergy = models.ForeignKey(
        "dietary.Allergy",
        on_delete=models.PROTECT,
        related_name="user_links",
        db_column="allergy_id",
    )

    class Meta:
        app_label = "dietary"
        db_table = "allergy_user"
        constraints = [
            models.UniqueConstraint(
                fields=["user", "allergy"],
                name="uq_user_allergy",
            ),
        ]

    def __str__(self) -> str:
        return f"User {self.user_id} → {self.allergy_id}"
```

```python
from django.core.management.base import BaseCommand

from apps.dietary.models import Allergy, DietPreference


DIET_PREFERENCES = [
    "vegetarian",
    "vegan",
    "halal",
    "kosher",
    "gluten-free",
    "diabetic-friendly",
    "low-sodium",
    "pureed",
]

ALLERGIES = [
    "peanut",
    "tree nut",
    "dairy",
    "egg",
    "soy",
    "shellfish",
    "gluten",
]


class Command(BaseCommand):
    help = "Seed the diet_preferences and allergies tables. Idempotent."

    def handle(self, *args, **options):
        dp_created = 0
        for name in DIET_PREFERENCES:
            _, was_created = DietPreference.objects.get_or_create(name=name)
            if was_created:
                dp_created += 1

        al_created = 0
        for name in ALLERGIES:
            _, was_created = Allergy.objects.get_or_create(name=name)
            if was_created:
                al_created += 1

        self.stdout.write(
            self.style.SUCCESS(
                f"seed_dietary: diet_preferences {dp_created}/{len(DIET_PREFERENCES)} new; "
                f"allergies {al_created}/{len(ALLERGIES)} new."
            )
        )
```

## Pitfalls reviewer will check

- **Table names exact:** `diet_preferences`, `diet_preference_user`,
  `allergies`, `allergy_user`. Singular vs plural matters.
- **`name` is `unique=True`.** Without it, the seed isn't really idempotent —
  a race could create "vegan" twice.
- **Through models use FKs, not a `ManyToManyField` on User.** The User
  model must stay schema-only and untouched here.
- **Seed list is exact.** "Tree nut" vs "tree nuts" vs "treenut" all differ.
  The test asserts the exact set.
- **`on_delete=PROTECT` on the right FK side** of the through tables
  (preserves the controlled vocab). `CASCADE` on the User FK is correct
  (when a user is hard-deleted, their links go).
- **Inline registration on UserAdmin** — `from apps.dietary.models import …`
  at the top of `apps/accounts/admin.py`. Cross-app imports flow accounts ←
  dietary (one direction).
- **`INSTALLED_APPS` order:** `apps.partners` must come before
  `apps.dietary` only if the latter referenced the former. They are
  independent — order alphabetically or by domain weight.

## Tech / library notes

- **`OneToOneField` vs `ForeignKey(unique=True)`.** Functionally
  identical at the DB layer (both produce a UK on the FK column).
  `OneToOneField` is the idiomatic Django expression of 1:1 and gives
  `delivery.feedback` reverse-access for free. Schema lists it as a
  `UNIQUE KEY uq_feedback_delivery` — we name the constraint to match.
- **`rating` is `NULL`-able per schema** (`TINYINT UNSIGNED NULL`). The
  member UI (Story 4.11) requires it, but rows can exist with just a
  note. Keep `null=True, blank=True` and let the form layer enforce
  presence.
- **`note` is `LONGTEXT`** in MySQL. Django's `TextField` translates to
  `LONGTEXT` on MySQL backend by default — no special field needed.
- **Constraint test strategy.** Use `pytest.raises(IntegrityError)` and
  `transaction.atomic()` to wrap the second insert. Without the
  `atomic()` wrapper, the test pollutes the connection state and the
  next assertion fails with a confusing "current transaction is
  aborted".

```python
from django.core.validators import MaxValueValidator, MinValueValidator
from django.db import models


class DeliveryFeedback(models.Model):
    """Schema only — exactly one feedback row per delivery.

    The form layer (Story 4.11) shapes how rating/note are captured;
    this file is just the table.
    """

    delivery = models.OneToOneField(
        "delivery.Delivery",
        on_delete=models.CASCADE,
        related_name="feedback",
        db_column="delivery_id",
    )
    rating = models.PositiveSmallIntegerField(
        null=True,
        blank=True,
        validators=[MinValueValidator(1), MaxValueValidator(5)],
    )
    note = models.TextField(null=True, blank=True)

    class Meta:
        app_label = "delivery"
        db_table = "delivery_feedback"
        constraints = [
            models.UniqueConstraint(
                fields=["delivery"], name="uq_feedback_delivery"
            ),
        ]

    def __str__(self) -> str:
        return f"Feedback(delivery={self.delivery_id}, rating={self.rating})"
```

> **Note on the redundant `UniqueConstraint`.** `OneToOneField` already
> emits a UK on the column. We add the named `UniqueConstraint` so the
> generated MySQL constraint name matches the schema's
> `uq_feedback_delivery`. Without it, Django auto-names the index and the
> schema reviewer has to chase a mismatch.

## Pitfalls reviewer will check

- **`OneToOneField` vs `ForeignKey(unique=True)`** — both legal, but we
  pick `OneToOneField` for the reverse-access ergonomics. Don't switch.
- **Cascade choice on `delivery`.** `CASCADE` — if the delivery is
  deleted (very rare; usually we soft-delete), the feedback row goes
  too. The audit log retains the original feedback content.
- **`PositiveSmallIntegerField` vs `IntegerField`.** Schema says
  `TINYINT UNSIGNED`. Django's `PositiveSmallIntegerField` is the
  closest (16-bit unsigned vs 8-bit, but accepts the same 1–5 range
  without trouble). Don't reach for `SmallIntegerField` — that's signed.
- **`TextField` vs `CharField(max_length=...)`** — schema is
  `LONGTEXT`. Django's `TextField` maps to `LONGTEXT` on MySQL. A
  `CharField(max_length=1000)` would silently truncate.
- **Named constraint** — the schema reviewer compares constraint names
  against the SQL file. If you skip the named `UniqueConstraint`, the
  generated index is `delivery_deliveryfeedback_delivery_id_xxxx_uniq`
  and the diff is loud. Keep the explicit name.
- **Transaction-wrap the duplicate-insert test** — without
  `with transaction.atomic():`, pytest's outer transaction breaks and
  subsequent assertions fail confusingly.
- **`null=True` on `rating`** — the schema is explicit. Don't tighten to
  `NOT NULL` because "the UI requires it" — that's the form's job.

## User story

> **As a** developer
> **I want** a `Delivery` row per (member, meal_plan, scheduled_date)
> **so that** the system has the smallest unit of "did Margaret get her meal today?".

      - returns the saved `Delivery`.
- [ ] Admin: `list_display` (id, member, scheduled_date, status, meal_type),
      `list_filter` on `status` + `scheduled_date` + `meal_type`,
      `date_hierarchy = scheduled_date`.

## Tech / library notes

- **`assign_meal_type` contract (Story 3.2).** Returns `"fresh"` or
  `"frozen"` given `(member, kitchen, scheduled_date)`. Rules (from Epic
  03): weekend → `frozen`; member's primary address > kitchen
  `service_radius_km` → `frozen`; otherwise `fresh`. The service in this
  story does **not** re-implement those rules — it imports and calls.
- **Placeholder volunteer for un-assigned deliveries.** The schema makes
  `volunteer_id` `NOT NULL`. The generator (Story 4.6) creates deliveries
  *before* routing, so we need a "to-be-routed" volunteer marker. Use a
  configured user fetched by `settings.DISPATCH_PLACEHOLDER_VOLUNTEER_ID`
  (`int`, optional). Provide a fallback: pick *any* user with
  `role="volunteer"` and `is_active=True`; if none, raise
  `RuntimeError("No volunteer available; seed one or set DISPATCH_PLACEHOLDER_VOLUNTEER_ID.")`.
  **Schema discrepancy noted at the top of the file** (see "Open
  questions" in the sprint summary).
- **`meal_type` field default.** Schema default `fresh`. The service
  always overrides via `assign_meal_type`, so the default only matters
  for admin-form inserts (rare).
- **`photo`** is a `URLField(max_length=512, blank=True, null=True)`
  pointing at S3. Upload pipeline ships in Story 4.9 (next sprint).
- **`latitude` / `longitude`** are `DecimalField(max_digits=10,
  decimal_places=7, null=True, blank=True)` — match the schema precision.

```python
from django.conf import settings
from django.db import models


class Delivery(models.Model):
    """Schema only — one (member, meal_plan, scheduled_date) tuple.

    Logic (`create_delivery`, `mark_delivered`, etc.) lives in
    `apps/delivery/services/`.
    """

    MEAL_TYPE_FRESH = "fresh"
    MEAL_TYPE_FROZEN = "frozen"
    MEAL_TYPE_CHOICES = [
        (MEAL_TYPE_FRESH, "Fresh"),
        (MEAL_TYPE_FROZEN, "Frozen"),
    ]

    STATUS_PENDING = "pending"
    STATUS_OUT_FOR_DELIVERY = "out_for_delivery"
    STATUS_DELIVERED = "delivered"
    STATUS_FAILED = "failed"
    STATUS_CHOICES = [
        (STATUS_PENDING, "Pending"),
        (STATUS_OUT_FOR_DELIVERY, "Out for delivery"),
        (STATUS_DELIVERED, "Delivered"),
        (STATUS_FAILED, "Failed"),
    ]

    route = models.ForeignKey(
        "delivery.Route",
        on_delete=models.SET_NULL,
        null=True,
        blank=True,
        related_name="deliveries",
        db_column="route_id",
    )
    meal_plan = models.ForeignKey(
        "planning.MealPlan",
        on_delete=models.PROTECT,
        related_name="deliveries",
        db_column="meal_plan_id",
    )
    volunteer = models.ForeignKey(
        settings.AUTH_USER_MODEL,
        on_delete=models.PROTECT,
        related_name="delivered_deliveries",
        db_column="volunteer_id",
    )
    member = models.ForeignKey(
        settings.AUTH_USER_MODEL,
        on_delete=models.PROTECT,
        related_name="received_deliveries",
        db_column="member_id",
    )
    member_address = models.ForeignKey(
        "accounts.UserAddress",
        on_delete=models.PROTECT,
        related_name="deliveries",
        db_column="member_address_id",
    )

    meal_type = models.CharField(
        max_length=10, choices=MEAL_TYPE_CHOICES, default=MEAL_TYPE_FRESH
    )
    status = models.CharField(
        max_length=20, choices=STATUS_CHOICES, default=STATUS_PENDING
    )
    scheduled_date = models.DateField(null=True, blank=True)
    delivered_time = models.DateTimeField(null=True, blank=True)
    latitude = models.DecimalField(
        max_digits=10, decimal_places=7, null=True, blank=True
    )
    longitude = models.DecimalField(
        max_digits=10, decimal_places=7, null=True, blank=True
    )
    photo = models.URLField(max_length=512, null=True, blank=True)
    created_at = models.DateTimeField(auto_now_add=True)
    updated_at = models.DateTimeField(auto_now=True)

    class Meta:
        app_label = "delivery"
        db_table = "deliveries"
        indexes = [
            models.Index(fields=["route"], name="idx_del_route"),
            models.Index(fields=["meal_plan"], name="idx_del_plan"),
            models.Index(fields=["member"], name="idx_del_member"),
            # For Story 4.6 idempotency (one delivery per member per date).
            models.Index(fields=["member", "scheduled_date"], name="idx_del_member_date"),
            models.Index(fields=["scheduled_date", "status"], name="idx_del_date_status"),
        ]

    def __str__(self) -> str:
        return f"Delivery #{self.pk or '?'} member={self.member_id} {self.scheduled_date}"
```

```python
# apps/delivery/services/deliveries.py
import datetime as dt
from typing import Optional

from django.db import transaction

from apps.delivery.models import Delivery, Route
from apps.planning.services import assign_meal_type


def create_delivery(
    *,
    member,
    member_address,
    meal_plan,
    volunteer,
    scheduled_date: dt.date,
    route: Optional[Route] = None,
) -> Delivery:
    """Create a `Delivery` with `meal_type` auto-resolved from member geography.

    `meal_type` is set via
    `planning.services.assign_meal_type(member, kitchen=meal_plan.kitchen,
    scheduled_date=scheduled_date)` — keeps the fresh/frozen rule in one
    place (Story 3.2).

    Raises:
        ValueError: if `member.role != "member"` or `member_address.user_id != member.id`.
    """
    if member.role != "member":
        raise ValueError(
            f"create_delivery: `member` must have role='member', got {member.role!r}"
        )
    if member_address.user_id != member.id:
        raise ValueError(
            "create_delivery: `member_address` does not belong to `member`."
        )

    meal_type = assign_meal_type(
        member=member,
        kitchen=meal_plan.kitchen,
        scheduled_date=scheduled_date,
    )

    with transaction.atomic():
        return Delivery.objects.create(
            route=route,
            meal_plan=meal_plan,
            volunteer=volunteer,
            member=member,
            member_address=member_address,
            meal_type=meal_type,
            status=Delivery.STATUS_PENDING,
            scheduled_date=scheduled_date,
        )
```

## Pitfalls reviewer will check

- **`on_delete` for `volunteer` + `member` + `meal_plan`** — `PROTECT`,
  not `CASCADE`. Deleting a meal plan must not orphan delivery history.
  Use soft-delete (`apps.core.services.soft_delete`) instead.
- **`route` is `SET_NULL`** — if a route gets cancelled and deleted, the
  delivery survives (it's the source of truth for "did Margaret get her
  meal"). The packer can re-route an orphaned delivery on the next run.
- **`assign_meal_type` import path** — must be the public re-export from
  `apps.planning.services`, not the internal module. If Story 3.2 puts
  it at `apps.planning.services.meal_type::assign_meal_type`, ensure the
  package `__init__.py` re-exports.
- **Forgetting `null=True` on `scheduled_date`** — schema says
  `DATE NULL`. The generator always passes a date, but the field must
  allow NULL.
- **`Decimal` precision** — 10 digits / 7 decimal places. Anything looser
  loses 0.001-degree (~110 m) precision needed for the 10 km rule and
  proof-of-delivery geotag.
- **Member-address ownership** — without the `user_id` check in the
  service, an admin could accidentally pair Margaret's delivery with
  Bob's address. The validation is cheap; ship it.
- **Placeholder volunteer for un-assigned deliveries** — flagged as an
  open question. For this story we accept that the test always passes a
  real volunteer; the generator (4.6) is where the placeholder strategy
  is finalised. If a different approach is taken (making the column
  nullable in a future migration), revisit here.

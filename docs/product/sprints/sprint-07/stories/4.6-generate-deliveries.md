## User story

> **As an** admin
> **I want** a job that turns "today's `MealPlan` × every active member"
> into one `Delivery` row per member
> **so that** I do not hand-build the day's delivery list.

## Acceptance criteria

- [ ] `apps/delivery/services/dispatch.py::generate_deliveries_for_date(date)`
      exists and:
  - Iterates active members (`role="member"`, `is_active=True`,
    `deleted_at IS NULL`).
  - For each member: finds their primary `UserAddress` (or, if no
    `is_primary` flag exists, the **earliest** address by `created_at`).
  - Finds the closest active `Kitchen` by Haversine distance from that
    address (`apps.core.geo.haversine_km`).
  - Looks up the `MealPlan` for `(kitchen, scheduled_date=date)`.
  - If no plan exists, **skip the member** and log
    `"no meal plan for kitchen=%s date=%s"`.
  - If the member has no address, **skip and log**
    `"member=%s skipped: no address"`.
  - Otherwise calls `create_delivery(...)` from Story 4.4 with a
    placeholder volunteer.
- [ ] **Idempotent.** Re-running for the same `date` does NOT create
      duplicate deliveries. Implementation: `get_or_create` on
      `(member, scheduled_date)` keyed inside the dispatch service. Existing
      rows are returned untouched (no `meal_type` re-resolution).
- [ ] Wired to **Django-Q2**: scheduled at **04:00 Australia/Melbourne**
      daily. The scheduled function is
      `apps.delivery.tasks.generate_deliveries.run_for_today`.
- [ ] Returns a `DispatchReport` dataclass: `created: list[Delivery]`,
      `skipped: list[tuple[member_id, reason]]`.
- [ ] Pytest tests (using `freezegun.freeze_time`):
  - **Happy path:** 3 members × 1 plan → 3 deliveries; re-run → still 3.
  - **No address:** member with no address → skipped + log line.
  - **No plan:** member's nearest kitchen has no plan for `date` → skipped.
  - **Weekend run:** `freeze_time("2026-06-13")` (Saturday) → all
    generated deliveries get `meal_type="frozen"` (driven by
    `assign_meal_type`).
  - **Idempotency:** running twice yields the same set of `Delivery.id`s.

## Tech / library notes

- **Haversine** — already exists at `apps.core.geo.haversine_km(lat1,
  lon1, lat2, lon2)`. Reuse, don't re-implement.
- **Closest-kitchen lookup** — naive O(M×K) is fine at charity scale
  (≤ 500 members, < 10 kitchens). No spatial index needed.
- **Primary address.** The schema does not include an `is_primary`
  column on `user_addresses`. Story spec from Epic 04 calls for
  `primary_address`. **Schema discrepancy** — until an `is_primary`
  column is added (Epic 02 backlog), fall back to "earliest by
  `created_at`, tiebreak by lowest `id`". Document this in the dispatch
  module's docstring and re-evaluate when the flag lands.
- **Placeholder volunteer.** See Story 4.4 — the service requires a
  non-null volunteer because the schema is `NOT NULL`. The generator
  picks a placeholder from:
  1. `settings.DISPATCH_PLACEHOLDER_VOLUNTEER_ID` if set, else
  2. The first `User` with `role="volunteer"` and `is_active=True`
     (ordered by `id`), else
  3. Raise `RuntimeError("No volunteer available for dispatch placeholder.")`.
  The actual volunteer is assigned by Story 4.7's packer.
- **Logging.** Use `logger = logging.getLogger("merrymeal.dispatch")`.
  Each skip emits a `WARNING` line that the Epic 06 admin widget will
  surface.
- **Django-Q2 scheduling.** Use `Q_CLUSTER` with `schedule` entries OR
  the `django_q.models.Schedule` API at app-startup. We use the
  `Schedule` API in an `AppConfig.ready()` hook so re-creating the
  schedule on every deploy is idempotent (`Schedule.objects.update_or_create`).
- **`freezegun`** — `freeze_time("2026-06-13T03:55:00+10:00")` for the
  Saturday case. Time zone matters because `assign_meal_type` looks at
  `scheduled_date.weekday()` in local time.

```python
"""apps/delivery/services/dispatch.py

Story 4.6 — generate today's `Delivery` rows.
Story 4.7 — pack them into `Route`s (added in the next story).

`generate_deliveries_for_date(date)` is idempotent: re-running for the
same date is a no-op for already-generated members.

Open issue: `UserAddress` does not yet have an `is_primary` flag. Until
Epic 02 ships it, we treat "earliest by created_at, tiebreak by id" as
the primary. See Story 4.6 for the schema discrepancy note.
"""
from __future__ import annotations

import dataclasses
import logging
from typing import Optional

from django.conf import settings
from django.contrib.auth import get_user_model
from django.db import transaction

from apps.accounts.models import UserAddress  # adjust import if app/path differs
from apps.core.geo import haversine_km
from apps.delivery.models import Delivery
from apps.delivery.services.deliveries import create_delivery
from apps.planning.models import Kitchen, MealPlan

logger = logging.getLogger("merrymeal.dispatch")
User = get_user_model()


@dataclasses.dataclass
class DispatchReport:
    created: list[Delivery]
    skipped: list[tuple[int, str]]  # (member_id, reason)


def _primary_address(member) -> Optional[UserAddress]:
    return (
        UserAddress.objects
        .filter(user=member, latitude__isnull=False, longitude__isnull=False)
        .order_by("created_at", "id")
        .first()
    )


def _closest_active_kitchen(lat: float, lon: float) -> Optional[Kitchen]:
    kitchens = list(Kitchen.objects.filter(is_active=True))
    if not kitchens:
        return None
    return min(
        kitchens,
        key=lambda k: haversine_km(
            float(lat), float(lon), float(k.latitude), float(k.longitude)
        ),
    )


def _placeholder_volunteer():
    placeholder_id = getattr(settings, "DISPATCH_PLACEHOLDER_VOLUNTEER_ID", None)
    if placeholder_id:
        try:
            return User.objects.get(pk=placeholder_id)
        except User.DoesNotExist:
            logger.warning(
                "DISPATCH_PLACEHOLDER_VOLUNTEER_ID=%s not found; falling back",
                placeholder_id,
            )
    vol = User.objects.filter(role="volunteer", is_active=True).order_by("id").first()
    if vol is None:
        raise RuntimeError(
            "No volunteer available for dispatch placeholder. "
            "Seed at least one volunteer or set DISPATCH_PLACEHOLDER_VOLUNTEER_ID."
        )
    return vol


def generate_deliveries_for_date(date) -> DispatchReport:
    """Create one `Delivery` per active member for `date`. Idempotent.

    Re-running on the same date is a no-op for members that already have a
    delivery for that date (uniqueness keyed on member_id + scheduled_date).
    """
    placeholder = _placeholder_volunteer()
    created: list[Delivery] = []
    skipped: list[tuple[int, str]] = []

    members = User.objects.filter(
        role="member", is_active=True, deleted_at__isnull=True,
    ).order_by("id")

    for member in members:
        # Idempotency: skip members that already have a delivery for this date.
        existing = Delivery.objects.filter(member=member, scheduled_date=date).first()
        if existing:
            logger.info("member=%s skipped: already has delivery=%s",
                        member.id, existing.id)
            continue

        address = _primary_address(member)
        if address is None:
            logger.warning("member=%s skipped: no address", member.id)
            skipped.append((member.id, "no_address"))
            continue

        kitchen = _closest_active_kitchen(address.latitude, address.longitude)
        if kitchen is None:
            logger.warning("member=%s skipped: no active kitchen", member.id)
            skipped.append((member.id, "no_kitchen"))
            continue

        plan = MealPlan.objects.filter(
            kitchen=kitchen, service_date=date,
        ).first()
        if plan is None:
            logger.warning(
                "member=%s skipped: no meal plan for kitchen=%s date=%s",
                member.id, kitchen.id, date,
            )
            skipped.append((member.id, "no_plan"))
            continue

        with transaction.atomic():
            delivery = create_delivery(
                member=member,
                member_address=address,
                meal_plan=plan,
                volunteer=placeholder,
                scheduled_date=date,
            )
        created.append(delivery)

    logger.info(
        "generate_deliveries_for_date date=%s created=%d skipped=%d",
        date, len(created), len(skipped),
    )
    return DispatchReport(created=created, skipped=skipped)
```

```python
"""Django-Q2 callable for the nightly dispatch.

The Q-cluster picks this up via the Schedule registered in
`DeliveryConfig.ready()`.
"""
import datetime as dt
import logging

from django.utils import timezone

from apps.delivery.services import generate_deliveries_for_date

logger = logging.getLogger("merrymeal.dispatch")


def run_for_today() -> dict:
    """Generate deliveries for *today's* date in local time.

    Returns a small dict so the Q-task result page shows useful counts.
    """
    today = timezone.localdate()
    report = generate_deliveries_for_date(today)
    return {
        "date": today.isoformat(),
        "created": len(report.created),
        "skipped": len(report.skipped),
    }
```

```python
import logging

from django.apps import AppConfig
from django.db.utils import OperationalError, ProgrammingError

logger = logging.getLogger("merrymeal.dispatch")


class DeliveryConfig(AppConfig):
    default_auto_field = "django.db.models.BigAutoField"
    name = "apps.delivery"
    label = "delivery"

    def ready(self):
        # Idempotent schedule registration. Wrapped because `ready()` runs
        # before migrations on a fresh install.
        try:
            from django_q.models import Schedule

            Schedule.objects.update_or_create(
                name="generate_deliveries_for_today",
                defaults={
                    "func": "apps.delivery.tasks.generate_deliveries.run_for_today",
                    "schedule_type": Schedule.DAILY,
                    "cron": "0 4 * * *",  # 04:00 Australia/Melbourne (USE_TZ=True)
                    "repeats": -1,
                },
            )
        except (OperationalError, ProgrammingError):
            # django_q tables don't exist yet (initial migrate). Safe to skip.
            logger.debug("django_q tables not ready; skipping schedule registration.")
        except Exception:  # noqa: BLE001
            logger.exception("Failed to register dispatch schedule.")
```

```python
import datetime as dt

from django.core.management.base import BaseCommand
from django.utils import timezone

from apps.delivery.services import generate_deliveries_for_date


class Command(BaseCommand):
    help = "Generate Delivery rows for a given date (default: today)."

    def add_arguments(self, parser):
        parser.add_argument(
            "--date", help="ISO date (YYYY-MM-DD). Defaults to today.",
        )

    def handle(self, *args, **opts):
        if opts["date"]:
            date = dt.date.fromisoformat(opts["date"])
        else:
            date = timezone.localdate()
        report = generate_deliveries_for_date(date)
        self.stdout.write(self.style.SUCCESS(
            f"date={date} created={len(report.created)} skipped={len(report.skipped)}"
        ))
        for member_id, reason in report.skipped:
            self.stdout.write(f"  skipped member={member_id} reason={reason}")
```

## Pitfalls reviewer will check

- **Idempotency check must use `(member, scheduled_date)`** — keying on
  `(member, meal_plan)` would silently double-write if the plan id
  changes (e.g. publisher re-creates the plan).
- **`get_or_create` is not used directly** — we lookup-then-create
  because `create_delivery` has side effects (calling
  `assign_meal_type`) that we do not want to fire when a row already
  exists. Use the explicit `filter().first()` + `create_delivery(...)`
  pattern shown in the service.
- **Closest-kitchen distance is computed in Python**, not SQL. At
  charity scale this is fine. Document that a Postgres + PostGIS
  migration would push this into a `KNN` index later.
- **`AppConfig.ready()` runs on `manage.py shell`** — wrap the schedule
  registration in `try/except (OperationalError, ProgrammingError)` so
  fresh installs don't crash before migrate.
- **Time zone of `cron: "0 4 * * *"`** — Django-Q2 with `USE_TZ=True`
  interprets cron in `TIME_ZONE`. Our settings already pin
  `Australia/Melbourne`. Don't introduce a `cron_tz` override.
- **Placeholder volunteer fallback** — if the test suite has zero
  volunteer users, the service raises `RuntimeError`. Tests use the
  `placeholder_volunteer` fixture to guarantee one exists.
- **`UserAddress` "primary"** — fallback logic (`order_by("created_at",
  "id")`) is a stop-gap until Epic 02 ships an `is_primary` flag. Open
  a follow-up issue and reference it in the dispatch module docstring.
- **`freezegun` and timezones** — `freeze_time("2026-06-13")` freezes at
  UTC midnight. `timezone.localdate()` in Melbourne then returns
  2026-06-13 (Saturday). The Saturday assertion relies on this; if a
  reviewer suggests `freeze_time(..., tz_offset=10)`, agree only if you
  also update the docstring.
- **No assignment of `route_id` here** — that's Story 4.7. The rows
  this service writes have `route=None`.

## User story

> **As a** volunteer
> **I want** to declare which days and times of the week I am available
> **so that** the auto-assigner knows when to count me in.

## Tech / library notes

- The schema column type is `ENUM(...)`. In Django that translates to a
  `CharField` with `choices=`. MySQL renders this as a `VARCHAR` under the
  hood — we do **not** need a custom field; the constraint is enforced by
  the `choices` list at the form/admin layer plus the DB `CharField`.
- **Validation rule:** the FK to `User` is unconstrained at the DB level
  (it's any user row), so the only place to enforce "volunteers only" is
  `clean()` and the create-service. Services (Story 4.2) call
  `instance.full_clean()` before `save()`.
- The `Availability` model is **schema only** per the project convention.
  Helpers like "toggle a slot" live in `apps/volunteers/services/` (added
  in Story 4.2). This file ships with fields + `Meta` + `__str__` only.
- **Why no unique constraint:** the editor in Story 4.2 needs to add and
  remove single (day, phrase) slots; modelling each slot as its own row
  with no compound uniqueness keeps the toggle endpoint a simple `delete`
  or `get_or_create`. The schema confirms this — there is no UK on the
  `volunteer_availabilities` table.

### Task 3: Implement the model (GREEN)

**Files:** `apps/volunteers/models/availabilities.py`, `apps/volunteers/models/__init__.py`

- [ ] **Step 3.1 — Create `apps/volunteers/models/availabilities.py`.**

```python
from django.conf import settings
from django.core.exceptions import ValidationError
from django.db import models


class Availability(models.Model):
    """Schema only — one row per (volunteer, day_of_week, day_phrase) slot.

    No `UniqueConstraint`: the editor in Story 4.2 toggles single slots in/out,
    and a volunteer may be available for `mon morning` AND `mon afternoon`.
    Logic (`toggle_slot`) lives in `apps/volunteers/services/`.
    """

    DAY_OF_WEEK_CHOICES = [
        ("mon", "Monday"),
        ("tue", "Tuesday"),
        ("wed", "Wednesday"),
        ("thu", "Thursday"),
        ("fri", "Friday"),
        ("sat", "Saturday"),
        ("sun", "Sunday"),
    ]

    DAY_PHRASE_CHOICES = [
        ("morning", "Morning"),
        ("afternoon", "Afternoon"),
        ("evening", "Evening"),
    ]

    volunteer = models.ForeignKey(
        settings.AUTH_USER_MODEL,
        on_delete=models.CASCADE,
        related_name="availabilities",
        db_column="volunteer_id",
    )
    day_of_week = models.CharField(max_length=3, choices=DAY_OF_WEEK_CHOICES)
    day_phrase = models.CharField(max_length=10, choices=DAY_PHRASE_CHOICES)

    class Meta:
        app_label = "volunteers"
        db_table = "volunteer_availabilities"
        indexes = [
            models.Index(fields=["volunteer"], name="idx_avail_volunteer"),
        ]

    def clean(self):
        # Schema only — the role check is the one piece of model-level
        # validation we keep, because no DB constraint can express it.
        if self.volunteer_id and self.volunteer.role != "volunteer":
            raise ValidationError(
                {"volunteer": "Only users with role='volunteer' may have availability."}
            )

    def __str__(self) -> str:
        return f"{self.volunteer_id}:{self.day_of_week}:{self.day_phrase}"
```

## Pitfalls reviewer will check

- **`app_label`** — must be `"volunteers"`, not `"apps.volunteers"`, otherwise
  Django generates the wrong migration package path.
- **`related_name="availabilities"`** — referenced from Story 4.7
  (`volunteer.availabilities.filter(day_of_week=..., day_phrase="morning")`).
  Renaming here breaks the dispatcher.
- **`db_column="volunteer_id"`** — without it, Django would name the column
  `volunteer_id` anyway by convention (FK → `_id`), but be explicit so a
  future rename of the field doesn't silently rename the DB column.
- **No `UniqueConstraint`** — if you add one (e.g. `(volunteer, day_of_week,
  day_phrase)`), the toggle endpoint in Story 4.2 can still work via
  `get_or_create` + `delete()`, but it changes the failure mode from
  silent-duplicate-row (currently impossible because the toggle deletes
  before inserting) to `IntegrityError`. We keep it simple.
- **`clean()` does not run on bulk operations** — services that call
  `Availability.objects.create(...)` directly must invoke `full_clean()`
  first. Document this in Story 4.2's service.
- **MySQL ENUM vs Django `CharField(choices=...)`** — Django doesn't emit a
  MySQL `ENUM` from a `CharField`. It emits `VARCHAR` and enforces the
  choice list at the form/admin layer. This is intentional: it makes
  migrations portable to Postgres later. If a teammate writes raw SQL
  expecting the literal `ENUM`, that's a doc/PR-review issue, not a
  schema mismatch — the values stored are identical.

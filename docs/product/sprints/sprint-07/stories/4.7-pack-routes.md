## User story

> **As an** admin
> **I want** today's `pending` deliveries grouped into `Route`s assigned
> to available volunteers, balanced by proximity
> **so that** the daily dispatch happens without human intervention 95%
> of the time.

## Acceptance criteria

- [ ] `apps/delivery/services/dispatch.py::assign_routes_for_date(date)`.
- [ ] For each `(kitchen, date)` pair with at least one unrouted pending
      `Delivery`:
  - Find volunteers whose `Availability` matches `day_of_week =
    {date.weekday()}` and `day_phrase = "morning"` (v1 hard-codes
    morning — afternoon/evening exist in the schema but the matcher
    ignores them for now).
  - **Greedy pack:** sort deliveries by Haversine distance from the
    kitchen, ascending. Chunk into groups of ≤ 12 deliveries each.
  - For each chunk, create one `Route(volunteer, route_date=date,
    status="planned")` and set `delivery.route_id` +
    `delivery.volunteer_id` to that route / volunteer.
- [ ] **Idempotent.** Re-running for the same date does not double-assign:
  - Existing `Route`s for `(volunteer, route_date=date,
    status="planned")` are deleted before re-packing (clean slate). Their
    deliveries get `route=None`, then are re-packed in the same
    transaction. Routes in `in_progress` / `completed` / `cancelled` are
    left alone.
- [ ] **Overflow.** When deliveries > (volunteers × 12), the excess
      deliveries remain `route=None` and `volunteer=placeholder` (the
      generator's value). The service logs a `WARNING`:
      `"overflow: kitchen=%s date=%s unassigned=%d"`. The admin widget
      (Epic 06) will surface this.
- [ ] Wired to Django-Q2: scheduled at **04:30 Australia/Melbourne**
      daily (30 minutes after Story 4.6).
- [ ] Returns a `PackReport` dataclass: `routes_created: list[Route]`,
      `unassigned: list[Delivery]`.
- [ ] Pytest tests (using `freezegun`):
  - **2 volunteers, 6 deliveries → 2 routes of 3 each.**
  - **1 volunteer, 30 deliveries → 1 route of 12, 18 unassigned, warning logged.**
  - **0 volunteers → no exception, all deliveries unassigned.**
  - **Idempotency:** re-run on same date → same set of routes
    (different ids — clean-slate strategy — but same counts and
    membership groups).
  - **Phrase filter:** a volunteer with `mon afternoon` only is NOT
    picked for a Monday morning route.

## Tech / library notes

- **Day-of-week mapping.** Python's `date.weekday()` returns 0=Monday.
  Map to schema enum: `["mon", "tue", "wed", "thu", "fri", "sat", "sun"][date.weekday()]`.
- **Greedy chunking.** After sorting deliveries by distance:
  ```python
  ROUTE_CAPACITY = 12
  chunks = [deliveries[i:i+ROUTE_CAPACITY] for i in range(0, len(deliveries), ROUTE_CAPACITY)]
  ```
- **Volunteer assignment within a kitchen.** Match chunks to volunteers
  by stable sort (`volunteer.id ascending`). No fairness algorithm in
  v1 — just "first available volunteer takes the first chunk". The
  admin can manually reassign (Story 4.14).
- **Why clean-slate idempotency, not merge?** Merging is fragile: if a
  delivery is removed between runs, "merge" would leave it orphaned in
  the route. Wiping planned routes and re-packing is simpler and only
  affects `status="planned"` routes (so an in-progress route from
  yesterday is untouched).
- **VRP is out of scope.** Document in the module docstring that v1 is a
  greedy nearest-from-kitchen pack. Real vehicle-routing optimisation
  (load balancing, two-opt, time windows) lands in Epic 07 backlog.
- **Transaction boundary.** Wrap the whole "delete planned routes →
  pack → create routes" in `transaction.atomic()` per `(kitchen, date)`
  pair so a partial failure does not strand deliveries with no route.

# (top-of-file additions)
import itertools

from apps.delivery.models import Route
from apps.volunteers.models import Availability


ROUTE_CAPACITY = 12
_WEEKDAY_TO_ENUM = ["mon", "tue", "wed", "thu", "fri", "sat", "sun"]


@dataclasses.dataclass
class PackReport:
    routes_created: list[Route]
    unassigned: list[Delivery]


def _pack_chunk_into_route(volunteer, route_date, chunk):
    route = Route.objects.create(
        volunteer=volunteer,
        route_date=route_date,
        status=Route.STATUS_PLANNED,
    )
    Delivery.objects.filter(pk__in=[d.pk for d in chunk]).update(
        route=route, volunteer=volunteer,
    )
    return route


def _available_volunteers(date) -> list:
    day_enum = _WEEKDAY_TO_ENUM[date.weekday()]
    vol_ids = (
        Availability.objects
        .filter(day_of_week=day_enum, day_phrase="morning")
        .values_list("volunteer_id", flat=True)
        .distinct()
    )
    return list(
        User.objects.filter(pk__in=vol_ids, is_active=True, role="volunteer")
        .order_by("id")
    )


def _clear_planned_routes_for_date(date) -> None:
    """Wipe `planned` routes for `date`, returning their deliveries to the
    unassigned pool. Leaves `in_progress` / `completed` / `cancelled`
    routes untouched.
    """
    planned = Route.objects.filter(route_date=date, status=Route.STATUS_PLANNED)
    Delivery.objects.filter(route__in=planned).update(route=None)
    planned.delete()


def assign_routes_for_date(date) -> PackReport:
    """Greedy-pack today's pending deliveries into `Route`s.

    v1 algorithm:
      1. Wipe planned routes for `date` (clean-slate idempotency).
      2. For each `(kitchen, date)` pair with unrouted pending deliveries:
         a. Sort deliveries by Haversine distance from the kitchen.
         b. Chunk into groups of `ROUTE_CAPACITY` (12).
         c. Match chunks to volunteers available on `day_of_week+morning`.
         d. Overflow (chunks > volunteers) → log warning, leave unassigned.

    Real VRP is out of scope — see Epic 07 backlog.
    """
    _clear_planned_routes_for_date(date)

    routes_created: list[Route] = []
    unassigned_all: list[Delivery] = []

    # Group deliveries by their meal_plan.kitchen.
    pending = (
        Delivery.objects
        .filter(scheduled_date=date, status=Delivery.STATUS_PENDING, route__isnull=True)
        .select_related("meal_plan__kitchen", "member_address")
    )
    by_kitchen: dict[int, list[Delivery]] = {}
    kitchens: dict[int, object] = {}
    for d in pending:
        k = d.meal_plan.kitchen
        by_kitchen.setdefault(k.id, []).append(d)
        kitchens[k.id] = k

    volunteers = _available_volunteers(date)

    for kitchen_id, deliveries in by_kitchen.items():
        kitchen = kitchens[kitchen_id]
        deliveries.sort(key=lambda d: haversine_km(
            float(kitchen.latitude), float(kitchen.longitude),
            float(d.member_address.latitude), float(d.member_address.longitude),
        ))

        chunks = [deliveries[i:i + ROUTE_CAPACITY]
                  for i in range(0, len(deliveries), ROUTE_CAPACITY)]

        # Match chunks to volunteers in order. Overflow chunks stay unassigned.
        matched = min(len(chunks), len(volunteers))
        with transaction.atomic():
            for chunk, vol in zip(chunks[:matched], volunteers[:matched]):
                routes_created.append(_pack_chunk_into_route(vol, date, chunk))

        # Unassigned remainder.
        overflow = list(itertools.chain.from_iterable(chunks[matched:]))
        if overflow:
            logger.warning(
                "overflow: kitchen=%s date=%s unassigned=%d",
                kitchen_id, date, len(overflow),
            )
        unassigned_all.extend(overflow)

    logger.info(
        "assign_routes_for_date date=%s routes=%d unassigned=%d",
        date, len(routes_created), len(unassigned_all),
    )
    return PackReport(routes_created=routes_created, unassigned=unassigned_all)

import datetime as dt

from django.core.management.base import BaseCommand
from django.utils import timezone

from apps.delivery.services import assign_routes_for_date


class Command(BaseCommand):
    help = "Pack today's pending Deliveries into Routes for available volunteers."

    def add_arguments(self, parser):
        parser.add_argument(
            "--date", help="ISO date (YYYY-MM-DD). Defaults to today.",
        )

    def handle(self, *args, **opts):
        date = (
            dt.date.fromisoformat(opts["date"]) if opts["date"]
            else timezone.localdate()
        )
        report = assign_routes_for_date(date)
        self.stdout.write(self.style.SUCCESS(
            f"date={date} routes={len(report.routes_created)} unassigned={len(report.unassigned)}"
        ))

## Pitfalls reviewer will check

- **Wiping `in_progress` routes** — the clear-step **must** filter on
  `status=planned` only. Easy to miss; covered by
  `test_inprogress_routes_from_prior_runs_are_left_alone`.
- **`Delivery.volunteer` update.** When packing, set both
  `delivery.route` AND `delivery.volunteer` to the new volunteer.
  Without the second update, the volunteer's "today" screen (4.8)
  wouldn't find the delivery.
- **Sorting by Haversine.** Pass floats, not `Decimal`, into
  `haversine_km` — the math module functions accept floats. Decimal
  silently gets coerced but slower.
- **`day_of_week` enum mapping** — `date.weekday()` is 0=Monday.
  Off-by-one (e.g. using `isoweekday()` which is 1=Monday) breaks
  Wednesday/Thursday matching. Test with `freeze_time` on at least two
  weekdays if you want extra confidence.
- **Phrase hardcoded to `morning`** — that's the v1 product call. Don't
  parameterise yet; afternoon/evening routing lands when we have
  evening volunteers (Epic 07).
- **Greedy pack vs balanced** — the simplest implementation gives one
  volunteer 12 deliveries and the next volunteer 1 (if there are 13
  deliveries). That's accepted v1 behaviour. Balance / VRP is Epic 07.
- **Logging duplicates on overflow** — emit the WARNING **once per
  kitchen-date pair**, not once per overflow delivery, otherwise the
  logs are unreadable at scale.
- **Race conditions.** Two cluster workers triggering at the same time
  could double-write. Django-Q's lock + the daily-cron schedule make
  this a non-issue in practice; document but don't add advisory locks
  unless the bug actually shows up.
- **Tests rely on Story 4.1's `AvailabilityFactory`** — if the factory
  signature changes (e.g. positional args), update the imports here.
- **Imports at module top.** Don't add Story-4.7-only imports
  (`Availability`, `Route`) inside functions to dodge a circular —
  there is no circular; the existing apps form a DAG.

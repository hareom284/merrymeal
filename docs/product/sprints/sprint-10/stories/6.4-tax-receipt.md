## User story

> **As a** donor at tax time
> **I want** a printable summary of every donation I made in an Australian financial year
> **so that** I have one page for my tax return.

## Acceptance criteria

- [ ] Route `/donor/receipts/<fy>/` where `<fy>` is the year the FY
      ends, e.g. `/donor/receipts/2026/` → FY 2025–26
      (period: `2025-07-01` to `2026-06-30` inclusive, Melbourne time).
- [ ] Gated by `@role_required('donor')`. A donor sees only their own
      donations.
- [ ] Page lists every donation in the FY with `status='completed'`:
  - Date, amount, campaign, transaction id.
- [ ] Summed total displayed prominently above the list.
- [ ] Charity ABN + address footer rendered from `settings`
      (`MERRYMEAL_ABN`, `MERRYMEAL_ADDRESS`). Placeholders OK; story 7.x
      replaces with the real values from accountant.
- [ ] Print CSS reduces the page to a clean A4 layout:
  - Hide site `nav`, `footer`, breadcrumbs.
  - Body becomes black text on white background.
  - Title + ABN top; table fills the page.
- [ ] JSON format at `?format=json` returns:
  ```json
  {
    "fy": 2026,
    "period": {"start": "2025-07-01", "end": "2026-06-30"},
    "donor": {"name": "...", "email": "..."},
    "donations": [{"date": "...", "amount_cents": ..., "campaign": "..."}],
    "total_cents": ...,
    "charity": {"abn": "...", "address": "..."}
  }
  ```
- [ ] FY in URL is validated: only `2024..(this_fy + 1)` allowed; bad
      input → 404.

## Tech / library notes

- **AU Financial Year.** FY 2026 = `2025-07-01` → `2026-06-30`.
  Implementation: given an integer `fy`, the period is
  `(date(fy-1, 7, 1), date(fy, 6, 30))`.
- **Boundary tests are mandatory.** Donations on `2025-06-30 23:59`
  belong to FY 2025; donations on `2025-07-01 00:00` belong to FY 2026.
  Melbourne time. Two parametrised tests pin this.
- **Print CSS.** Linked separately at `media="print"` rather than
  embedded — keeps the screen view clean. Class-based selectors
  (`@media print { .no-print { display: none; } ...}`).
- **JSON response.** Use `django.http.JsonResponse`; no DRF needed.
- **Settings placeholders.**
  ```python
  MERRYMEAL_ABN = env.str("MERRYMEAL_ABN", default="00 000 000 000")
  MERRYMEAL_ADDRESS = env.str(
      "MERRYMEAL_ADDRESS",
      default="123 Example St, Melbourne VIC 3000",
  )
  ```
- **Reuse service from 6.3.** Donor history already has a "list" — add
  a sibling `list_for_fy(donor_id, fy)` instead of duplicating SQL.

```python
"""Australian financial-year helpers.

FY N runs (N-1)-07-01 → N-06-30. So FY 2026 = 2025-07-01 → 2026-06-30.
"""
from __future__ import annotations

from datetime import date

from django.utils import timezone


def fy_period(fy: int) -> tuple[date, date]:
    return date(fy - 1, 7, 1), date(fy, 6, 30)


def is_valid_fy(fy: int) -> bool:
    this_year = timezone.localdate().year
    # An FY ending up to next calendar year is permitted (we are in it now).
    return 2024 <= fy <= this_year + 1
```

- [ ] **Step 2.2 — Add `list_for_fy` to `donor_history` service.**

```python
def list_for_fy(donor_id: int, fy: int) -> tuple[list[dict], int]:
    from apps.fundraising.models import Donation
    from apps.dashboards.services.fy import fy_period
    start, end = fy_period(fy)
    qs = (
        Donation.objects
        .filter(
            donor_id=donor_id,
            status="completed",
            created_at__date__gte=start,
            created_at__date__lte=end,
        )
        .select_related("campaign")
        .order_by("created_at")
    )
    rows = [
        {
            "id": d.id,
            "created_at": d.created_at,
            "amount_cents": d.amount_cents,
            "campaign_name": d.campaign.name,
            "transaction_id": d.transaction_id or "",
        }
        for d in qs
    ]
    total = sum(r["amount_cents"] for r in rows)
    return rows, total
```

> Using `created_at__date__gte` evaluates the boundary in the project
> TIME_ZONE; the boundary tests pin this behaviour.

```python
from datetime import date

from django.conf import settings
from django.http import Http404, JsonResponse
from django.shortcuts import render

from apps.core.decorators import role_required
from apps.dashboards.services import donor_history, fy as fy_service


@role_required("donor")
def fy_receipt(request, fy: int):
    if not fy_service.is_valid_fy(fy):
        raise Http404("Unknown FY")
    start, end = fy_service.fy_period(fy)
    rows, total = donor_history.list_for_fy(request.user.id, fy)

    if request.GET.get("format") == "json":
        return JsonResponse({
            "fy": fy,
            "period": {"start": start.isoformat(), "end": end.isoformat()},
            "donor": {
                "name": request.user.full_name,
                "email": request.user.email,
            },
            "donations": [
                {
                    "date": r["created_at"].date().isoformat(),
                    "amount_cents": r["amount_cents"],
                    "campaign": r["campaign_name"],
                    "transaction_id": r["transaction_id"],
                }
                for r in rows
            ],
            "total_cents": total,
            "charity": {
                "abn": settings.MERRYMEAL_ABN,
                "address": settings.MERRYMEAL_ADDRESS,
            },
        })

    return render(request, "dashboards/donor/fy_receipt.html", {
        "fy": fy, "period_start": start, "period_end": end,
        "rows": rows, "total_cents": total,
        "abn": settings.MERRYMEAL_ABN,
        "address": settings.MERRYMEAL_ADDRESS,
    })
```

## Pitfalls reviewer will check

- **UTC vs Melbourne.** A donation at `2025-06-30 22:00 UTC` is
  `2025-07-01 08:00` Melbourne and belongs to FY 2026. Boundary tests
  catch this; don't compare naive UTC.
- **Calendar year mistaken for FY.** "FY 2026" is **not** the year 2026
  — it is July 2025 through June 2026.
- **Print CSS not linked.** A `<style>` block inside the template is
  fine for prototyping; for the final story, the linked `print.css` is
  the requirement.
- **ABN hard-coded.** Must come from settings so the accountant value
  can be swapped without code changes.
- **`fy_period` returning datetimes.** The function returns `date`
  objects — using `__date__gte` on a timezone-aware datetime field
  evaluates in the project TZ.
- **JSON omits the period.** Accountants need start/end explicit; do
  not omit because "it's implied by FY".

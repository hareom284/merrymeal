## User story

> **As a** social worker at a referring charity
> **I want** a public form to submit a member on behalf of my client, attributed to my organisation
> **so that** the charity can see referral volume per partner and the member's record is correctly tagged.

## Acceptance criteria

- [ ] Route `/partners/refer/` is **public** (no login required).
- [ ] Form mirrors the Story 1.10 caregiver-on-behalf form **plus**:
  - **Your organisation** — dropdown populated from
    `partners WHERE type='charity'` (alphabetical by `legal_name`).
  - **Your name** — `CharField(max_length=255)`.
  - **Your email** — `EmailField()`.
  - These three new fields are required.
- [ ] On submit, an `Application` row is created with:
  - `status='submitted'`.
  - `partner_id` set to the selected partner.
  - A new `application_referrals` row (or JSON column on Application) capturing
    `partner_contact_name` + `partner_contact_email`.
- [ ] On admin approval (Story 1.7 flow), the resulting `Member`/`User`
      row carries the `partner_id` so Story 6.2's count picks it up.
      A unit test simulates approval and asserts `User.partner_id`.
- [ ] CSRF enabled (Django default); a hidden honeypot field rejects
      basic bot submissions.
- [ ] Rate-limited by IP: 5 submissions per hour
      (`django-ratelimit` `key='ip'`).
- [ ] Mobile-first at 375 px: the form is single-column, large tap
      targets, no horizontal scroll.
- [ ] Successful submission redirects to a "thank you" page that
      surfaces the application reference id.

## Tech / library notes

- **No login.** Confirm the URL is not behind any
  `LoginRequiredMiddleware` filter; Django's default permissive routing
  already covers it, but check `config/middleware.py`.
- **Honeypot.** Add a `<input name="website" class="hidden">` field;
  if non-empty on POST, silently succeed (do NOT log; honeypots reveal
  themselves to attackers if you 4xx).
- **Charity dropdown.** Source from `Partner.objects.filter(type='charity').order_by('legal_name')`.
  Use `ModelChoiceField`. Static at render-time — refresh on every GET.
- **Application schema.** Story 1.10 added `Application`. If the
  `partner_id` column is not on the model yet, this story adds it
  (migration). If it already exists, just use it.
- **Approval propagation.** In `apps/accounts/services/applications.py`
  the existing `approve_application(application)` function creates the
  `User`. Extend it to copy `application.partner_id` →
  `user.partner_id` if non-NULL.
- **`partner_contact_name` + `partner_contact_email`.** Store on the
  Application row in a JSON column `metadata` (preferred — no schema
  change) OR as new columns. Prefer JSON; Story 6.x reporting reads it
  if needed.

```python
from django import forms

from apps.accounts.models import Application, Partner


class PartnerReferralForm(forms.Form):
    """Public form a social worker fills in on behalf of a member."""

    # Honeypot — must remain empty
    website = forms.CharField(required=False, widget=forms.HiddenInput)

    # The partner-side fields
    partner_id = forms.ModelChoiceField(
        queryset=Partner.objects.filter(type="charity").order_by("legal_name"),
        label="Your organisation",
        empty_label="— choose your organisation —",
    )
    partner_contact_name = forms.CharField(
        max_length=255, label="Your name"
    )
    partner_contact_email = forms.EmailField(label="Your email")

    # Re-use Story 1.10 member fields (mirror exactly).
    member_full_name = forms.CharField(max_length=255,
                                       label="Member's name")
    member_email = forms.EmailField(label="Member's email", required=False)
    member_dob = forms.DateField(label="Date of birth")
    # ... mirror remaining 1.10 fields here ...

    def clean_website(self):
        # Always silently invalidate when honeypot present.
        return self.cleaned_data.get("website", "")

    @property
    def is_bot(self) -> bool:
        return bool(self.data.get("website"))
```

```python
from typing import Optional


def create_partner_referral(*, partner_id, partner_contact_name,
                            partner_contact_email, member_data) -> "Application":
    from apps.accounts.models import Application
    return Application.objects.create(
        status="submitted",
        partner_id=partner_id,
        full_name=member_data["full_name"],
        email=member_data.get("email") or "",
        dob=member_data["dob"],
        metadata={
            "partner_contact_name": partner_contact_name,
            "partner_contact_email": partner_contact_email,
            **member_data.get("extras", {}),
        },
    )


def approve_application(application) -> "User":
    """Existing 1.7 entry point — extended to propagate partner_id."""
    from apps.accounts.models import User
    user = User.objects.create(
        full_name=application.full_name,
        email=application.email,
        role="member",
        partner_id=application.partner_id,  # NEW LINE
        dob=application.dob,
    )
    application.status = "approved"
    application.save(update_fields=["status"])
    return user
```

- [ ] **Step 3.2 — Create `apps/accounts/views/partner_referral.py`.**

```python
from django.http import HttpResponseRedirect
from django.shortcuts import render, redirect
from django.urls import reverse
from django_ratelimit.decorators import ratelimit

from apps.accounts.forms.partner_referral import PartnerReferralForm
from apps.accounts.services.applications import create_partner_referral


@ratelimit(key="ip", rate="5/h", block=True)
def partner_referral_form(request):
    form = PartnerReferralForm(request.POST or None)
    if request.method == "POST":
        # Honeypot first — bail before validation to avoid leaking
        # which field failed.
        if form.is_bot:
            return redirect("partner_referral_thanks")
        if form.is_valid():
            partner = form.cleaned_data["partner_id"]
            app = create_partner_referral(
                partner_id=partner.id,
                partner_contact_name=form.cleaned_data["partner_contact_name"],
                partner_contact_email=form.cleaned_data["partner_contact_email"],
                member_data={
                    "full_name": form.cleaned_data["member_full_name"],
                    "email": form.cleaned_data.get("member_email"),
                    "dob": form.cleaned_data["member_dob"],
                },
            )
            return redirect(
                reverse("partner_referral_thanks") + f"?ref={app.id}"
            )
    return render(request, "accounts/application/partner_referral.html",
                  {"form": form})


def partner_referral_thanks(request):
    return render(request, "accounts/application/partner_referral_thanks.html",
                  {"ref": request.GET.get("ref")})
```

## Pitfalls reviewer will check

- **Login wall.** A leftover `LoginRequiredMiddleware` or a default
  `@login_required` on the URL include would silently break the public
  flow. Verify by hitting the URL in a private window.
- **Dropdown leaks all partners.** Filtering only by `is_active` (etc.)
  not `type='charity'` would expose restaurants and corporates; the
  social worker should not see them.
- **CSRF accidentally disabled.** Some teams disable CSRF on "public"
  forms; do NOT — Django needs CSRF for any session that may include
  cookies from prior browsing.
- **Honeypot 4xx leak.** Returning a 400 when the honeypot is full
  teaches the bot which field triggers; redirect to the thank-you page
  as if it succeeded.
- **Approval forgets `partner_id`.** Without copying the FK, Story 6.2's
  count silently undercounts. The dedicated approval test pins this.
- **Bot rate-limit too aggressive.** 5/hour by IP is a balance:
  a public library IP serving multiple social workers may exceed it.
  Document this caveat in the PR description.
- **Metadata JSON unindexed.** Storing partner-contact email in
  `metadata` is fine for now; admin search by social-worker email is
  Story 7.x.

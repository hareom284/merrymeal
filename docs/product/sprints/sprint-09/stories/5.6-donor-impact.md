## User story

> **As a** donor
> **I want** to see what my money paid for
> **so that** I feel my gift mattered and consider giving again.

## Acceptance criteria

- [ ] `apps/donations/services/impact.py::meals_for_amount(amount_cents: int) -> int`
      returns the **floor** of `amount_cents / MEAL_COST_CENTS`.
- [ ] `MEAL_COST_CENTS` lives in `config/settings/base.py`, defaults to `300`
      ($3), overridable via env (`env.int("MEAL_COST_CENTS", default=300)`).
- [ ] Function is **pure** (no DB, no IO) — trivially testable.
- [ ] Used by:
  - The thanks page (Story 5.5) — already wired.
  - The receipt email (Story 5.5) — already wired.
  - The donate page chips (Story 5.3) — caption "$50 ≈ 17 meals" below each chip.
- [ ] A template tag `meals_for(amount_cents)` mirrors the service for use
      from templates (`{{ 5000|meals_for }}` → `17`).
- [ ] Tests cover:
  - $30 (3000c) → 10 meals.
  - $1 (100c) → 0 meals (floor).
  - $3 (300c) → 1 meal (boundary).
  - $2.99 (299c) → 0 meals.
  - $0 → 0 meals.
  - Negative → raises `ValueError`.
  - Float input → raises `TypeError`.
  - Different `MEAL_COST_CENTS` setting → recomputed correctly (use `override_settings`).

## Tech / library notes

- **Type discipline:** money is `int`. The function rejects floats with a
  `TypeError` — same rule as the `dollars` filter (Story 5.1). This is the
  invariant that protects us from "amount_cents = 50.0" sneaking in.
- **Pure function:** no Django app loaded — the tests run without
  `@pytest.mark.django_db`.
- **Reading settings:** import `django.conf.settings` lazily inside the
  function so `override_settings` works in tests.
- **Floor division** (`//`) — never `round()`, never `math.floor(float(...))`.
- **Stubbing for 5.5:** if Story 5.5 lands before this one, ship a placeholder
  `meals_for_amount` that returns `0`. This story replaces it.

### Task 2: Implement (GREEN)

`apps/donations/services/impact.py`:

```python
"""Pure helper. No DB, no IO. Money is integer cents."""
from __future__ import annotations

from django.conf import settings


def meals_for_amount(amount_cents: int) -> int:
    if isinstance(amount_cents, bool) or not isinstance(amount_cents, int):
        raise TypeError(
            f"meals_for_amount expects int cents, got {type(amount_cents).__name__}"
        )
    if amount_cents < 0:
        raise ValueError("amount_cents must be non-negative")
    cost = int(getattr(settings, "MEAL_COST_CENTS", 300))
    if cost <= 0:
        raise ValueError("MEAL_COST_CENTS must be a positive integer")
    return amount_cents // cost
```

> Note the explicit `isinstance(..., bool)` check: in Python, `True` is also
> `int`. We reject it because "amount_cents=True" is a bug, not a donation.

Run tests → 8 passed.

    if amount_cents in (None, ""):
        return 0
    try:
        return meals_for_amount(int(amount_cents))
    except (TypeError, ValueError):
        return 0

## Pitfalls reviewer will check

- **Float division** — `amount_cents / cost` returns a float. Must use `//`.
- **`MEAL_COST_CENTS` read at import time** — if you cache the value at
  module load, `override_settings` won't affect it. Read inside the function.
- **`bool` accepted as `int`** — `True // 300 == 0` doesn't raise; the
  explicit bool check is required.
- **Negative amounts** — silently returning a negative meal count is a bug.
  Tests assert it raises.
- **Template tag too strict** — if it raised on bad input, every preview-mode
  template would crash. Catch and return 0.
- **Caption uses dollars, not cents, in `meals_for` call** — easy to miss
  when wiring the chip. Reviewer eyeballs that `5000` (cents) is passed,
  not `50`.

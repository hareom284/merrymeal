## User story

> **As an** admin / dietitian
> **I want** to maintain a list of meals (recipes) with prep + cook times
> **so that** the planner has a menu to schedule from.

## Acceptance criteria

- [ ] New Django app `apps/meals/` registered in `INSTALLED_APPS`.
- [ ] `apps/meals/models/meals.py::Meal` matches the SQL schema exactly:
  - `name` `VARCHAR(160)` NOT NULL.
  - `description` `TEXT` NULL.
  - `prep_time_minutes` `INT UNSIGNED` NULL.
  - `cook_time_minutes` `INT UNSIGNED` NULL.
  - `is_active` BOOLEAN NOT NULL DEFAULT 1.
  - `created_at`, `updated_at`, `deleted_at` (soft delete via `apps.core.models.SoftDeleteModel`).
  - `db_table = "meals"`.
- [ ] Django admin CRUD; list shows `name`, `prep_time_minutes`, `cook_time_minutes`, `is_active`; `list_filter = ("is_active",)`.
- [ ] `Meal.objects` excludes soft-deleted rows; `Meal.all_objects` includes them.
- [ ] pytest covers: `__str__`, defaults, soft-delete manager filtering, admin smoke.

## Tech / library notes

- **Soft delete is shared infrastructure.** Inherit `apps.core.models.SoftDeleteModel` — it brings `created_at`, `updated_at`, `deleted_at`, and both managers (`objects` = active-only, `all_objects` = everything). Do **not** redeclare these columns.
- **Schema only.** Fields, choices, `Meta`, `__str__`. No `delete()` override, no helper methods. Soft-deleting a meal happens via `apps.core.services.soft_delete(instance)` (Sprint 05 will add a service if a UI button needs it).
- **Positive-integer fields.** Use `PositiveIntegerField(null=True, blank=True)` to mirror `INT UNSIGNED NULL`.
- **`is_active` flag, NOT a soft-delete substitute.** `is_active=False` means "temporarily off the menu"; `deleted_at` means "removed from the catalogue forever". Both exist on purpose.

```python
from django.db import models

from apps.core.models.soft_delete import SoftDeleteModel


class Meal(SoftDeleteModel):
    """A recipe / dish on the MerryMeal menu. Schema only.

    Inherits `created_at`, `updated_at`, `deleted_at`, and the soft-delete
    managers from `SoftDeleteModel`. Anything that mutates state belongs in
    `apps.meals.services` (none needed for this story).
    """

    name = models.CharField(max_length=160)
    description = models.TextField(null=True, blank=True)
    prep_time_minutes = models.PositiveIntegerField(null=True, blank=True)
    cook_time_minutes = models.PositiveIntegerField(null=True, blank=True)
    is_active = models.BooleanField(default=True)

    class Meta:
        app_label = "meals"
        db_table = "meals"

    def __str__(self) -> str:
        return self.name
```

## Pitfalls reviewer will check

- **`db_table = "meals"`** — without it, Django builds `meals_meal`.
- **Inheriting `SoftDeleteModel`, not redeclaring `created_at`/`updated_at`/`deleted_at`.** Duplicating columns is a silent migration bug.
- **`is_active` is independent from soft-delete.** A reviewer who proposes "use `deleted_at` instead of `is_active`" should be redirected — the schema has both for distinct reasons (off-menu vs. removed).
- **`PositiveIntegerField`, not `IntegerField`.** Schema says `INT UNSIGNED`. Django's `PositiveIntegerField` maps to it and rejects negatives.
- **No `delete()` override.** Soft-deletion is a service-level operation; calling `meal.delete()` should still be a hard delete (rarely used).
- **App label `meals`** — matches the URL of admin and the migration namespace.

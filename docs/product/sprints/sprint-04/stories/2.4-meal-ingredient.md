> **I want** to associate ingredients with a recipe, including quantity
> **so that** stock-consumption planning becomes possible later.

## Tech / library notes

- **Where does the through-model live?** In `apps/kitchens` — the schema groups it with ingredients (`meal_ingredients` joins kitchens-domain `ingredients` to meals-domain `meals`). The admin **inline** lives on `apps/meals/admin.py` because it edits a Meal. Cross-app imports are fine in admin code.
- **Through-model, no `ManyToManyField`.** We declare `MealIngredient` directly with two `ForeignKey`s plus a `unique_together`. This is the explicit-through pattern Django supports; it keeps the model layer honest about the `quantity` column.
- **Schema only.** No methods on `MealIngredient`. Quantity validation uses `MinValueValidator(Decimal("0.01"))` on the field — picked up by `full_clean()` and the admin form.
- **FK `on_delete=PROTECT`.** A meal with ingredients should not be silently destroyed when an ingredient is deleted; protecting prevents accidents and matches food-safety expectations.
- **DB column names.** Use `db_column="meal_id"` and `db_column="ingredient_id"` on the FKs so the resulting columns match the SQL schema exactly (Django would otherwise append `_id` to whatever attribute name you choose — happily the convention aligns, but stating it explicitly is reviewer-friendly).

```python
from decimal import Decimal

from django.core.validators import MinValueValidator
from django.db import models


class MealIngredient(models.Model):
    """Recipe line — joins a Meal to an Ingredient with a quantity.

    Schema only. The schema unit (g/kg/ml/l/unit) lives on Ingredient; this
    row's quantity is in *that* ingredient's unit. Cooking conversions are
    deliberately out of scope.
    """

    meal = models.ForeignKey(
        "meals.Meal",
        on_delete=models.PROTECT,
        related_name="ingredients",
        db_column="meal_id",
    )
    ingredient = models.ForeignKey(
        "kitchens.Ingredient",
        on_delete=models.PROTECT,
        related_name="used_in_meals",
        db_column="ingredient_id",
    )
    quantity = models.DecimalField(
        max_digits=10,
        decimal_places=2,
        validators=[MinValueValidator(Decimal("0.01"))],
    )

    class Meta:
        app_label = "kitchens"
        db_table = "meal_ingredients"
        constraints = [
            models.UniqueConstraint(
                fields=["meal", "ingredient"],
                name="uq_meal_ingredient",
            )
        ]

    def __str__(self) -> str:
        return f"{self.meal.name} — {self.ingredient.name} ({self.quantity})"
```

## Pitfalls reviewer will check

- **`db_table = "meal_ingredients"`** — match the schema.
- **`UniqueConstraint` not `unique_together`.** Modern Django prefers `constraints=[UniqueConstraint(...)]`.
- **`on_delete=PROTECT` on both FKs.** `CASCADE` would silently wipe recipe lines when an ingredient is deleted. We want the admin to *fail* that delete and force the operator to clean up first.
- **`autocomplete_fields = ("ingredient",)`** — when the catalogue grows past 30 ingredients, a dropdown becomes painful. Autocomplete requires `IngredientAdmin.search_fields` (already set in Story 2.2).
- **Cross-app model reference via string** (`"meals.Meal"`, `"kitchens.Ingredient"`) — avoids import cycles.
- **`MinValueValidator(Decimal("0.01"))`, not `Decimal("0")`.** A quantity of zero is meaningless for a recipe line; reject it at the boundary.
- **The inline lives on the Meal admin**, not on a hypothetical Ingredient admin. Editing a recipe from the meal page is the natural UX.

## User story

> **As an** admin
> **I want** to register the kitchens we use (own + outsourced) with their location and service radius
> **so that** later, planning can decide who gets fresh vs frozen meals based on the 10 km rule.

## Acceptance criteria

- [ ] New Django app `apps/kitchens/` registered in `INSTALLED_APPS`.
- [ ] `apps/kitchens/models/kitchens.py::Kitchen` matches the SQL schema exactly:
  - `name` `VARCHAR(160)` NOT NULL.
  - `partner_id` `BIGINT UNSIGNED` NULL — FK to `partners.id`.
  - `is_outsourced` BOOLEAN NOT NULL DEFAULT 0.
  - `latitude` `DECIMAL(10,7)` NOT NULL.
  - `longitude` `DECIMAL(10,7)` NOT NULL.
  - `service_radius_km` `DECIMAL(5,2)` NOT NULL DEFAULT 10.00.
  - `created_at`, `updated_at` timestamps (via `TimeStampedModel`).
  - `db_table = "kitchens"`.
- [ ] Django admin CRUD page; list view shows `name`, `is_outsourced`, `latitude`, `longitude`, `service_radius_km`.
- [ ] Admin form validation rejects `latitude` outside `[-90, 90]` and `longitude` outside `[-180, 180]`.
- [ ] pytest suite covers: model `__str__`, default radius, lat/lng range validators, admin list shows the columns above.

## Tech / library notes

- **Schema only.** `Kitchen` has fields, `Meta` (`db_table`, indexes), `__str__`. No methods.
- **Layering.** Use the package layout (`models/`, no `models.py`). One file per feature area.
- **Timestamps.** Inherit `apps.core.models.TimeStampedModel` — gives `created_at` / `updated_at` without restating columns.
- **No FK to `Partner` model yet?** Epic 01 lands a `Partner` model. Use the same approach `User.partner_id` uses (see `apps/accounts/models/users.py`): store as `BigIntegerField(null=True, blank=True)` if the Partner model is not yet importable in this branch. The DB column name `partner_id` matches the schema regardless.
- **Validators.** `MinValueValidator(-90)` / `MaxValueValidator(90)` for latitude; `±180` for longitude. Apply at the field level so the admin form picks them up automatically.
- **DB column types.** `DecimalField(max_digits=10, decimal_places=7)` for lat/lng; `DecimalField(max_digits=5, decimal_places=2, default=Decimal("10.00"))` for radius.

```python
from decimal import Decimal

from django.core.validators import MaxValueValidator, MinValueValidator
from django.db import models

from apps.core.models.timestamped import TimeStampedModel


class Kitchen(TimeStampedModel):
    """A physical kitchen — own or outsourced — that prepares meals.

    Schema only: fields, Meta, __str__. Business logic (e.g. is_in_radius)
    belongs in apps.kitchens.services.
    """

    name = models.CharField(max_length=160)
    partner_id = models.BigIntegerField(null=True, blank=True)
    is_outsourced = models.BooleanField(default=False)
    latitude = models.DecimalField(
        max_digits=10,
        decimal_places=7,
        validators=[MinValueValidator(Decimal("-90")), MaxValueValidator(Decimal("90"))],
    )
    longitude = models.DecimalField(
        max_digits=10,
        decimal_places=7,
        validators=[MinValueValidator(Decimal("-180")), MaxValueValidator(Decimal("180"))],
    )
    service_radius_km = models.DecimalField(
        max_digits=5,
        decimal_places=2,
        default=Decimal("10.00"),
    )

    class Meta:
        app_label = "kitchens"
        db_table = "kitchens"
        indexes = [models.Index(fields=["partner_id"], name="idx_kitchen_partner")]

    def __str__(self) -> str:
        return self.name
```

## Pitfalls reviewer will check

- **`db_table = "kitchens"`** — without it, Django builds `kitchens_kitchen`. The schema is authoritative.
- **`partner_id` as `BigIntegerField`, not `ForeignKey`** — the `Partner` model is not in this story's scope. The column name still matches the schema (`partner_id`). When `Partner` lands, a later migration converts the field with `db_column="partner_id"` preserved.
- **`DecimalField`, not `FloatField`** — `FloatField` would silently drop precision and break the 10 km calc later.
- **Validators on the field, not in `clean()`** — putting them on the field means the admin form auto-wires them; `full_clean()` calls them; you do not maintain two copies.
- **`__str__` returns `name`**, not `repr(self)` or `f"Kitchen({self.pk})"` — the admin list dropdown uses it.
- **No business logic on the model** — `is_in_radius(point)` does not belong here. It belongs in `apps/kitchens/services/geo.py` (lands in Sprint 06).
- **App label `kitchens`** — keeps `app_label="kitchens"` consistent with the URL of admin (`/admin/kitchens/…`) and the migration namespace.

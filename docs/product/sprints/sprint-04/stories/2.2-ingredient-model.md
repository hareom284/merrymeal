## User story

> **As an** admin
> **I want** to maintain a catalogue of ingredients with units of measure
> **so that** recipes and stock can reference the same names without typos.

## Acceptance criteria

- [ ] `apps/kitchens/models/ingredients.py::Ingredient` matches the schema:
  - `name` `VARCHAR(160)` NOT NULL.
  - `unit` `ENUM('g','kg','ml','l','unit')` NOT NULL.
  - `db_table = "ingredients"`.
- [ ] `Ingredient` is registered in Django admin with `search_fields=("name",)` and `list_filter=("unit",)`.
- [ ] Management command `python3 manage.py seed_ingredients` creates **30** common ingredients (rice, pumpkin, chicken, etc.) idempotently — running it twice does not duplicate rows.
- [ ] pytest suite covers: model `__str__`, unit choices, seed creates 30 rows, seed is idempotent.

## Tech / library notes

- **Choices constant.** Define `UNIT_CHOICES` on the model so the admin filter and tests can import the same source of truth.
- **Idempotency.** Use `Ingredient.objects.update_or_create(name=…, defaults={"unit": …})` — case-sensitive on `name`. We do not enforce a DB-unique constraint on `name` (the schema does not have one), but the seed command uses `name` as its identity key.
- **Source-of-truth list.** Hard-code 30 ingredients inside the command. Order: staples → proteins → veg → pantry. Keep the list inline so the seed is self-contained.
- **Management command discovery.** Both `apps/kitchens/management/__init__.py` and `apps/kitchens/management/commands/__init__.py` must exist (empty is fine).

from django.db import models


class Ingredient(models.Model):
    """A catalogue ingredient. Schema only — no business logic."""

    UNIT_CHOICES = [
        ("g", "Gram"),
        ("kg", "Kilogram"),
        ("ml", "Millilitre"),
        ("l", "Litre"),
        ("unit", "Unit"),
    ]

    name = models.CharField(max_length=160)
    unit = models.CharField(max_length=10, choices=UNIT_CHOICES)

    class Meta:
        app_label = "kitchens"
        db_table = "ingredients"

    def __str__(self) -> str:
        return self.name
```

from django.core.management.base import BaseCommand

from apps.kitchens.models import Ingredient

# 30 common ingredients for the MerryMeal kitchens.
# Order is staples → proteins → veg → pantry so the demo seed reads naturally.
SEED_INGREDIENTS: list[tuple[str, str]] = [
    # Staples
    ("Rice", "kg"),
    ("Pasta", "kg"),
    ("Bread", "unit"),
    ("Potato", "kg"),
    ("Sweet potato", "kg"),
    # Proteins
    ("Chicken breast", "kg"),
    ("Chicken thigh", "kg"),
    ("Beef mince", "kg"),
    ("Lamb shoulder", "kg"),
    ("Fish fillet", "kg"),
    ("Tofu", "kg"),
    ("Lentils", "kg"),
    ("Chickpeas", "kg"),
    ("Egg", "unit"),
    # Vegetables
    ("Pumpkin", "kg"),
    ("Carrot", "kg"),
    ("Onion", "kg"),
    ("Garlic", "kg"),
    ("Tomato", "kg"),
    ("Spinach", "kg"),
    ("Broccoli", "kg"),
    ("Capsicum", "kg"),
    ("Zucchini", "kg"),
    # Pantry
    ("Olive oil", "l"),
    ("Coconut milk", "ml"),
    ("Soy sauce", "ml"),
    ("Salt", "g"),
    ("Pepper", "g"),
    ("Curry powder", "g"),
    ("Stock cube", "unit"),
]


class Command(BaseCommand):
    help = "Idempotently seed 30 common ingredients used by demo recipes."

    def handle(self, *args, **options):
        created = 0
        for name, unit in SEED_INGREDIENTS:
            _, was_created = Ingredient.objects.update_or_create(
                name=name, defaults={"unit": unit}
            )
            if was_created:
                created += 1
        self.stdout.write(
            self.style.SUCCESS(
                f"seed_ingredients: {created} new, "
                f"{len(SEED_INGREDIENTS) - created} updated. "
                f"Total now: {Ingredient.objects.count()}."
            )
        )
```

## Pitfalls reviewer will check

- **30 ingredients exactly.** The list defined in the command is the source of truth; the test asserts the count. If you add a 31st, update the test in the same commit.
- **`update_or_create`, not `get_or_create`.** `get_or_create` would silently keep a stale `unit` if a future seed changes it.
- **`unit` is a string-coded enum (`g`, `kg`, …), not an integer.** Matches the MySQL `ENUM(...)` order.
- **No unique constraint on `name`.** The schema does not declare one. The seed relies on `update_or_create(name=…)` as a *logical* uniqueness — fine for a seed, do not extrapolate to user-facing CRUD.
- **Don't move the seed list to a fixture.** A management command with hard-coded data is easier to diff and review than a JSON fixture.
- **Both `management/__init__.py` and `management/commands/__init__.py` must exist** — Django discovers commands by walking those packages.

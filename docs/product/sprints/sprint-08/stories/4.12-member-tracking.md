## User story

> **As** Margaret (or her daughter) at lunchtime
> **I want** my home page to show whether today's meal is pending, on the
> way, or delivered — and to refresh itself without me reloading
> **so that** I do not have to phone the kitchen to find out.

## Acceptance criteria

- [ ] On the member home (Story 3.4), the today card shows a **status
      line** reflecting the live `Delivery.status` for today:
  - `pending` → "Cooking — pickup at HH:mm"
  - `out_for_delivery` → "On the way with <first> <L.>"
  - `delivered` → "Delivered ✓ at HH:mm" (green pill)
  - `failed` → "Couldn't deliver — we'll call you" (orange pill)
- [ ] For `out_for_delivery`, the volunteer's first name + last initial
      is rendered (e.g. "On the way with Sarah K.").
- [ ] HTMX polling: while status ∈ {`pending`, `out_for_delivery`}, the
      card includes `hx-trigger="every 60s"` against
      `GET /member/delivery/<id>/status/` and swaps itself with the
      returned partial.
- [ ] Once status flips to `delivered` or `failed`, the returned
      partial **omits** the `hx-trigger`, so polling stops naturally.
- [ ] Caregivers (linked via `member_caregivers`) can also see this
      partial when impersonating / viewing the member's dashboard
      (Epic 06 wires the navigation; here we only need the partial to be
      role-agnostic on the read).
- [ ] Mobile-first at 375 × 667 px, 18 px base, status pill ≥ 44 px tall.

## Tech / library notes

- **Polling frequency.** 60 seconds is a deliberate trade-off between
  freshness and load. Two simultaneous browsers across 200 members =
  6.7 req/s avg — well within a single Gunicorn worker.
- **Server-Sent Events / websockets are out of scope** until Epic 07
  (PWA). HTMX polling is good enough.
- **Stopping the poll.** The trick: the partial that gets swapped in
  for terminal states (`delivered`, `failed`) does **not** carry
  `hx-trigger`, so HTMX has nothing to fire next. No JavaScript needed
  to "stop" it.
- **Volunteer name visibility.** Show first name + last initial. The
  full name and phone are only revealed via Story 4.18 (consent),
  which is backlog.
- **Time display** — server-side via Django `{{ time|time:"H:i" }}` in
  the `Asia/Melbourne` timezone (Sprint 02 set `TIME_ZONE`).

```python
"""Service: shape the member tracking card."""
from apps.delivery.models import Delivery


TERMINAL = {"delivered", "failed"}


def _vol_display(v) -> str:
    if not v:
        return ""
    last_initial = (v.last_name[:1] + ".") if v.last_name else ""
    return f"{v.first_name} {last_initial}".strip()


def get_tracking_context(delivery: Delivery, viewer) -> dict:
    status = delivery.status
    polling = status not in TERMINAL

    label_map = {
        "pending": "Cooking — pickup soon",
        "out_for_delivery": f"On the way with {_vol_display(delivery.volunteer)}",
        "delivered": "Delivered ✓",
        "failed": "Couldn't deliver — we'll call you",
    }

    return {
        "delivery": delivery,
        "status": status,
        "label": label_map.get(status, status),
        "volunteer_display": _vol_display(delivery.volunteer),
        "delivered_at": delivery.delivered_time,
        "polling": polling,
    }
```

### Task 3: View + URL

**Files:** `apps/delivery/views/tracking.py`, `apps/delivery/urls.py`

- [ ] **Step 3.1 — Create the view.**

```python
from django.contrib.auth.decorators import login_required
from django.shortcuts import get_object_or_404, render
from apps.delivery.models import Delivery
from apps.delivery.services.tracking import get_tracking_context


@login_required
def tracking_status_view(request, pk: int):
    # Member sees own; caregiver sees linked member's (basic check).
    qs = Delivery.objects.filter(pk=pk)
    if request.user.role == "member":
        qs = qs.filter(member=request.user)
    elif request.user.role == "caregiver":
        qs = qs.filter(member__caregivers__caregiver=request.user)
    else:
        qs = qs.none()
    delivery = get_object_or_404(qs)
    ctx = get_tracking_context(delivery, request.user)
    return render(request, "delivery/member/_tracking_card.html", ctx)
```

## Pitfalls reviewer will check

- **`hx-trigger` left on terminal-state partial** — silent polling
  forever, drains the user's data and your Gunicorn pool. The partial
  template must guard with `{% if polling %}`.
- **`hx-swap="outerHTML"` vs `innerHTML`** — must be `outerHTML` so
  the swapped-in partial brings its own (absent) `hx-trigger`.
- **First-name-only display** — if `last_name` is empty, the helper
  must not render `Sarah .` with a stray period.
- **Timezone** — `delivered_time|time` renders in Django's active TZ.
  Confirm `TIME_ZONE = "Australia/Melbourne"` and `USE_TZ = True` in
  base settings, otherwise UTC leaks into the badge.
- **Caregiver impersonation** — for v1 the partial supports caregiver
  read via `member__caregivers__caregiver=request.user`. Confirm the
  `member_caregivers` table reverse relation name matches; if you used
  `related_name="caregivers"` on the M2M, the lookup above is correct.
- **HTMX header for tests** — Django returns the same partial whether
  called via XHR or full page nav; tests assert on `b"hx-trigger"` in
  body, which is template content not headers.
- **Polling at 60 s vs 60_000 ms** — HTMX expects `every 60s` not
  `every 60`. Typo silently disables polling.
- **N+1 on volunteer name** — `select_related("volunteer")` on the
  service queryset to avoid per-poll DB hits.

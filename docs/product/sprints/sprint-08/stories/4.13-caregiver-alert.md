## User story

> **As** Margaret's daughter on a Wednesday afternoon
> **I want** an email **and** a text within 60 seconds of a delivery
> failing
> **so that** I can call Mum, drop by, or arrange redelivery without
> first being told by the kitchen.

## Acceptance criteria

- [ ] A Django `post_save` signal on `Delivery` detects a status
      transition to `failed` (was-something-else → now `failed`) and
      enqueues a Django-Q2 task per linked caregiver.
- [ ] The task sends **both** an email and an SMS containing:
  - Member's first + last name
  - Scheduled date
  - The reason (parsed from `failure_reason`)
  - A "Call the office: 03 9000 0000" line (and the volunteer's phone
    iff Story 4.18 consent is present — out of scope this sprint, so
    show only the office line for v1).
- [ ] **SMS provider abstraction.** New module
      `apps/core/services/sms.py` exposes `send_sms(to: str, body: str)`.
      Two backends:
  - `console` (dev / test) — prints to stdout, stores in a
    `apps.core.testing.sms_outbox` list for assertions.
  - `twilio` (prod) — calls Twilio REST API with env vars.
- [ ] **Rate limit.** Per `(member_id, date)`, at most **1 SMS** is
      sent. Subsequent failures for the same member on the same day
      still send email but skip SMS (mitigates Twilio bill spike risk
      from the sprint plan).
- [ ] **Fallback.** If a member has zero linked caregivers, the email
      goes to the office address (`OFFICE_ALERT_EMAIL` setting). No SMS.
- [ ] Twilio env vars added to `.env.example`. Settings choose backend
      via `SMS_BACKEND` (`console` in dev/test, `twilio` in prod only).
- [ ] Tests use `django.core.mail.outbox` for email and the in-memory
      `sms_outbox` for SMS — no network calls.

## Tech / library notes

- **Signal scope.** Use `post_save` with `update_fields` check — only
  fire when `status` is in the saved field set and current value is
  `failed`. Also track previous status via an `__init__` snapshot or
  a simple `Delivery.tracker` field — for v1 the simplest is to read
  the previous status in the signal handler from the audit log; or
  shortcut: in the `mark_failed` service, set a request-local flag.
  **Easiest correct version**: compare `instance.status == 'failed'`
  *and* check there is no prior alert log entry for this delivery
  before enqueuing.
- **Idempotency.** Use a `Delivery.failure_alert_sent_at` column? No —
  schema keeps `failure_reason` as the only new column (Story 4.10).
  Use a `CaregiverAlertLog` table or simply rely on a unique cache key
  per delivery. For v1 use a one-line `cache.add("alert:%d" % d.id, 1,
  timeout=3600)` — `add` returns `False` if it already exists.
- **Rate limit key:** `sms:%d:%s % (member_id, scheduled_date.isoformat())`,
  TTL 24 h.
- **Django-Q2** is already wired in Sprint 01. Enqueue via
  `async_task("apps.delivery.tasks.alerts.send_failure_alert", delivery.id)`.
- **Email template** uses `render_to_string` for HTML + txt; send via
  `EmailMultiAlternatives` so non-HTML clients still get readable text.
- **Twilio**: only one outbound API call per recipient. Catch
  `twilio.base.exceptions.TwilioRestException` and log; do not retry
  inside the task — Django-Q2 already retries on raise.

```python
from apps.core.services.sms_backends import ConsoleBackend, TwilioBackend


BACKENDS = {
    "console": ConsoleBackend,
    "twilio": TwilioBackend,
}


def send_sms(*, to: str, body: str) -> None:
    name = getattr(settings, "SMS_BACKEND", "console")
    cls = BACKENDS.get(name)
    if cls is None:
        raise ValueError(f"unknown SMS_BACKEND: {name}")
    cls().send(to=to, body=body)
```

### Task 3: Alert service GREEN

**Files:** `apps/delivery/services/alerts.py`

- [ ] **Step 3.1 — Write the alert service.**

```python
"""Notify caregivers when a delivery fails. Email + (rate-limited) SMS."""
from django.conf import settings
from django.core.cache import cache
from django.core.mail import EmailMultiAlternatives
from django.template.loader import render_to_string

from apps.core.services.sms import send_sms
from apps.delivery.models import Delivery


SMS_TTL = 24 * 60 * 60


def _rate_key(delivery: Delivery) -> str:
    return f"sms:{delivery.member_id}:{delivery.scheduled_date.isoformat()}"


def _recipients(delivery: Delivery):
    caregivers = list(delivery.member.caregivers.select_related("caregiver"))
    if caregivers:
        return [
            (mc.caregiver.email, mc.caregiver.phone)
            for mc in caregivers
            if mc.caregiver.email
        ]
    return [(settings.OFFICE_ALERT_EMAIL, None)]


def notify_caregivers_of_failure(delivery: Delivery) -> None:
    ctx = {
        "delivery": delivery,
        "member": delivery.member,
        "reason": (delivery.failure_reason or "").split(":", 1)[0],
        "office_phone": getattr(settings, "OFFICE_PHONE", "03 9000 0000"),
    }
    html = render_to_string("delivery/emails/delivery_failed.html", ctx)
    txt = render_to_string("delivery/emails/delivery_failed.txt", ctx)
    sms_body = render_to_string("delivery/sms/delivery_failed.txt", ctx).strip()

    rate_key = _rate_key(delivery)
    sms_allowed = cache.add(rate_key, 1, SMS_TTL)

    for email, phone in _recipients(delivery):
        msg = EmailMultiAlternatives(
            subject=f"MerryMeal: today's delivery for {delivery.member.first_name} failed",
            body=txt,
            to=[email],
        )
        msg.attach_alternative(html, "text/html")
        msg.send()

        if phone and sms_allowed:
            send_sms(to=phone, body=sms_body)
```

```python
"""Fire caregiver alerts on Delivery.status transitions to 'failed'."""
from django.core.cache import cache
from django.db.models.signals import post_save
from django.dispatch import receiver
from django_q.tasks import async_task

from apps.delivery.models import Delivery


@receiver(post_save, sender=Delivery)
def on_delivery_saved(sender, instance: Delivery, created: bool, update_fields, **kwargs):
    if instance.status != "failed":
        return
    # Only fire when status was actually touched
    if update_fields is not None and "status" not in update_fields:
        return
    # Dedupe: alert per delivery at most once.
    if not cache.add(f"alert-fired:{instance.id}", 1, 60 * 60):
        return
    async_task("apps.delivery.tasks.alerts.send_failure_alert", instance.id)
```

## Pitfalls reviewer will check

- **Signal fires on every save** — without the `update_fields` /
  `cache.add` dedupe, marking a row failed twice in a tight loop sends
  two alerts. Cache key must be per-delivery and TTL ≥ 1 h.
- **Rate limit boundary** — key is `(member, date)`, not `(delivery,
  date)`. Two failed deliveries for the same member must collapse to
  one SMS.
- **Sync mode in tests** — `Q_CLUSTER["sync"] = True` makes
  `async_task` run inline. Without it, tests assert on an empty
  outbox.
- **Twilio import at module load** — `from twilio.rest import Client`
  inside the backend method, not at top of file. Otherwise dev boxes
  without `twilio` installed crash on import.
- **`cache.add` returning `False`** — the rate-limit gate must use
  `add`, not `set`. `set` overwrites and never blocks.
- **Email-only fallback for office** — office gets a copy of the email
  but never SMS (no phone). Confirmed by `phone=None` in `_recipients`.
- **Template typos** — Playwright cannot catch these; verify Django
  `render_to_string` ran by parsing the email body with
  `BeautifulSoup` in a test.
- **`Delivery.member.caregivers`** — relies on `MemberCaregiver`
  having `related_name="caregivers"` (or reverse `member_caregiver_set`
  default). Check the actual related name in the existing model and
  adjust the queryset.
- **PII in error logs** — when Twilio raises, log the recipient *count*
  not the phone number.
